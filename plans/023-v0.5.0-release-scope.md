# Plan 023: 0.5.0 Release Scope

> **Status:** scope draft for review. Defines the 0.5.0 workstreams, sequencing,
> and what is explicitly out of scope. Detailed designs live in their own plans
> (Plan 022) or breadcrumbs (BC-115..118); this is the umbrella.

## Theme

Harden the integration edges (LDAP, outbound HTTP) and lean into the
**regulated-SMB observability** story: first-class alerting channels, SIEM/log
export, and an auditor-facing compliance view.

## Workstreams

### A. Security hardening batch
The three confirmed security findings, plus the shared infrastructure they need.

- **[[BC-115]] LDAP StartTLS bind fix** (High). Always bind after StartTLS on both
  the service and user connections; credentials never cross an unencrypted
  channel. Needs a mocked-`ldap3` test (no coverage today).
- **SSRF-safe HTTP opener** (shared infra). One `scan.safe_urlopen` that resolves
  → checks `_is_blocked_ip` → validates **every redirect hop** → enforces an
  `http(s)` scheme allowlist. Policy is **explicit and configurable** (reuse
  `allow_private` / `allowed_subnets`), not hardcoded — see BC-116 rationale.
  - **[[BC-116]] Webhook redirect SSRF** (Medium) — route `send_webhook` + digest
    delivery through it; reconcile the env-vs-GUI private-IP inconsistency.
  - **[[BC-117]] Revocation-probe SSRF** (Medium) — route OCSP/CRL probes through
    it; thread `allow_private`/`allowed_subnets` into `evaluate_posture`; emit a
    clear "blocked by policy" Finding.

> Caveat to set expectations: a urllib-redirect-handler opener validates each
> hop's resolved IP but urllib re-resolves on connect, so it's a large
> improvement, not the airtight pinned-IP guarantee the host scanner has. Don't
> claim it "fully closes SSRF."

### B. LDAP portability
- **[[BC-118]] Configurable group filter** (Medium). `LDAP_GROUP_FILTER` with a
  `{group}` placeholder, defaulting to the current AD transitive OID. Unblocks
  OpenLDAP/FreeIPA `LDAP_REQUIRED_GROUPS`.

### C. Alert channel adapters — **Plan 022**
Teams (Adaptive Card / Workflows, new format only), Discord, PagerDuty (Events
API v2, trigger + resolve-on-renewal). Delivers through the SSRF-safe opener from
workstream A, so sequence it after that lands.

### D. Compliance / Auditor Export *(new — feature)*
A "Compliance View" + one-click **Auditor Export** (PDF / signed CSV) summarizing
posture: % of certs with SHA-256+, RSA-2048+/ECDSA, TLS 1.2+, CAA present, plus
an active-remediation schedule (what's expiring and when). Builds on existing
posture grading + fleet lenses; this is the headline differentiator for the
regulated-SMB target. Graduate to its own plan when picked up.

### E. ACME renewal-window alert *(new — feature)*
A signal distinct from the generic expiry threshold: "cert is inside its renewal
window (e.g. 25 days out) and no successor cert has appeared," which flags a
broken Certbot/cert-manager job long before outage. Builds on existing renewal
tracking. Small; spec at pickup.

### F. SIEM / log export *(new — feature; subsumes the old "syslog" ask)*
Make cert-watch's audit log and alerts consumable by a SIEM. The audit rows are
already structured (`ts, actor, action, target_type, target_id, detail,
source_ip`).

- **Splunk HEC exporter** (primary) — POST audit events (and optionally alerts) as
  JSON to a Splunk HTTP Event Collector endpoint with a token, through the
  SSRF-safe opener. Config: `CERT_WATCH_HEC_URL`, `CERT_WATCH_HEC_TOKEN` (+`_FILE`),
  optional index/sourcetype. Structurally the same adapter pattern as Plan 022.
- **Syslog handler** (generic) — RFC 5424 via `logging.handlers.SysLogHandler`,
  gated by `CERT_WATCH_SYSLOG_*`. Serves Splunk's syslog/UF path **and any other
  SIEM** (QRadar, Sentinel, etc.).
- **Windows Event Log writer** (IIS target) — so the Windows deployment has a
  native sink AMA/agents already collect.

**Microsoft Monitoring Agent: explicitly not built.** MMA (the Log Analytics
agent) was retired 2024-08-31 and superseded by Azure Monitor Agent (AMA). You
don't wire an app to MMA/AMA directly — they collect from syslog / Windows Event
Log / files. So the syslog + Event Log outputs above *are* the Azure-native path
(AMA → Log Analytics/Sentinel via a Data Collection Rule), forward-compatible and
agent-agnostic. A direct Azure Monitor **Logs Ingestion API** exporter is a
possible future HEC-analog but is deferred (Azure-specific setup, narrow payoff
for an SMB tool).

## Sequencing

1. **A** first — BC-115 (independent), then the SSRF-safe opener, then BC-116/117
   on top of it. The opener is a dependency for C and F.
2. **B** anytime (independent).
3. **F (Splunk HEC + syslog)** and **C (adapters)** both consume the opener — do
   them after A. They share the "format → POST through safe opener" shape; build
   the opener once, reuse for adapters + HEC.
4. **D** and **E** are independent features; schedule by appetite. D is the
   biggest single item and the strongest differentiator.

## Out of scope (explicitly deferred)

- Microsoft Monitoring Agent integration (deprecated; see F).
- Azure Monitor Logs Ingestion API direct exporter (possible future).
- Multi-replica / HA and shared-secret distribution — the single-replica SQLite
  model (Recreate + RWO PVC + PDB) is intentional for the SMB target; document the
  constraint rather than chase active-active.
- Audit-log tamper-evidence (hash chaining) — the real compliance hardening beyond
  export; worth its own effort later, not 0.5.0.
- Fernet decrypt-failure behavior change — evaluate separately from this batch
  (current fail-loud + the startup `check_encrypted_values` warning is defensible).

## Open question

- **Version:** ship as **0.5.0** (recommended — these are features + hardening,
  0.4.0 is already tagged and deployable) vs. holding 0.4.0. If 0.5.0, the 0.4.0
  push can go out now independently.
