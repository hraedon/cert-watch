## Context

cert-watch reached reasonable feature-completeness at v0.6.0 (RBAC, API keys,
full E2E). The biggest remaining risk to its "Real Software" credibility is
**surfaces that look done but aren't**, and a backlog that drifted from reality.
v0.6.0 already surfaced two (RBAC inert; a test that was theater); the v0.6.5
review surfaced a third (RBAC inert *for AD specifically* — BC-150). This
iteration hunts those, hardens the credibility-facing paths, runs a security
review, and will cut a clean release with honest boundaries.

**Decision on file:** mis-issuance — removed the stubbed UI now, build it real
in 1.1 (BC-151).

## Division of labor

This plan was split deliberately (see PR #4 description):

- **Agent (reserved — judgment-heavy / credibility-defining / hard-to-reverse):**
  security review, the truth fixes, the Discover-load architecture, the visual
  gate decision, and the release cut.
- **Team (well-specified / mechanical / parallelizable):** the routine low-sev
  backlog and C2 / D2 / D3 below — each has a tracking breadcrumb and an existing
  test or a clear spec, so they can proceed independently.

### B. Security review — ✅ DONE (PR #4)
- Reviewed the v0.5.3→HEAD surface (API-key bearer auth, RBAC wiring, OAuth claim
  trust, SSRF opener, CSP/CSRF, migrations).
- **HIGH finding fixed — BC-150.** RBAC was silently **inert for real LDAP/AD**:
  group values are DNs with commas, and the session token's `,`-join encoding
  (`auth/session.py` `_encode_list`/`_decode_list`) shredded each DN into RDN
  fragments, so they never matched `CERT_WATCH_ROLE_MAP` → every AD user fell
  back to `viewer`. Same comma-split root cause as the old `LDAP_REQUIRED_GROUPS`
  bug, reintroduced in the BC-145 session propagation; the v0.6.0 RBAC E2E missed
  it by using comma-free names. **Fix:** lossless base64url(JSON) encoding;
  regression `tests/test_session_groups.py`; the RBAC browser E2E now uses real
  DN groups. Everything else reviewed clean (OAuth groups/roles are only trusted
  after JWKS/userinfo validation; API-key tokens are 256-bit, hashed-at-rest,
  rate-limited).

### C. Performance / correctness on credibility paths

  reconciliation cache (`ct_monitor.peek_reconciliation`, no I/O) and warms
  stale/missing domains off-thread (`start_reconciliation_refresh`), with a
  "reconciling…" / "updated Ns ago" indicator. Tests:
  `tests/test_discover_decouple.py`.
- **C2 Compliance report (BC-120, BC-122)** — ✅ DONE. Added

  `list_dashboard_rows` memory load. All 28 compliance tests pass; full suite
  1324 passed, 89.58% coverage.

### F. Post-review hardening — ⬜ REMAINING (agent, after maturity review)

A maturity review surfaced five gaps. Three are small enough to close under the
v0.6.5 feature-freeze; two are deliberately pushed to later plans.

**In scope for v0.6.5:**
- **F1 Backlog hygiene.** Regenerate `OPEN_BREADCRUMBS.txt` from the agent-notes
  DB (`agent-notes breadcrumb export-index`), sync resolved breadcrumbs (BC-097,
  BC-099, BC-120, BC-122, BC-129, BC-150 are already resolved), and prune stale
  entries. Low risk, high trust signal.
- **F2 HSTS header (BC-154).** The app scans remote hosts for HSTS but does not
  set `Strict-Transport-Security` on its own responses. Add the header in
  `security_headers_middleware` when `CERT_WATCH_COOKIE_SECURE=1` (or document
  the omission). Add a ratchet test. Small, bounded, security-relevant.
- **F3 scan.py coverage (BC-155).** `scan.py` is at 74% because the
  `openssl s_client` subprocess path and socket-error branches are unmocked.
  Add subprocess/socket mocking for the two dominant untested branches (chain
  extraction via openssl fallback, and connection refused / timeout handling) to
  push the module above 85%. Bounded; no behaviour change.

**Out of scope (pushed back):**
- **Spec completeness** — `docs/spec/` only covers the v0.1.0 MVP. Writing specs
  for auth, RBAC, compliance, posture, alert adapters, and SIEM is significant
  documentation work (6+ files). Defer to a dedicated **0.7.0 docs plan**; it
  does not fit a feature-freeze hardening iteration.
- **config.py decomposition** — The 700-line procedural `Settings.from_env()`
  is a known smell, but refactoring it is a risky, cross-cutting change that
  touches every setting and every test. A hardening iteration is the wrong place
  for a config-layer refactor; defer to a dedicated plan with dedicated QA.
