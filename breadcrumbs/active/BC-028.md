# BC-028: openssl s_client fallback opens a second TLS connection per scan

**Type:** defect
**Severity:** low
**Status:** new
**Filed:** 2026-05-28

## Problem

In `scan.py:_get_chain_via_openssl()` (line 267), the openssl s_client fallback for Python 3.12 chain extraction connects to the host a second time using the peer address from the first connection. This means:

1. The original TLS connection (via `_open_tls_connection`) fetches the leaf cert via `getpeercert(True)` but lacks chain methods on Python 3.12.
2. `_get_chain_via_openssl()` opens a brand-new TCP+TLS connection to the same host:port, runs `openssl s_client -showcerts`, and parses the PEM output.
3. If the host is behind a load balancer with multiple backends, the second connection may hit a different backend with a different certificate.
4. The second connection doubles the scan time for affected hosts.

The fallback was added as BC-022 to work around Python 3.12 lacking `get_unverified_chain()`/`get_verified_chain()`. On Python 3.13+ these methods exist and the fallback is skipped.

## Suggested fix

Pass the raw socket from the first connection to openssl via its stdin/stdout (similar to `openssl s_client -connect fd:0`), or cache the DER bytes from the first TLS handshake for chain extraction. Alternatively, use the `cryptography` library's PKCS#7 parsing on the TLS handshake output if available.
