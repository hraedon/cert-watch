## Context

The adversarial review fix wrapped 4 write routes in `get_write_lock()` (delete_certificate_cascade, store_uploaded, add_host, delete_host). However, ~14 additional API write routes still don't acquire the write lock: `PATCH /api/certificates/{id}/notes`, `PUT /api/certificates/{id}/tags`, `PATCH /api/hosts/{id}/owner`, `PATCH /api/hosts/{id}/notes`, `PUT /api/hosts/{id}/tags`, `PUT /api/hosts/{id}/issuers`, `POST/DELETE /api/alert-groups/{id}/certs/{cert_id}`, and 6 HTML form equivalents.

These are single-statement updates that complete quickly, so the SQLite `busy_timeout=15000` usually handles contention. But under scheduler load (scan cycle holding the write lock for minutes), these can still hit `SQLITE_BUSY` and return 500 to the user. The fix is either to wrap each in `get_write_lock()`, or to wrap the repository write methods at a lower level so every write path is automatically coordinated.
