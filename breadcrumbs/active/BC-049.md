# BC-049: In-memory rate limiting is not shared across multiple workers

**Type:** defect
**Severity:** medium
**Status:** new
**Filed:** 2026-05-30

## Problem

`middleware.py` implements sliding-window rate limiting with a process-local dictionary (`_rate_windows: dict[str, collections.deque]`). This works for a single-process/server deployment, but as soon as `cert-watch` runs with multiple Gunicorn/uvicorn workers, `gunicorn --workers 4`, or a Kubernetes deployment with `replicas > 1`, each worker maintains its own independent counter. An attacker can issue 60 requests × N workers before being throttled on any single worker.

## Suggested fix

Option A (short-term): Store rate-limit windows in the existing SQLite database. A dedicated table (`rate_limit_windows`) with columns `(key, timestamps_json, last_activity)` and an in-memory `asyncio.Lock`-protected cache in the middleware layer. SQLite is single-writer but WAL mode reads can scale; this is still better than completely independent process memory.

Option B (long-term): If BC-031 (PostgreSQL/MSSQL support) lands, a native `INSERT ... ON CONFLICT UPDATE` pattern in PostgreSQL would scale better than SQLite for this use case.

Option C (simplest): Document the limitation — rate limiting is “best effort per worker” — and recommend running `replicas=1` or using an external WAF/ingress rate limit.
