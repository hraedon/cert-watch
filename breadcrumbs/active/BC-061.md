# BC-061: "Chain validation" never verifies signatures — name matching only

**Type:** defect
**Severity:** medium
**Status:** open
**Filed:** 2026-05-31

## Problem

The chain trust logic in `cert_chain.py` is purely name/structure based; no
cryptographic signature is ever checked. Functions:

- `validate_chain_order` (cert_chain.py:141-152) compares only DER-encoded
  *Name* bytes: `_issuer_bytes(chain[i]) == _subject_bytes(chain[i+1])`.
- `validate_chain_with_anchors` (184-198) — same issuer==subject name walk, plus
  a final name-match against an anchor.
- `is_anchored_by_user` (201-215) — fingerprint OR name-linkage match.
- `chain_status` (218-245) — the function actually used by the scanner
  (`scan.py:657`) and feeding posture grading — calls only `validate_chain_order`
  + name/anchor checks.

Confirmed by grep over `src/cert_watch/`: there is **no** call to
`public_key().verify(...)`, `verify_directly_issued_by`, or any signature
verification anywhere. So a chain assembled from certificates whose
Subject/Issuer *names* line up but whose keys are forged/mismatched is reported
as `"public"` / `"private"` / structurally valid, and `posture.py` will grade it
accordingly (chain_completeness "pass").

This is an over-claim: a "chain validation" / "chain trust status" feature that
cannot detect a forged intermediate gives false assurance. README/UI imply trust
status; the value only reflects name linkage.

## Suggested fix

- In `chain_status`/`validate_chain_with_anchors`, verify each cert is actually
  signed by the next (`x509 verify_directly_issued_by` on cryptography >=40, or
  manual `issuer_pubkey.verify(...)`), and that the top chains to a trust anchor
  by signature — not just name.
- If full path validation is out of scope, rename the statuses/fields so they do
  not imply cryptographic validity (e.g. "names-link" vs "verified").
- Add tests with a name-matching-but-bad-signature chain asserting it is NOT
  reported valid/public.
