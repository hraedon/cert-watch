# BC-044: Unbounded scan_history and alerts queries load all rows into memory

**Type:** defect
**Severity:** medium
**Status:** new
**Filed:** 2026-05-30

## Problem

`list_scan_history()` and `list_alerts_with_subject()` in `database/queries.py` execute `fetchall()` with no LIMIT clause. The `scan_history` table grows by one row per host per scan cycle — after months of daily scans of hundreds of hosts, these queries return tens of thousands of rows loaded entirely into Python memory. Both the `/scan-history` and `/alerts` HTML views call these with no pagination.

Similarly, `api_list_hosts` and `api_list_alerts` in `routes/api.py` load the full dataset via `list_all()` or `list_alerts_with_subject()` before slicing in Python, discarding most rows for any non-first page. This is O(n) memory per request.

## Suggested fix

Add `page`/`limit` parameters and `LIMIT/OFFSET` clauses to `list_scan_history` and `list_alerts_with_subject`, matching the pattern used by `list_dashboard_rows`. Add SQL-level pagination to the hosts API (the repository already supports `list_all` but should gain `list_page`). The HTML views should also paginate.
