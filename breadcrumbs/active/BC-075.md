## Problem

Plan 020 S4 tried to harden the CSP by replacing `script-src 'self'
'unsafe-inline'` with a per-request nonce. It shipped and broke the UI: every
button stopped working and 4 e2e tests went red (`.cw-slide.on` never appears,
upload/add-host rows time out). Root cause: the templates use ~24 inline
event-handler attributes (`onclick=`, `onchange=` in `dashboard.html`,
`certificate_detail.html`, `settings.html`, `base.html`). CSP nonces only
whitelist `<script>` blocks — they do **not** apply to inline event-handler
attributes — so removing `'unsafe-inline'` blocked all of them.

S4 was reverted (CSP back to `script-src 'self' 'unsafe-inline'`; nonce
generation + template `nonce=` attributes removed) to restore green. So
injected `<script>` is still permitted by CSP; `escHtml()` + the urgency-class
whitelist remain the XSS mitigations.

## Suggested fix

Do this during the upcoming **design-session template rewrite** (it replaces
these templates anyway, so refactoring handlers now would be throwaway work):

1. Replace inline `onclick=`/`onchange=` handlers with `data-*` attributes +
   delegated `addEventListener` listeners inside a single nonce'd `<script>`
   (e.g. `data-action="open-slide"`, `data-group="{{ id }}"`, `data-tab="scan"`,
   `data-href="/certificates/{{ id }}"`).
2. Re-introduce the per-request nonce in `security_headers_middleware` and drop
   `'unsafe-inline'` from `script-src`.
3. Add an e2e assertion that a button-driven flow (Add host → slide opens)
   still works under the nonce CSP, so the regression can't recur.
