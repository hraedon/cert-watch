## Problem
`pagerduty_routing_key` is a member of `SENSITIVE_SETTING_KEYS`
(`config/helpers.py`), so in the **alert-delivery** path it is encrypted at rest
and masked in the UI. The **event-stream** path does not use that machinery:
`routes/settings/events.py` reads `pagerduty_routing_key` from the form and
stores it via `EventStreamConfig.to_json()` into the `event_stream_config`
kv_store blob — unencrypted — and `templates/settings_events.html:62-63`
renders the real key into the password field's `value="{{ config.pagerduty_routing_key }}"`.

## Impact
Two related gaps, both admin-gated (low severity, but real):
1. **At rest:** the PagerDuty routing key is stored as plaintext JSON inside the
   `event_stream_config` kv value, not as an `enc:v1:` secret. Contrast
   `routes/settings/config.py:_effective_config`, which transparently
   decrypts `pagerduty_routing_key` for the alert path.
2. **In page source:** the real routing key is written into the HTML `value=""`
   of a `type=password` input — visible via view-source/devtools, retained in
   browser DOM/form history, and offered to password managers. The other
   sensitive fields (e.g. `ldap_bind_password`, `smtp_password`,
   `webhook_headers` after v0.9.2) use the empty-`value` + mask-as-placeholder
   convention so the secret is never round-tripped through the page.

## Found
End-of-session review while closing v0.9.3 (noticed the `value=` render in
`settings_events.html` during WI-060 work), 2026-06-17. Pre-existing — not
introduced this release.

## Fix sketch
Mirror the alert-path handling: store `pagerduty_routing_key` as an
`enc:v1:` secret (the `EventStreamConfig` blob would need to exclude it from
the JSON and persist it via `kv_set_secret`), and render the Events page field
with `value=""` + `••••••••` placeholder when set (the existing blank-skip
round-trip in `_save_config_section` handles "leave unchanged"). Touches
`EventStreamConfig` (events.py), the events render + save routes, and
`settings_events.html`.

## Scope note
Not in v0.9.3 (separate subsystem; the alert-path handling is already correct).
Track here.
