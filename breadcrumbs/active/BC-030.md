# BC-030: app.py is a monolith mixing routes, middleware, helpers, and business logic

**Type:** design-debt
**Severity:** medium
**Status:** in-progress
**Filed:** 2026-05-29
**Progress:** Initial decomposition complete. app.py reduced from ~1500 lines to ~110 lines.

## Problem

`app.py` is ~1500 lines containing all route handlers, middleware (CSRF, rate limiting, auth), template filters, DN parsing helpers, rate limiter state, and business logic (host import, scan orchestration). This makes it hard to reason about changes — modifying the rate limiter risks breaking a route, and adding a new endpoint requires understanding the entire file.

The file has grown organically as features were added. The middleware functions at the top, helper functions in the middle, and route handlers scattered throughout. Imports are a mix of top-level and inline.

## Suggested fix

Split into modules following the existing pattern (the project already has `scan.py`, `alerts.py`, `auth.py`, etc.):

- `routes/` — route handlers grouped by resource (hosts, certificates, alerts, api, auth)
- `middleware.py` — CSRF, rate limiting, auth middleware
- `filters.py` — Jinja template filters and DN parsing helpers
- `app.py` — app factory, lifespan, mount points, and middleware registration

Each route module uses `APIRouter` (FastAPI's router pattern). The app factory imports and mounts them. This is the standard FastAPI project structure.
