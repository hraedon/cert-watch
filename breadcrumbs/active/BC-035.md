# BC-035: database.py is the new monolith — 1200 lines mixing schema, repositories, and dashboard queries

**Type:** design-debt
**Severity:** medium
**Status:** new
**Filed:** 2026-05-29

## Problem

After `app.py` was decomposed into `routes/`, `middleware.py`, and `filters.py` (BC-030), `database.py` is now the largest file at ~1200 lines. It contains schema DDL, migration logic, `init_schema()`, four repository classes (`CertificateRepository`, `AlertRepository`, `SqliteHostRepository`, `SqliteTrustAnchorRepository`), dashboard query functions (`list_unified_entries`, `list_dashboard_rows`, `count_dashboard_leaves`), and private helpers (`_connect`, `_iso`, `_parse_iso`).

This makes it hard to reason about changes: modifying a repository method risks breaking a dashboard query, and the schema migration code interleaves with business logic. It's the same problem BC-030 solved for `app.py`.

## Suggested fix

Split into `database/` package:
- `database/schema.py` — DDL strings, `init_schema()`, migration logic
- `database/repo.py` — Repository classes (or one file per repo)
- `database/queries.py` — Dashboard and unified-entries query functions
- `database/connection.py` — `_connect`, `_iso`, `_parse_iso`, thread-local caching

Imports change from `from cert_watch.database import X` to `from cert_watch.database import X` (package `__init__.py` re-exports), so no external API change.
