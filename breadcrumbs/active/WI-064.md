## Context
v0.9.3 (WI-061) shipped the v1 tier-decoupling: a role with `scope_tags` set is
*scoped* and contributes only its tags to visibility/alert routing — it can no
longer elevate the user's permission tier. Effective tier = highest tier among
the user's UNSCOPED (global) roles; a user holding only scoped roles defaults to
`viewer`. This closes the immediate footgun (adding a scoped role for visibility
silently elevating privileges) without a rewrite.

## What v1 does NOT do
Per-tag permission tiers. Today a user's tier is global: if they hold an
unscoped operator role, they are an operator across *all* their visible tags,
not just the unscoped role's intent. The fully-correct model is per-tag-scoped:
a user could be `operator` for tag A and `viewer` for tag B, with permission
checks evaluated against the tag of the resource being acted on. Under v1 such a
user is operator everywhere (their unscoped operator role sets the global tier).

## Target
Move to per-tag-scoped permissions as part of any 1.0 release. This is an
architectural change: permission checks (`may_write`, route guards, the API
write deps) become tag-aware — they must consult the tier scoped to the
*resource's* tag, not a single global tier. Touches `AuthContext`,
`auth/rbac.py`, `routes/_scoped.py`, the write-path deps in `middleware.py`, and
the role-resolution path.

## Why deferred
Larger blast radius than v0.9.3 should absorb in maintenance mode (every write
path becomes tag-aware). v1 removes the privilege-elevation footgun, which is
the security-critical part; per-tag tiering is a correctness/least-privilege
refinement, not a vulnerability. Decision (user, 2026-06-17): do A now, track B
for 1.0.
