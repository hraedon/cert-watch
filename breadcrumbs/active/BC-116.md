identifier: BC-116
title: Webhook delivery follows redirects without SSRF re-validation
kind: security
status: open
severity: medium
---

## Problem

`_validate_webhook_url` (`routes/api.py:83`) blocks private/loopback/link-local
targets when a webhook URL is saved via the API/GUI. But delivery
(`alerts.py:337-345` in `send_webhook`, and `alerts.py:539-545` in the digest
path) uses `urllib.request.urlopen`, which **follows 3xx redirects by default to
unvalidated targets**. A public webhook can 302 to `http://127.0.0.1/...` or
`169.254.169.254` and the request is sent.

Two gaps compound it:
- The redirect target is never re-checked against the blocklist.
- The **env path** (`ALERT_WEBHOOK_URL`, read in `config.build_webhook_config`)
  never goes through `_validate_webhook_url` at all, so even the initial target is
  unvalidated for env-configured webhooks.

Trust boundary note: the webhook URL is admin/operator-configured, so this is
hardening (compromised/careless admin, redirect abuse) rather than an
unprivileged-user SSRF — hence Medium, not High.

## Suggested fix

Route delivery through a shared SSRF-safe opener that validates **every hop**
(initial + each redirect) by resolving and checking the IP against
`scan._is_blocked_ip`, and enforces an `http(s)` scheme allowlist. Make the
private-IP policy **explicit and consistent** between the env and GUI paths
(today they disagree) rather than hardcoding `allow_private=False` — some SMBs run
internal webhook receivers. Add a test: a 302 → loopback is blocked.

Shares the opener with [[BC-117]]. Scoped for 0.5.0 — see Plan 023.
