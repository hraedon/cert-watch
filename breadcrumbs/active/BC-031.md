# BC-031: Add PostgreSQL and MSSQL support alongside SQLite

**Type:** feature
**Severity:** medium
**Status:** deferred (2026-06-07)
**Filed:** 2026-05-29

## Problem

The application is hard-coded to SQLite (`database.py` uses `sqlite3` directly). This limits deployment options — organizations that standardize on PostgreSQL or MSSQL for operational reasons (backups, replication, access control, existing tooling) cannot adopt cert-watch without a database layer rewrite.

The current repository pattern (`CertificateRepository`, `AlertRepository`, `SqliteHostRepository`) provides a clean abstraction boundary in name, but the concrete implementations are deeply SQLite-specific. The problem is worse than it appears:

- `_connect()` returns a raw `sqlite3.Connection` with `row_factory = sqlite3.Row`. Every repository method uses `dict(r)` to access columns, which is sqlite3-specific.
- `_iso()`/`_parse_iso()` do manual ISO format string serialization — other databases have native datetime support.
- Raw SQL strings use SQLite dialect (`?` placeholders, `INSERT OR IGNORE`, `ON CONFLICT`, `strftime`, `BOOLEAN` as `INTEGER`).
- The `init_schema()` function does ad-hoc column migrations via `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` — this won't port to PostgreSQL's DDL model.
- Dashboard queries like `list_unified_entries()` join across tables in memory rather than using proper SQL joins, which would perform poorly on remote databases with network latency.

The SQLite abstraction leaks into 25+ call sites through `database.py`'s public API, so changing the backend is not just a matter of swapping a driver — it requires restructuring how the entire data layer works.

## Suggested fix

1. Extract `database.py` into a `database/` package (see BC-035).
2. Define abstract repository interfaces with proper return types.
3. Create backends: `sqlite.py`, `postgres.py`, `mssql.py`.
4. Use SQLAlchemy Core (not ORM) or a lightweight query builder for dialect portability.
5. Add `DATABASE_URL` env var (e.g. `sqlite:///path`, `postgresql://...`, `mssql://...`).
6. Keep SQLite as default for backward compatibility.
7. This is blocked by FEAT-006 (migration tooling) — multi-database support needs proper schema migrations.
8. Dashboard queries should be rewritten as proper SQL joins rather than in-memory merging, which also helps BC-033.
