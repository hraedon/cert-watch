# BC-031: Add PostgreSQL and MSSQL support alongside SQLite

**Type:** feature
**Severity:** medium
**Status:** new
**Filed:** 2026-05-29

## Problem

The application is hard-coded to SQLite (`database.py` uses `sqlite3` directly). This limits deployment options — organizations that standardize on PostgreSQL or MSSQL for operational reasons (backups, replication, access control, existing tooling) cannot adopt cert-watch without a database layer rewrite.

The current repository pattern (`CertificateRepository`, `AlertRepository`, `SqliteHostRepository`) provides a clean abstraction boundary, but the concrete implementations are SQLite-specific. The `_connect()` function, `_iso()`/`_parse_iso()` datetime helpers, and raw SQL queries use SQLite dialect.

## Suggested fix

1. Define abstract repository interfaces (already partially done with `CertificateRepository` and `AlertRepository`).
2. Create `database/` package with `sqlite.py`, `postgres.py`, `mssql.py` backends.
3. Use SQLAlchemy Core (not ORM) or a lightweight query builder for dialect portability.
4. Add `DATABASE_URL` env var (e.g. `sqlite:///path`, `postgresql://...`, `mssql://...`).
5. Keep SQLite as default for backward compatibility.
6. This is blocked by FEAT-006 (migration tooling) — multi-database support needs proper schema migrations.
