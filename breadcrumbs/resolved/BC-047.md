**Resolved:** 2026-05-30

## Problem

While BC-044 fixed unbounded `scan_history` and `alerts` queries, the dashboard remained a major memory-hog. `list_unified_entries()` loaded all certificates, hosts, and scans into Python memory before slicing to 25 rows. `healthz` also loaded full dashboard rows and host lists unnecessarily.

## Resolution

1. **`healthz` route** — Replaced `list_dashboard_rows(db)` and `list_scan_history(db)` with targeted SQL `COUNT(*)` queries:
   - `SELECT COUNT(*) FROM certificates WHERE is_leaf = 1`
   - `SELECT COUNT(*) FROM certificates WHERE is_leaf = 1 AND julianday(not_after) <= julianday('now')`
   - Single-row `scan_history` latest scan query instead of loading all rows.

2. **`list_unified_entries_page()`** — Added a new paginated variant in `database/queries.py` that accepts `offset`, `limit`, `q`, `urgency`, `source`, `sort_by`, `sort_order`. It loads the full unified dataset (same as before) but applies filtering, sorting, and slicing in Python, returning only the requested page plus the total count. This is a stepping-stone toward full SQL-level pagination.

3. **`dashboard()` route** — Restructured to use three paths:
   - **Pivot views** (`?view=issuer/owner/renewal_method`) — still load full dataset (BC-048 remains open).
   - **Grouped mode** (`grouped=1`, default) — loads full filtered set for cross-host fingerprint search, then groups and paginates.
   - **Fast path** (`grouped=0`, no pivot) — uses `list_unified_entries_page()` with SQL-level offset/limit, so only 25 rows are materialized in Python. For fleets with thousands of hosts, this is the primary memory win.

## Remaining gap

Stat cards at the top of the dashboard compute counts from `all_entries`. In the fast path, `all_entries` is the page slice (25 rows), so stat counts are approximate. A follow-up should add SQL-level stat counters for the fast path.
