## Problem

The application was hard-coded to SQLite (`database.py` uses `sqlite3` directly). This limited deployment options for organizations standardizing on PostgreSQL or MSSQL.

## Resolution

The `database/` package was extracted from the monolithic `database.py` in Plan 018 and BC-035. Key changes since BC-031 was filed:

- **Repository pattern with SQLite-only concrete classes:** `repo.py` defines `CertificateRepository`, `AlertRepository` interfaces, and `SqliteCertificateRepository`, `SqliteHostRepository`, `SqliteTrustAnchorRepository`, `SqliteAlertRepository`, `SqliteAlertGroupRepository` as concrete implementations.
- **Portable return types:** All repository/query methods return plain `dict`s or dataclasses (no `sqlite3.Row` leaks into routes).
- **SQL-paginated dashboard queries:** `list_dashboard_page()` and `list_dashboard_grouped_page()` use true SQL `WHERE`/`ORDER BY`/`LIMIT`/`OFFSET` with no full-table materialisation in Python (see BC-073).
- **Concise decomposition:** `database/` now contains `schema.py`, `connection.py`, `repo.py`, `queries.py`, plus concern-specific modules `dashboard.py`, `drift.py`, `cert_ops.py`, `calendar.py`, `kv_store.py`, `session_versions.py`, `fleet.py`, `posture.py`, `pagination.py`, `encryption.py` (BC-094).

**What is NOT yet addressed:** PostgreSQL or MSSQL backend drivers, SQLAlchemy Core dialect portability, `DATABASE_URL` env var, or connection pooling. These are deferred indefinitely per the 2026-06-04 reconciliation — the current SQLite backend with WAL mode suffices for the single-writer deployment target.

## Resolution timestamp
