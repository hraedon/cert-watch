# BC-055: Rate limiter trusts X-Forwarded-For implicitly via request.client.host

**Type:** defect
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-31

**Resolved:** 2026-05-31 — CERT_WATCH_TRUST_PROXY=1 enables X-Forwarded-For extraction. CERT_WATCH_TRUSTED_PROXIES for trusted proxy list. _extract_client_ip() helper in middleware.py.

## Problem

`middleware.py:167` extracts the client IP for rate limiting via:
```python
client = request.client.host if request.client else "unknown"
```

When cert-watch runs behind a reverse proxy (nginx, Traefik, Kubernetes ingress), `request.client.host` is always the proxy's IP (e.g., `10.0.0.1`). This means:
- All requests share the same rate limit key (`api:10.0.0.1`)
- One aggressive client can exhaust the rate limit for everyone
- An attacker's requests are indistinguishable from legitimate ones

FastAPI/Starlette does not automatically parse `X-Forwarded-For` or `X-Real-IP` headers. The `request.client` comes from the ASGI scope, which is the direct TCP peer.

## Plan

Resolved by Plan 012 §3.5.

## Suggested fix

Option A: Add a `CERT_WATCH_TRUST_PROXY` env var. When set, extract the client IP from `X-Forwarded-For` (rightmost untrusted entry) or `X-Real-IP` header instead of `request.client.host`.

Option B: Use Starlette's `TrustedHostMiddleware` or a custom middleware that populates `request.scope["client"]` from forwarded headers when a trusted proxy list is configured.

Option C: Document the limitation — rate limiting is per-proxy-IP when behind a reverse proxy, and recommend WAF/ingress-level rate limiting instead.
