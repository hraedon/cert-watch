---
identifier: BC-090
title: 'BC-083 secure-by-default check regressed IIS HttpPlatformHandler startup (reads CERT_WATCH_HOST, not the actual uvicorn --host bind)'
kind: bug
status: resolved
severity: high
---

**Found + fixed 2026-06-02 (wrap-up regression sweep).** The BC-083 startup
refusal in `app.py` (~line 143) reads the bind host from the `CERT_WATCH_HOST`
env var, defaulting to `0.0.0.0`. The recommended IIS model
(`deploy/iis/web.config`, HttpPlatformHandler) binds loopback via the uvicorn
`--host 127.0.0.1` CLI argument and never sets `CERT_WATCH_HOST`; it also bypasses
`__main__.py`. So a default IIS install (shipped with `AUTH_PROVIDER` commented
out, no `CERT_WATCH_ALLOW_UNAUTH`) resolved `bind_host=0.0.0.0` and raised
`SystemExit` → HTTP 502.5, app failed to start. Reproduced by driving
`app.router.lifespan_context` with the exact env `web.config` produces.

Initial stop-gap (commit 640ded4) added `CERT_WATCH_HOST=127.0.0.1` to
`deploy/iis/web.config` so the check saw the loopback bind.

**Proper resolution.** The root fragility was that the [[BC-083]] check trusted
`CERT_WATCH_HOST` as a proxy for the actual bind, which isn't guaranteed when the
process binds via uvicorn args / a programmatic launch. Fixed by making the
`cert-watch` entrypoint (`__main__`) the single source of truth: it now accepts
`--host`/`--port` and normalizes `CERT_WATCH_HOST` to the host it actually passes
to uvicorn, so the check can never diverge from the real bind. IIS
HttpPlatformHandler now launches via `-m cert_watch --host 127.0.0.1` (through the
entrypoint) instead of `-m uvicorn` directly, so the stop-gap env var was removed.
Launches that bypass the entrypoint (raw `uvicorn cert_watch.app:app`) fail closed
unless `CERT_WATCH_HOST`/`CERT_WATCH_ALLOW_UNAUTH` is set (the secure direction).

Also replaced the BC-083 tests, which were boolean replicas that never read
`CERT_WATCH_HOST` (why they missed this) — they now drive the real lifespan and
the entrypoint host-normalization. The reverse-proxy IIS model and
systemd/k8s/compose were unaffected.
