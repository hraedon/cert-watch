title: Chain trust — final hop to the SYSTEM trust store is still name-based (not

status: resolved

## Problem

BC-061 added real cryptographic signature verification of every *presented*
link in a chain (`validate_chain_signatures` in `cert_chain.py`), and verifies
user-uploaded anchors by signature (`_is_signature_anchored_by_user`). That
closes the forged-intermediate hole for self-signed-root and user-anchored
chains.

But the **system trust store** hop in `chain_status` is still name-based.
`_is_anchored_by_system_root(full)` (`cert_chain.py:118-152` area) returns true
when the top presented cert's *issuer Name* matches a system-root *subject
Name* — it never loads the system root and verifies that the top cert is
actually signed by it.

Residual attack: present `[leaf, intermediate]` where `leaf` is genuinely
signed by `intermediate` (so `validate_chain_signatures` passes), and
`intermediate.issuer` Name is spoofed to equal a real public-CA root's subject
Name (but `intermediate` is NOT actually signed by that root). `chain_status`
grades it `public`. A real TLS client would reject this chain.

This is narrower than the pre-BC-061 bug (you can no longer forge a middle
link), but it is not zero, and `public` is the strongest trust grade the UI/
posture grader shows.

## Suggested fix

Verify the final hop against the system trust store by signature, not name:
load candidate system roots (e.g. via the `certifi`/OS trust bundle), find the
one whose subject matches, and require `_is_signed_by(top, root)`. Better, adopt
`cryptography`'s `x509.verify.PolicyBuilder` / `Verifier` for full RFC 5280 path
validation (validity-period nesting, BasicConstraints pathlen, KeyUsage
`keyCertSign`, EKU, name constraints) and optionally revocation — which also
covers the gaps BC-061's resolution note named as out of scope.

## Provenance

Residual of BC-061 (resolved 2026-06-01); the fixing agent flagged
`PolicyBuilder` as the follow-up path. Filed so it is tracked in the backlog
rather than only in the BC-061 resolution note.
