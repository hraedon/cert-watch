# BC-052: /healthz loads full scan_history into memory via list_scan_history()

**Type:** defect
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-31

**Resolved:** 2026-05-31 — /healthz uses targeted SQL (SELECT scanned_at, status FROM scan_history ORDER BY scanned_at DESC LIMIT 1) instead of list_scan_history(). Also uses _connect() cached connection.

## Problem

`views.py:65` calls `list_scan_history(db)` with no `limit` parameter, which fetches every row from the `scan_history` table into Python memory. On a busy fleet with thousands of hosts scanned daily, this table grows unbounded (one row per scan per host per day).

The healthz endpoint is hit frequently by load balancers and Kubernetes liveness probes (typically every 10-30s), so this O(n) scan runs on every probe.

The only data healthz needs from scan_history is the most recent entry (scanned_at, status). A `SELECT ... ORDER BY scanned_at DESC LIMIT 1` would suffice.

Additionally, healthz uses its own `sqlite3.connect()` call (line 57) instead of the cached `_connect()` helper, creating a throwaway connection on every probe.

## Plan

Resolved by Plan 012 §2.2.

## Suggested fix

Replace the `list_scan_history(db)` call with:
```python
with _connect(db) as conn:
    row = conn.execute(
        "SELECT scanned_at, status FROM scan_history ORDER BY scanned_at DESC LIMIT 1"
    ).fetchone()
```

This eliminates the full-table scan and the throwaway connection.
