**Resolved 2026-06-02 (Plan 018 B1 core):** Signing material now lives in an
immutable `SecurityContext` resolved once at startup and carried on
`app.state.security`; the session + CSRF *request path* read it from there
instead of a mutated module global. `create_app(security=, auth_provider=,
settings=)` injects dependencies (tests no longer reload modules — `conftest`
and the `reload_app` fixture are reload-free). The operational symptom was
already mitigated by persisted secrets (see prior note). Two documented
partials remain as a small follow-up, not a defect: (1) the lifespan still
seeds the module globals to back the OAuth-state signing path (which has no
request and is a deployment-shared key); (2) ~8 individual test files still use
inline `importlib.reload` — some legitimately test import/lifespan behavior, the
rest are a mechanical cleanup.

**Update 2026-06-02 (re-assessed):** The operational bug is **already
mitigated** and downgraded to low. The `auth/session.py` import-time
`secrets.token_hex(32)` is only a fallback — the lifespan calls
`config.resolve_or_persist_secret("CERT_WATCH_AUTH_SECRET", data_dir,
".auth_secret")` and `set_signing_key()`, persisting the key to
`data_dir/.auth_secret` (0600) and reading it back on restart, so **sessions
survive restarts** even without the env var. The CSRF secret is derived from
the same persisted auth secret, so it's stable too. cert-watch's deployment is
single-process / single-writer (`Recreate` k8s strategy), so the multi-worker
divergence below isn't the real-world shape; even under `--workers N` all
workers' lifespans read the same `.auth_secret` and converge (narrow first-boot
write race aside). The misleading import-time "sessions invalidated on every
restart" warning was demoted to debug (it contradicted the persistence path).

What remains is **architectural, not a security/operational defect**: the
module-global `_signing_key` / `_csrf_secret_val` are mutated at lifespan time
(`set_signing_key`/`set_csrf_secret`), which is fragile and forces the
reload-based test harness. Plan 018 B1 (SecurityContext on `app.state` +
`create_app` factory) is the clean fix — tracked as a code-quality item, no
longer urgent.

## Problem (original)

When `CERT_WATCH_AUTH_SECRET` is not set, `auth.py` generates an ephemeral
`secrets.token_hex(32)` at import time. Every process restart invalidates all
existing sessions. In multi-worker deployments (gunicorn with workers > 1),
each worker has a different key, so sessions created by one worker are invalid
on another.

The same issue applies to `CERT_WATCH_CSRF_SECRET` in middleware.py.

## Suggested fix

Plan 018 B1 (SecurityContext + create_app factory) moves signing-key state onto
`app.state.security` instead of a mutated module global. Now an architectural
cleanup (the operational symptom is mitigated by persistence), best done in a
dedicated PR with a clean tree.
