# BC-048: Fleet pivot views (`?view=issuer`, `?view=owner`, `?view=renewal_method`) load full inventory into memory

**Type:** defect
**Severity:** medium
**Status:** resolved (2026-05-31)
**Filed:** 2026-05-30
**Updated:** 2026-05-30

## Problem

The fleet dashboard pivot views reuse `list_unified_entries(db)` (see BC-047). `list_unified_entries` fetches every host, every certificate, and every latest scan_history row into Python memory, performs in-memory merging and grouping, and only then hands a grouped list to the template.

At fleet scale this is exactly the O(n) memory issue that BC-044 fixed for `scan_history` and `alerts`, but now hits the main dashboard. This is worse for pivot views because a large fleet may have thousands of hosts and hundreds of thousands of cert rows.

`list_fleet_pivot()` in `database/queries.py` calls `entries = list_unified_entries(db)` and then groups in Python. Since most of the work is in `list_unified_entries`, the pivot view pays the full cost even though it only needs aggregated counts per group.

## Partial mitigation (BC-047 resolution)

The `dashboard()` route now has three paths:
- **Pivot views** — still load full dataset (this breadcrumb).
- **Grouped mode** — loads full filtered set for cross-host fingerprint search.
- **Fast path** (`grouped=0`, no pivot) — uses `list_unified_entries_page()` with offset/limit, so only the page slice is materialized.

For operators not using pivot views, the default dashboard (grouped mode) is still the primary memory concern, but the ungrouped fast path is now efficient.

## Suggested fix

Option A (small): Create `list_unified_entries_page(db, offset, limit, filters...)` that pushes `WHERE`, `ORDER BY`, and `LIMIT` into SQL and only builds dashboard rows for the returned slice.

Option B (larger, better): Write dedicated SQL for each pivot:
- **view=issuer** — group by `friendly_issuer(certificates.issuer)` in SQL (or a new indexed column), aggregate COUNT, MIN(not_after).
- **view=owner** — group by `hosts.owner_name`, aggregate COUNT, MIN(certificates.not_after).
- **view=renewal_method** — group by `hosts.renewal_method`, same.

Each pivot screen only shows one page of groups plus expandable detail rows; SQL-level grouping avoids loading the full dataset.

## Resolution (2026-05-31)

Implemented Option B with lazy loading:

1. **`list_fleet_pivot()`** — Rewritten to use SQL-level `GROUP BY` with `MIN(julianday(...))` for efficient aggregation. Returns summary data only (key, count, worst_urgency, earliest_expiry). The `entries` field is `None`.

2. **`get_pivot_group_entries()`** — New function that loads entries for a single pivot group on demand. Used by the lazy-loading API endpoint.

3. **`GET /api/pivot/{pivot}/{group_key:path}`** — New API endpoint for lazy-loading group entries. Returns filtered unified entries for the requested group.

4. **Dashboard template** — Detail rows show a "Click to load" placeholder. JavaScript fetches entries via the API endpoint on first expand, caches the result, and renders host rows dynamically.

5. **Dashboard route** — Pivot stats computed from targeted SQL (`SUM(CASE WHEN...)`) instead of iterating all entries.
