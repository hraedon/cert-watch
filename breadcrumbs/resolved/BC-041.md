title: Auth test _reload_app causes module state pollution for subsequent tests

**Resolved:** 2026-05-30

## Problem

The `_reload_app()` helper in `tests/test_auth.py` uses `importlib.reload()` on `cert_watch.config` and `cert_watch.app`, which replaces module globals like `_signing_key`, `app.state.auth_provider`, and `app.state.settings`. When auth tests that use `_reload_app` (e.g., `test_auth_enabled_redirects_to_login`, `test_authz_gate_denies_user_without_group`, local admin integration tests) run before other test modules like `test_dashboard.py` or `test_rest_api.py`, those modules' tests get 401 responses because the reloaded app still has auth middleware active.

This is the root cause of ~39 test failures when running `pytest tests/` as a whole, even though each module passes individually.

The root cause was class-identity drift: `importlib.reload(cert_watch.auth)` creates new `NoAuthProvider`/`LocalAdminProvider` classes, but `cert_watch.middleware` still held references to the OLD classes. The `isinstance(auth, NoAuthProvider)` check in `auth_middleware` returned `False` for a perfectly valid `NoAuthProvider` instance, because it was an instance of the wrong class.

## Resolution

Added an `autouse` fixture `_restore_modules` in `tests/test_auth.py` that saves and restores the module dicts of `cert_watch.config`, `cert_watch.auth`, and `cert_watch.app` after every test. This prevents:
1. Class-identity drift (middleware's `NoAuthProvider` reference matches the actual class)
2. Auth provider leaking (app.state.auth_provider and settings are restored)
3. Config leaking (module-level Settings/variables restored)

The `test_local_admin_disabled_when_unset` test at line 775 also reloads `cert_watch.auth` directly — the autouse fixture covers this too.

All 349 tests now pass as a full suite.
