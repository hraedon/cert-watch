---
identifier: BC-083
title: Open-by-default auth posture (two permissive defaults stack)
kind: security
status: resolved
severity: medium
---

**ChatGPT review #1/#3 root.** With no `AUTH_PROVIDER`, all routes are open
(backward-compat, documented, startup warning). Combined with private-scanning-by-
default, an out-of-the-box LAN-exposed instance lets anyone add a host and enumerate
internal TLS certs. The setup wizard is a convenience redirect, not an auth gate, and
is bypassed if any host already exists. This — not the private-IP default — is the
real "any user can add a host" gate.

**Decided 2026-06-02 — agreed direction (not yet implemented):** move to
**secure-by-default with an explicit opt-out**. When no `AUTH_PROVIDER` is
configured and `CERT_WATCH_ALLOW_UNAUTH` is unset, the app should **refuse to
serve on a non-loopback bind** (or hard-force the `/setup` wizard regardless of
whether hosts already exist) instead of silently serving open. Loopback binds
stay open (local dev). Homelab / air-gapped operators set
`CERT_WATCH_ALLOW_UNAUTH=1` once. This converts a silent security downgrade into
a loud, one-line-fixable startup error.

Implementation notes:
- Enforce in the lifespan / a startup check (it already computes `needs_setup`
  and the non-loopback warning — promote the warning to a hard stop unless
  `allow_unauth`). The check already has the bind host + auth provider + db.
- The current gap is the `host_count == 0` condition on `needs_setup`: an
  upgraded instance with existing hosts skips setup. Drop that condition (or
  gate purely on "no auth configured and not allow_unauth").
- **Breaking change** for existing open deployments on upgrade — must be
  release-noted + version-bumped, with a precise "set AUTH_PROVIDER, run /setup,
  or set CERT_WATCH_ALLOW_UNAUTH=1" message. Keep loopback exempt so local dev
  and the test suite (which sets `CERT_WATCH_ALLOW_UNAUTH=1`) are unaffected.
