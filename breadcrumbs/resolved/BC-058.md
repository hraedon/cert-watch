# BC-058: OAuth userinfo fallback bypasses ID token verification

**Type:** defect
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-31

**Resolved:** 2026-05-31 — When ID token verification fails, auth is rejected instead of falling back to userinfo. Userinfo is only used when no ID token was present in the token response.

## Problem

In `auth.py:609-619`, when ID token verification fails or returns no username, the OAuth provider falls back to the userinfo endpoint:

```python
if not username:
    userinfo_endpoint = endpoints.get("userinfo_endpoint", "")
    if userinfo_endpoint:
        resp = client.get(userinfo_endpoint)
        if resp.status_code == 200:
            info = resp.json()
            username = (
                info.get("preferred_username")
                or info.get("email")
                or info.get("sub", "")
            )
```

The userinfo endpoint response is **not** verified against the ID token claims. This means:
- If the ID token verification failed due to a key rotation, the userinfo fallback still succeeds, potentially with stale or different claims.
- The userinfo endpoint is fetched using the OAuth access token, which could have been issued to a different client if the token endpoint was compromised.
- There is no signature verification on the userinfo response — it's trusted over HTTPS only.

The more concerning scenario: if `_verify_id_token()` returns `None` (line 600-601), the code falls through to userinfo. An attacker who can intercept the token response (e.g., via a compromised proxy) could strip the ID token, causing verification to fail, and then the system falls back to userinfo which may return different claims.

## Plan

Resolved by Plan 012 §3.2.

## Suggested fix

When an ID token is present, its claims should be authoritative. The userinfo fallback should only be used when no ID token was provided in the token response. If ID token verification fails, the authentication should fail rather than silently falling back.
