## Problem

Plan 020 S4 tried to harden the CSP by replacing `script-src 'self'
'unsafe-inline'` with a per-request nonce. It shipped and broke the UI: every
button stopped working and 4 e2e tests went red (`.cw-slide.on` never appears,
upload/add-host rows time out). Root cause: the templates use ~24 inline
event-handler attributes (`onclick=`, `onchange=` in `dashboard.html`,
`certificate_detail.html`, `settings.html`, `base.html`). CSP nonces only
whitelist `<script>` blocks — they do **not** apply to inline event-handler
attributes — so removing `'unsafe-inline'` blocked all of them.

S4 was reverted (CSP back to `script-src 'self' 'unsafe-inline'`; nonce
generation + template `nonce=` attributes removed) to restore green. So
injected `<script>` is still permitted by CSP; `escHtml()` + the urgency-class
whitelist remain the XSS mitigations.

## Suggested fix

1. Replace inline `onclick=`/`onchange=` handlers with `data-*` attributes +
   delegated `addEventListener` listeners inside a single nonce'd `<script>`
   (e.g. `data-action="open-slide"`, `data-group="{{ id }}"`, `data-tab="scan"`,
   `data-href="/certificates/{{ id }}"`).
2. Re-introduce the per-request nonce in `security_headers_middleware` and drop
   `'unsafe-inline'` from `script-src`.
3. Add an e2e assertion that a button-driven flow (Add host → slide opens)
   still works under the nonce CSP, so the regression can't recur.

## Progress (2026-06-02) — enablement landed; flip still pending

Front-loaded the de-risking ahead of the design work so new screens are
compliant by construction:

- **Nonce plumbing re-introduced (safely).** `CSPNonceMiddleware` (pure ASGI,
  outermost) issues a per-request nonce into `scope['state']`; templates read it
  via `{{ request.state.csp_nonce }}`. (Pure ASGI, not `BaseHTTPMiddleware`,
  because `request.state` set in a `BaseHTTPMiddleware` doesn't reach the
  endpoint — task isolation.) The nonce is **not yet in the CSP header**:
  `_build_csp(nonce)` still emits `'unsafe-inline'` and ignores the nonce,
  because adding it now would make browsers ignore `'unsafe-inline'` and break
  the remaining inline handlers (the original S4 regression). The flip is now a
  one-line change in `_build_csp`.
- **Guardrail added.** `tests/test_no_inline_handlers.py` is a ratchet: per-file
  inline-handler budgets that only go down; new handlers (or a new template with
  any) fail the build. Current budgets: `dashboard.html` 16, `certificate_detail.html`
  4, `settings.html` 3 — **23 left** (was 24).
- **Reference conversion.** `base.html` is at **0** (`dismissHealth` → `data-*` +
  `addEventListener`); its three `<script>` blocks are nonce-stamped — the
  copy-paste pattern for the rest.

**Remaining:** convert the 23 handlers in `dashboard.html` / `certificate_detail.html`
/ `settings.html` (lowering the budgets to 0 as each screen is built/reconciled),
then flip `_build_csp` to `script-src 'self' 'nonce-{nonce}'` and drop
`'unsafe-inline'`. `style-src 'unsafe-inline'` stays (inline `style=` custom
properties can't be nonced).
