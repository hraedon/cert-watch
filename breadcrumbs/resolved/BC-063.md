# BC-063: Daily scheduled scan does not pin IP — DNS-rebinding hardening (BC-053) bypassed on the primary scan path

**Type:** defect
**Severity:** medium
**Status:** resolved (2026-06-01)
**Filed:** 2026-05-31

## Problem

BC-053 added `pinned_ip` to `scan_host` to close the resolve-then-connect
DNS-rebinding window. The *manual* and *bulk* host routes correctly pass it:

- `routes/hosts.py:116,140` (add_host), `216,254` (bulk import), `318,332`
  (scan_host_now) all call `_is_blocked_host_check(...)` and pass the returned
  `pinned_ip` into `scan_host(..., pinned_ip=...)`.

But the **daily scheduled scan — the primary operational path — does not**:

`app.py:97-106` builds `scan_fn=lambda host, port: scan_host(host, port,
verify=..., allow_private=..., dns_servers=...)` with NO `pinned_ip`, and
`run_scan_now` (scheduler.py:185-224) just calls `scan_fn(host, port)`. So on
every nightly re-scan the hostname is resolved inside `_open_tls_connection` and
connected by name, leaving the TOCTOU/rebinding window BC-053 claims to close
fully open for the path that runs most often and unattended.

The scheduled path also does no SSRF blocklist check of its own beyond what
`_resolve_host` does at connect time (which is the rebindable resolution).

## Suggested fix

- In the scheduler `scan_fn`, run `_is_blocked_host_check` (or resolve+vet once)
  per host and pass the vetted IP as `pinned_ip`, matching the route paths.
- Better: resolve-and-pin inside `scan_host` itself so every caller is hardened
  by default and the parameter becomes an override, not a requirement.
- See BC-062: even when `pinned_ip` is passed, the HSTS probe re-resolves by name
  with no blocklist, so it must be fixed too for the pin to be meaningful.
