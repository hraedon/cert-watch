# BC-033: Dashboard grouping by certificate fingerprint with per-host status

**Type:** feature
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-29

## Problem

Wildcard and SAN certificates deployed on multiple hosts create duplicate rows in the dashboard. If `*.internal.example.com` is on 10 hosts, the dashboard shows 10 identical rows. This causes:

1. **Clutter** — hard to see the actual cert landscape when duplicates dominate.
2. **No blast radius visibility** — you can't tell at a glance how many hosts a cert expiry affects.
3. **Alert noise** — 10 separate alerts for the same cert, when one with a host list would suffice.
4. **Renewal coordination gap** — after renewal, you don't have a quick list of hosts that need the new cert deployed.

## Suggested fix

**Group by fingerprint in the dashboard query.** When multiple host entries share the same leaf certificate fingerprint, render them as a single expandable row.

### Collapsed view (default)
- Show the cert subject, issuer, expiry, urgency once.
- Show a **host count badge** (e.g., "5 hosts").
- Show a **status summary indicator** reflecting the distribution of per-host statuses. Options:
  - A compact "x/y healthy" pill (e.g., "3/5 healthy, 2 expiring") with color gradient (green/yellow/red).
  - A small segmented bar or dot cluster where each dot's color represents one host's status.
  - The summary should make it immediately obvious if any host in the group is in a different state (e.g., 9 healthy + 1 critical should visually stand out).

### Expanded view (click to expand)
- List each `hostname:port` individually.
- Each host shows its **own** scan status, last scan time, and any scan errors — not the cert's global status.
- This is important because hosts can drift: one host may have a stale cert (failed renewal deployment), a scan failure, or a different cert entirely (fingerprint mismatch after partial rollout).
- Include per-host actions (rescan, delete).

### Implementation notes
- The underlying data model stays per-host (no schema change). Each host has its own cert entry and scan history.
- Grouping is a read-time dedup by `fingerprint_sha256` in `list_unified_entries()`.
- The collapsed/expanded state is a UI toggle (JavaScript), not a server round-trip.
- The status summary color should degrade gracefully: if all hosts share the same status, show a solid color; if mixed, show a gradient or the "x/y" label.

### Rolling renewal behavior
During a rolling cert renewal, hosts will have different fingerprints for the same cert (some on the new cert, some still on the old). This creates two fingerprint groups for the same domain (e.g., two "*.hraedon.com" entries). This is **correct behavior** — it shows the real state of the infrastructure and surfaces stuck cutover deployments.

To make this useful rather than confusing:
- Use the `replaces_cert_id` lineage to label groups: the old cert shows "pending replacement" or "previous cert", the new cert shows "newly deployed" with a timestamp.
- If two groups share overlapping SANs, show a visual connection (banner, shared row, or color link) indicating "cutover in progress."
- The cutover window should be short (minutes/hours for a coordinated rollout). If two groups persist for days, that's a signal something is stuck — the dashboard should make this obvious, not hide it.

## Resolution (2026-05-29)

Implemented core fingerprint grouping (collapsed view with expand/collapse). Changes:

### Backend (`src/cert_watch/database/queries.py`)
- Added `fingerprint_sha256` and `san_dns_names` to `_build_dashboard_rows()` output and pending/uploaded entries
- Added `group_entries_by_fingerprint()` function that groups scanned entries sharing the same leaf fingerprint into `kind == "grouped"` entries with a `hosts` sub-list, urgency summary, and host count

### Route (`src/cert_watch/routes/views.py`)
- Added `grouped` query parameter (default `1`) to dashboard route
- Search matches across all hosts within a group
- Source filter includes `"grouped"` kind when filtering by `"scanned"`

### Template (`src/cert_watch/templates/dashboard.html`)
- Grouped entries render as a clickable header row with: cert subject, "N hosts" badge, "x/y healthy" status summary, chain status chips, expiry bar
- Clicking the header toggles expandable sub-rows listing each hostname:port with per-host urgency pill, scan status, last scan time, and rescan button
- Added `chevron-down` SVG icon with rotation animation on expand
- JS `toggleGroup()` for client-side expand/collapse (no server round-trip)

### CSS (`src/cert_watch/static/css/tokens.css`)
- Added styles for `.cw-group-header`, `.cw-group-chevron`, `.cw-chip-hosts`, `.cw-group-hosts-cell`, `.cw-host-row`, `.cw-host-row-info`, `.cw-host-row-actions`

### Tests (`tests/test_dashboard.py`)
- `test_group_entries_by_fingerprint` — unit test for grouping logic (shared FP → grouped, unique FP → solo, uploaded → solo)
- `test_group_entries_mixed_urgency` — group urgency reflects worst host
- `test_dashboard_grouped_by_fingerprint` — integration test with real DB + HTTP
- `test_dashboard_grouped_search_matches_host` — search matches hosts inside a group
- `test_dashboard_grouped_disabled` — `grouped=0` shows individual rows

### Not yet implemented (deferred)
- Rolling renewal visual connection (overlapping SANs, "cutover in progress" banner)
- Per-host delete action in expanded view
- Alert deduplication across grouped hosts
