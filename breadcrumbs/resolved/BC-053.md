# BC-053: DNS rebinding window between SSRF check and TLS connection

**Type:** defect
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-31

**Resolved:** 2026-05-31 — _is_blocked_host_check() now returns (error, pinned_ip) tuple. scan_host() accepts pinned_ip parameter, passing it through to _open_tls_connection() and _scan_via_openssl() to prevent DNS rebinding between SSRF check and TLS connection.

## Problem

The SSRF protection in cert-watch has a TOCTOU (time-of-check-time-of-use) gap:

1. `_is_blocked_host_check()` in `routes/hosts.py:38` resolves the hostname and checks the IP against the blocklist.
2. `scan_host()` in `scan.py:360` resolves the hostname *again* when establishing the TLS connection.

Between steps 1 and 2, an attacker-controlled DNS server can return a different IP (DNS rebinding). The first resolution passes the SSRF check (returns a public IP), but the second resolution returns an internal IP (e.g., `10.0.0.1`, `169.254.169.254`).

This is especially relevant for:
- Manual "Add host" flow (SSRF check then immediate scan)
- CSV bulk import (SSRF check per row, then concurrent scans)
- Scheduled scans (no SSRF re-check at scan time)

The scheduled scan path (`scheduler.py` → `run_scan_now()`) does NOT perform any SSRF check at all — it scans whatever is in the `hosts` table. If an attacker can add a host that resolves to a public IP at add-time, then changes DNS to point to an internal IP, all subsequent scheduled scans will hit the internal address.

## Plan

Resolved by Plan 012 §3.4.

## Suggested fix

Option A (recommended): Resolve the hostname once and pass the resolved IP to `scan_host()`. The SSRF check and the connection should use the same resolution.

Option B: Re-check the resolved IP inside `_open_tls_connection()` / `_resolve_host()` — this already happens (`_is_blocked_ip` is called in `_resolve_host`), but the DNS resolution itself can return a different IP than what was checked. The check in `_resolve_host` IS the re-check, so the actual risk is only in the `add_host` flow where the pre-check and the scan use different resolutions.

Option C: Pin the resolved IP from the SSRF check and pass it directly to the TLS connection (skip re-resolution).
