identifier: WI-066
title: "Non-ASCII tag case-folding parity gap: dashboard + match-preview (SQLite LIKE) vs alert engine (Python casefold)"
kind: bug
status: resolved
severity: low
---

## Problem
Tag matching is implemented two ways that disagree on non-ASCII casing:

- **Alert engine** (`cert_watch.tags.tags_match`, `alerts.py`): Python
  `.casefold()`, Unicode-aware. "İ" folds to "i̇", "ß" to "ss", etc.
- **Dashboard scope filter + alert-group match-preview**
  (`database/dashboard._add_effective_tag_filter`,
  `routes/settings/alert_groups._match_preview`): SQLite `LIKE`, which is
  ASCII-case-insensitive only.

For tags that differ only by non-ASCII casing (Turkish dotless-i, German ß,
Cyrillic look-alikes), the dashboard and the WI-060 preview can **undercount**
relative to what the alert engine actually routes.

## Impact
Low for typical SMB estates (ASCII tag corpora). The parity gap becomes a
correctness issue for any deployment using non-ASCII tags: the UI shows a
different match count than the engine delivers, which is exactly the
"preview lies to the operator" class WI-060 exists to prevent (for ASCII it
does not lie; for non-ASCII it can).

## Found
Adversarial review of WI-060 (finding H1), 2026-06-17. Pre-existing in the
dashboard; WI-060's preview inherited it. Documented inline in the
`_match_preview` docstring.

## Fix sketch
Make the SQL-side filters Unicode-case-insensitive to match `tags_match`.
Options: (a) apply `COLLATE NOCASE` (ASCII only — insufficient), (b) store a
`casefold()`-ed normalized tag column and match on that, or (c) move the
dashboard/preview tag filtering to Python over a prefetched cert set (loses
the SQL-side pagination win). Non-trivial because tag matching is load-bearing
across dashboard pagination, scope enforcement, and now the preview.

## Scope note
Not in v0.9.3 (documented inline; low impact; shared with the dashboard).
Track here.

## Resolution
Fixed 2026-06-17. Took option (f) (schema-free) over option (b) (normalized
column) to avoid a schema migration + write-path sweep for a low-severity item:
- Registered a `cw_casefold` SQL function on every `_connect` connection
  (`database/connection.py`) that exposes Python `str.casefold()` to SQL.
  Inert unless referenced — zero impact on existing queries.
- Wrapped both sides of the tag-filter `LIKE` in `cw_casefold(...)` in
  `_add_effective_tag_filter`, `_add_grouped_effective_tag_filter`, and
  `_match_preview`. Since both sides are casefolded, SQLite's ASCII-CI `LIKE`
  compares the normalized forms exactly, giving Unicode-case-insensitive
  parity with the `tags_match` engine (ß/SS, Turkish dotless-i, etc.) while
  preserving SQL-side pagination. Tag dedup now keys on `casefold()` too.
- Removed the "Non-ASCII limitation" note from the `_match_preview` docstring.

No schema migration, no write-path changes, no normalized column to keep in
sync. Regression tests: `tests/test_alert_group_match_preview.py` and
`tests/test_tag_scoped_access.py` assert the ß/SS case-variant now matches in
both the preview/dashboard SQL and `tags_match` (break-and-watch confirmed the
test catches the gap: without `cw_casefold` the preview returns 0).
