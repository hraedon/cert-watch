**Update 2026-06-02 (Plan 018 B2):** Partially resolved, downgraded to low.
The ungrouped dashboard path now uses `list_dashboard_page()` with true
SQL-level WHERE/ORDER BY/LIMIT/OFFSET (no full-table load). The *grouped*
default view still materializes the filtered inventory via
`list_dashboard_grouped_page()` because worst-urgency grouping depends on
chain-status computed in Python and cross-host fingerprint search needs the
full filtered set — a faithful SQL `GROUP BY` risks urgency-semantics
divergence. Acceptable at the target SMB scale (hundreds of certs; see Plan
018's scope note). Revisit only if the grouped view becomes a measured
bottleneck.

## Problem

`_list_unified_entries_raw()` in queries.py executes `SELECT * FROM
certificates`, `SELECT * FROM hosts`, and `SELECT * FROM trust_anchors`
with no LIMIT. The grouped dashboard path (default view) loads the entire
result set into Python before grouping. At fleet scale (thousands of certs),
this causes significant memory pressure and latency.

Plan 018 B2 covers the fix (purpose-built dashboard queries with SQL-level
GROUP BY). This breadcrumb tracks the known issue until that plan is
implemented.

## Suggested fix

Replace callers with SQL-level `list_dashboard_page()` and
`list_dashboard_grouped_page()` as specified in Plan 018 B2.
