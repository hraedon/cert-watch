# BC-064: TLS verification off by default; recorded cert facts are attacker-influenceable and "verified" is never persisted

**Type:** design
**Severity:** low
**Status:** open
**Filed:** 2026-05-31

## Problem

`CERT_WATCH_TLS_VERIFY` defaults to `0` (config.py:132). With the default,
`_open_tls_connection` (scan.py:237-243) sets `check_hostname=False` and
`verify_mode=ssl.CERT_NONE`. This is a defensible choice for an *inventory* tool
(you want to record self-signed / expired / internal certs), but the
consequences are not surfaced:

- The dashboard expiry dates, SANs, subject, and posture grade are computed from
  whatever certificate the endpoint presents. A network MITM can feed cert-watch
  arbitrary cert data and it is stored and graded as fact.
- `store_scanned` passes `chain_valid=None` (scan.py:612), so even when chain
  trust *could* be evaluated, no per-scan "this cert validated / hostname
  matched" boolean is persisted. An operator cannot distinguish "verified chain"
  from "accepted blindly" in the UI.

## Suggested fix

- Persist a per-scan `verified` boolean (chain trusted AND hostname matched),
  independent of the inventory function, and display it.
- Consider verifying by default and recording failures as a posture finding,
  rather than disabling verification globally.
- Document clearly that, with the default, all recorded certificate facts are
  untrusted / influenceable by anyone on the network path.
