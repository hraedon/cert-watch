**Resolved:** 2026-05-29

## Resolution

Replaced the double-connection approach (Python TLS + openssl s_client fallback) with a single-connection strategy. On Python < 3.13, `_scan_host_once()` now uses `_scan_via_openssl()` which makes one `openssl s_client` call to get both leaf and chain. If openssl is unavailable, it falls back to the Python TLS connection for leaf-only. On Python >= 3.13, the native `get_unverified_chain()` / `get_verified_chain()` path is used (unchanged). This eliminates the second TLS connection that could hit a different backend behind a load balancer.

Note: BC-028 was a duplicate of BC-027 (same root cause). BC-027 should also be resolved.

## Original Problem

In `scan.py:_get_chain_via_openssl()`, the openssl s_client fallback for Python 3.12 chain extraction connected to the host a second time. This meant behind a load balancer with multiple backends, the second connection could hit a different backend with a different certificate, and it doubled the scan time for affected hosts.
