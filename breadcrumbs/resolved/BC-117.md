## Problem

When `CERT_WATCH_CHECK_REVOCATION=1`, posture grading probes the OCSP responder
and CRL distribution-point URLs **taken from the certificate's own AIA/CRL
extensions** (`posture.py:_check_ocsp_reachable` / `_check_crl_reachable`,
lines 73-91) via `urllib.request.urlopen` with no blocklist check.

The URLs are attacker-influenceable: any host you scan can present a cert whose
AIA points at `http://10.0.0.5/` or `169.254.169.254`, and posture grading runs
on **uploaded** certs too (`routes/certificates.py:293`), so a malicious uploaded
cert is also a vector. It's a blind SSRF (only a reachability bool is returned),
but it can reach internal endpoints and port-scan by timing.

Mitigations that cap severity: **off by default**, and blind. Hence Medium
(High when the feature is enabled).

## Suggested fix

Validate each OCSP/CRL URL through the same SSRF-safe opener as [[BC-116]] before
probing (resolve → check `scan._is_blocked_ip`, honoring `allow_private` /
`allowed_subnets`; `http(s)` scheme allowlist; validate redirects too). When a URL
is blocked by policy, emit a clear posture Finding ("endpoint blocked by SSRF
policy") rather than a generic unreachable. `evaluate_posture` /
`check_revocation_endpoints` need to accept and thread `allow_private` /
`allowed_subnets` (resolve from settings via `Settings.from_env_with_kv`).

Add a test: a cert with a private/loopback AIA URL is refused, not probed.
Scoped for 0.5.0 — see Plan 023.
