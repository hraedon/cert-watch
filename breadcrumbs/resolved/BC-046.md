# BC-046: init_schema called redundantly on every repository instantiation

**Type:** quality
**Severity:** low
**Status:** resolved
**Filed:** 2026-05-30
**Resolved:** 2026-05-30

## Problem

Each `SqliteXxxRepository.__init__` calls `init_schema()`. While the second-call fast path (`_initialized_paths` set in `schema.py`) avoids re-running DDL, it still takes a set-membership check per instance. Route handlers commonly create 2-3 repo instances per request (e.g., `SqliteHostRepository` + `SqliteCertificateRepository` + `SqliteTrustAnchorRepository`), each paying this overhead.

More importantly, any code path that creates repos before the lifespan `init_schema` runs could race with the migration runner, since the migration lock is not held during the set-membership check.

## Resolution

- Removed `init_schema()` calls from `SqliteCertificateRepository`, `SqliteAlertRepository`, `SqliteTrustAnchorRepository`, and `SqliteHostRepository` constructors (`repo.py`).
- Repository constructors now only store the `db_path`.
- All callers outside of `app.py:lifespan` that need the schema (e.g. test fixtures, CLI commands) call `init_schema` explicitly.
- This eliminates per-request schema-check overhead and removes the race window.
