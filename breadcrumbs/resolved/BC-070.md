**Resolved:** 2026-06-02 — `check_csrf()` no longer reads `_csrf_token` from
the query string; only the `x-csrf-token` header and the `_csrf_token` form
field are accepted. Regression test in `tests/test_csp_csrf_hardening.py`
(`test_csrf_query_param_token_rejected`).

## Problem

`check_csrf()` in middleware.py accepts `_csrf_token` in the query string as
a fallback after checking the header and form field. Query string tokens are
logged in browser history, server access logs, and leaked via Referer headers.
While no current route uses query-param CSRF on state-changing operations (all
use form fields), the fallback exists and weakens the double-submit pattern.

## Suggested fix

Remove the query-parameter fallback for CSRF on POST/PUT/DELETE routes, or
restrict it to GET-only endpoints that perform non-mutating operations.
