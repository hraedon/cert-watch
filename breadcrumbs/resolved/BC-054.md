**Resolved:** 2026-05-31 — Empty strings now treated as unset for CERT_WATCH_AUTH_SECRET and CERT_WATCH_CSRF_SECRET. Uses explicit None check with fallback to secrets.token_hex(32).

## Problem

In `auth.py:30`:
```python
_signing_key = read_secret("CERT_WATCH_AUTH_SECRET") or secrets.token_hex(32)
```

If `CERT_WATCH_AUTH_SECRET` is set to an empty string `""`, `read_secret()` returns `""` (truthy for the `or` check), and `_signing_key` becomes `""`. An empty HMAC key means all session signatures use an empty key, which is cryptographically weak.

The same issue affects `middleware.py:25`:
```python
_CSRF_SECRET = os.environ.get("CERT_WATCH_CSRF_SECRET") or secrets.token_hex(32)
```

An empty `CERT_WATCH_CSRF_SECRET=""` results in `_CSRF_SECRET = ""`.

## Plan

Resolved by Plan 012 §1.1.

## Impact

- Session tokens signed with an empty key can be forged by anyone who knows the algorithm (the code is open source).
- CSRF tokens signed with an empty key can be forged, bypassing CSRF protection.

## Suggested fix

Treat empty strings the same as unset:
```python
_signing_key = read_secret("CERT_WATCH_AUTH_SECRET") or None
if not _signing_key:
    _signing_key = secrets.token_hex(32)
    logger.warning(...)
```

Same pattern for `_CSRF_SECRET`.
