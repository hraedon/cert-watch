**Resolved:** 2026-06-01

## Resolution

Approach: **real signature verification** (not the rename fallback). Chain trust
grading now requires that every link is cryptographically signed by the next,
not merely that issuer/subject Names line up. A forged intermediate (names link,
key does not) is now caught and graded `invalid`.

Changes in `src/cert_watch/cert_chain.py`:

- `_load_x509()` (cert_chain.py:141-153) — parse a `Certificate`'s `raw_der`
  into an `x509.Certificate`; returns None defensively rather than raising.
  (`raw_der` is a NOT NULL column, populated for both scanned and DB-loaded
  certs, so this normally succeeds.)
- `_is_signed_by()` (cert_chain.py:156-176) — verifies a child is actually
  signed by an issuer's key via `x509.verify_directly_issued_by` (cryptography
  >=40; pyproject pins >=43). Catches InvalidSignature / name-mismatch /
  unsupported-key and returns False.
- `validate_chain_signatures()` (cert_chain.py:179-189) — verifies every
  adjacent pair in the chain by signature; returns None for len < 2 (mirrors
  `validate_chain_order`).
- `_is_signature_anchored_by_user()` (cert_chain.py:192-211) — anchor match is
  now fingerprint identity OR a verified signature by the anchor key, not name
  linkage.
- `chain_status()` (cert_chain.py:243-281) — after the existing name-order
  check, now also requires `validate_chain_signatures(full) is True`; a
  name-matching-but-forged chain returns `invalid`. `private` uses the
  signature-anchored check. A self-signed root must actually self-sign
  (`_is_signed_by(last, last)`) to grade `public`. Status string VALUES are
  unchanged (self-signed/unknown/invalid/private/public/incomplete), so
  templates, `posture.py`, and `queries.py` consumers do not regress.
- `validate_chain_with_anchors()` and `is_anchored_by_user()` kept (used only by
  tests; no production callers) but docstrings now state plainly they are
  name-linkage only and NOT a trust decision — the trust decision is
  `chain_status()`.

Inventory behavior preserved: self-signed / internal / expired certs still parse,
store, and scan exactly as before. Only the *trust grade* changed — a self-signed
cert still grades `self-signed` (honest, not "trusted"), and a forged chain grades
`invalid` instead of `public`/`private`.

Regression tests in `tests/test_cert_chain.py`:

- `test_validate_chain_signatures_genuine` — genuine triplet verifies True.
- `test_validate_chain_signatures_forged` — names link but forged intermediate
  fails signature verification.
- `test_chain_status_forged_intermediate_not_public` — forged chain grades
  `invalid`, never `public`/`private`, even with the real root uploaded as an
  anchor.
- `test_chain_status_genuine_chain_private_with_anchor` — genuine chain still
  grades `private` when anchored.
- `test_chain_status_genuine_chain_public_root` — genuine chain to its
  self-signed root still grades `public`.
- `test_self_signed_cert_still_scans_no_inventory_regression` — self-signed cert
  still parses, retains `raw_der`, grades `self-signed` (no inventory regression).

Suite: `409 passed, 8 deselected` (full `pytest -q`). `ruff check` clean on
changed files.

## Residual gap (honest)

This implements **pairwise signature verification** along the presented chain plus
a signature check to the trust anchor / self-signed root. It is NOT full RFC 5280
path validation: it does not check validity-period nesting, BasicConstraints
pathlen, KeyUsage `keyCertSign`, name constraints, EKU, or revocation (CRL/OCSP).
For cert-watch (an inventory tool with TLS verification off by default, BC-064)
this closes the over-claim — a forged intermediate can no longer be reported as
trusted — but a chain that is signature-valid yet violates other RFC 5280
constraints could still grade `public`/`private`. A follow-up could adopt
`cryptography`'s `x509.verification.PolicyBuilder` for full path validation if a
stronger guarantee is needed.

## Original Problem

The chain trust logic in `cert_chain.py` was purely name/structure based; no
cryptographic signature was ever checked. `validate_chain_order` compared only
DER-encoded Name bytes; `validate_chain_with_anchors` and `is_anchored_by_user`
matched names/fingerprints; `chain_status` (used by `scan.py:657` and feeding
`posture.py` grading) called only the name walks. A chain assembled from certs
whose Subject/Issuer Names line up but whose keys are forged/mismatched was
reported as `public`/`private`/valid — an over-claim: a "chain validation"
feature that could not detect a forged intermediate gave false assurance.
