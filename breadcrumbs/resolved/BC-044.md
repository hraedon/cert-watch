---
identifier: BC-044
title: Unbounded scan_history and alerts queries load all rows into memory
kind: defect
status: resolved
severity: medium
---

**Resolved:** 2026-05-30

## Problem

`list_scan_history()` and `list_alerts_with_subject()` in `database/queries.py` execute `fetchall()` with no LIMIT clause. The `scan_history` table grows by one row per host per scan cycle — after months of daily scans of hundreds of hosts, these queries return tens of thousands of rows loaded entirely into Python memory. Both the `/scan-history` and `/alerts` HTML views call these with no pagination.

Similarly, `api_list_hosts` and `api_list_alerts` in `routes/api.py` load the full dataset via `list_all()` or `list_alerts_with_subject()` before slicing in Python, discarding most rows for any non-first page. This is O(n) memory per request.

## Resolution

- Added `page`/`limit` parameters and SQL `LIMIT`/`OFFSET` clauses to `list_scan_history` and `list_alerts_with_subject` in `queries.py`.
- Added `_total_alerts` and `_total_scan_history` helper functions to get counts efficiently without loading rows.
- Updated `/alerts` and `/scan-history` HTML views in `routes/views.py` to paginate at 50 entries per page, using the new helpers.
- Updated `/api/alerts` and `/api/hosts` to paginate at the SQL level.
- Added `SqliteHostRepository.list_page()` / `count_all()` methods in `repo.py` for SQL-level host pagination.
- Updated `healthz` to query only the latest scan row (subquery) instead of loading the full table.
- Pagination nav rendering added to `alerts.html` and `scan_history.html` templates.
