## Problem
`events.py:_deliver_webhook` constructs the delivery `WebhookConfig` without
passing `allow_private` / `allowed_subnets`, so those fields take the
`WebhookConfig` dataclass defaults (`allow_private=False`, `allowed_subnets=()`).
The alert-delivery path (`Settings.build_webhook_config`) and the SMTP path
(`Settings.build_alert_config`, wired in v0.9.3) both honor
`CERT_WATCH_ALLOW_PRIVATE_IPS`; the event-stream path does not.

## Impact
An operator who sets `CERT_WATCH_ALLOW_PRIVATE_IPS=1` (the default) and points
the **event-stream** webhook at an internal host (e.g.
`http://mattermost.internal:8065/hooks/xxx`) has those deliveries silently
**blocked** by SSRF policy, even though their explicit operator setting
permits private targets and even though alert-delivery webhooks to the same
host would succeed. There is no error surfaced — the webhook just doesn't fire.

Same class of bug as the alert-SMTP/`build_alert_config` gap closed in v0.9.3,
in a different subsystem (event streaming vs alert delivery).

## Found
Adversarial review of the v0.9.3 SMTP SSRF change (finding F4), 2026-06-17.
Pre-existing — not introduced by the SMTP work.

## Fix sketch
`EventStreamConfig` (events.py:50) carries no SSRF policy fields and is
hydrated from `kv_store`, not `Settings`. The fix is to thread
`allow_private`/`allowed_subnets` into `_deliver_webhook`:
either add the fields to `EventStreamConfig` (+ serialize in `to_json` /
`from_json` + populate from `Settings` where the stream is started / events are
emitted), or pass them as explicit params through `emit_event` →
`_deliver_webhook` from the call sites that hold `Settings` (scheduler.py
emit_event callers, and any route-level emit). Mirror `Settings.build_webhook_config`.

## Scope note
Not in v0.9.3 scope (separate subsystem; deserves its own focused fix + tests).
Track here.

## Resolution
Fixed 2026-06-17. `events._deliver_webhook` now resolves the SSRF allow-policy
live at delivery time via `_resolve_ssrf_policy(db_path)` and passes
`allow_private`/`allowed_subnets` into the `WebhookConfig`:
- `allow_private` from `CERT_WATCH_ALLOW_PRIVATE_IPS` (env-only).
- `allowed_subnets` env (`CERT_WATCH_ALLOWED_SUBNETS`) winning over the
  kv_store allowlist set by the setup wizard.

Resolved live (not snapshotted into `EventStreamConfig`) so a policy change
takes effect without re-saving the event-stream config, and so it works in the
background delivery thread / scheduler path that has no request/app context.
Mirrors `Settings.build_webhook_config`. Regression tests in
`tests/test_events.py::TestDeliverWebhookSSRFPolicy` cover env-disable,
default-allow, kv allowlist, and env-wins-over-kv.
