# BC-056: /metrics endpoint exposes internal infrastructure without authentication

**Type:** defect
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-31

**Resolved:** 2026-05-31 — CERT_WATCH_METRICS_TOKEN env var gates /metrics with bearer token auth. check_metrics_token() validates Authorization header. Backward compatible — no token = open (current behavior).

## Problem

`views.py:324` defines the `/metrics` endpoint which is publicly accessible (listed in `is_public_path()` via `path.startswith("/metrics")`). It returns a Prometheus-format text response containing:

- All hostnames being monitored (including internal/private hosts)
- Certificate subjects (may include internal service names)
- Days until expiry per cert (operational intel)
- Total host and certificate counts

When `AUTH_PROVIDER` is set, `/metrics` bypasses the auth middleware. This means anyone who can reach the cert-watch instance can enumerate the entire monitored infrastructure.

This is a deliberate tradeoff noted in AGENTS.md (`/metrics is currently still open; see Plan 007 for hardening it`), but no breadcrumb exists to track it.

## Plan

Resolved by Plan 012 §3.3.

## Suggested fix

Option A: Gate `/metrics` behind bearer token auth (`Authorization: Bearer <token>`) via a `CERT_WATCH_METRICS_TOKEN` env var. Prometheus scrapes with the token; unauthorized requests get 401.

Option B: Add `/metrics` to the authenticated paths when `AUTH_PROVIDER` is set, and document that Prometheus needs to present a session cookie or bearer token.

Option C: Strip hostnames from the metrics output, using anonymized identifiers instead.
