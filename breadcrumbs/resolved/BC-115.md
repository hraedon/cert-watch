title: LDAP StartTLS skips bind — anonymous service search + cleartext user bind
kind: security

## Problem

In `auth/ldap_provider.py` the StartTLS path is broken two ways:

1. **Service connection** (`ldap_provider.py:143-146`):
   ```python
   if self.start_tls and tls:
       conn.start_tls()
   else:
       conn.bind()
   ```
   `start_tls()` negotiates TLS but does **not** bind. So with `LDAP_START_TLS=1`
   the service-account bind never happens and the subsequent `conn.search()`
   (lines 159-163) runs **anonymous**. On AD (anonymous search denied) every login
   fails; on a permissive directory it leaks directory metadata unauthenticated.

2. **User-verification connection** (`ldap_provider.py:181-184`): `user_conn` is
   built with `use_ssl=use_ssl` (false for StartTLS) and **never calls
   `start_tls()`**, so in a StartTLS config the user's password is bound in
   **cleartext**. The cleartext-bind guard at `ldap_provider.py:45-49` does not
   cover this path.

So StartTLS is both non-functional *and* a credential-exposure bug.

## Suggested fix

- Service conn: always `conn.bind()` after an optional `conn.start_tls()`.
- User conn: `auto_bind=False`, then `user_conn.start_tls()` when StartTLS is
  enabled, then `user_conn.bind()` — credentials only ever cross an encrypted
  channel.
- Add a test with a mocked `ldap3` asserting `bind()` is called and that TLS is
  negotiated before any bind on both connections (this path has no coverage today,
  which is why it regressed unnoticed).

Scoped for the 0.5.0 security batch — see Plan 023.
