# BC-043: OAuth ID token signature not verified (JWKS) — Plan 010 Slice 4 deferred

**Type:** defect
**Severity:** high
**Status:** resolved
**Resolved:** 2026-05-30
**Filed:** 2026-05-30

## Problem

`OAuthProvider.complete_oauth_flow()` in `auth.py` decodes the ID token by base64-decoding the payload (`parts[1]`) and checking `iss`/`aud` claims on the **unverified** decoded JSON. It does NOT verify the JWT signature against the IdP's JWKS endpoint.

This means an attacker who can intercept the callback (e.g., network-level on an internal deployment) could forge an ID token with arbitrary claims — including `preferred_username` — and bypass authentication entirely. The `iss`/`aud` checks provide no protection against forgery since the attacker controls the entire token.

Plan 010 Slice 4 specifies the fix: use `authlib`'s built-in JWT verification with `jwks_uri` from OIDC discovery, then check `iss`/`aud`/`exp`/nonce on the **verified** claims.

## Suggested fix

1. Replace the manual `base64.urlsafe_b64decode(parts[1])` with `authlib.integrations.requests_client.OAuth2Session.parse_id_token()` or equivalent JWT verification that fetches JWKS from the issuer's `.well-known/openid-configuration`.
2. Verify `iss`, `aud`, `exp`, and `nonce` on the verified claims.
3. Gate on `roles` claim (App Roles) instead of raw `groups` for Entra.
4. Tests: token signed with unknown key rejected; missing required role denied; valid token + role admitted; expired token rejected.

## Resolution

Implemented Plan 010 Slice 4 — full JWKS-based ID token verification in `auth.py`:

1. Replaced manual `base64.urlsafe_b64decode(parts[1])` with `_verify_id_token()` that uses `joserfc.jwt.decode()` (authlib 1.7+) to verify the JWT signature against JWKS fetched from the IdP's `jwks_uri`.
2. `_discover()` now captures `jwks_uri` and `id_token_signing_alg_values_supported` from OIDC discovery.
3. `_fetch_jwks()` fetches and caches the JWKS; `_verify_id_token()` decodes the token, validates signature via `KeySet.import_key_set()`, then validates claims via `authlib.oidc.core.CodeIDToken` (iss/aud/exp/nonce/at_hash).
4. When JWKS verification fails (wrong key, expired, issuer mismatch), the flow falls back to the userinfo endpoint (which requires a valid access token) rather than trusting unverified claims.
5. 11 new tests cover: valid token, wrong key rejection, expired rejection, issuer/audience mismatch, no JWKS, malformed token, roles claim, end-to-end flow with verified claims, forged token fallback, userinfo-only flow. Plus 7 tests for `_validate_claims_manual` fallback.
