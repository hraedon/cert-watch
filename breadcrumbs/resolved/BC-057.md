# BC-057: /api/audit endpoint has no route-level auth enforcement

**Type:** defect
**Severity:** low
**Status:** resolved
**Filed:** 2026-05-31

**Resolved:** 2026-05-31 — All API read endpoints now have route-level auth checks. _require_api_auth() in routes/api.py. audit.py and hosts.py export also check auth.

## Problem

`routes/audit.py:68` defines `api_audit()` which returns JSON audit log data. This endpoint relies entirely on the `auth_middleware` in `middleware.py` for authentication — the route handler itself does not check `request.scope.get("auth_user")` or validate the session cookie.

By contrast, mutating API routes like `PATCH /api/certificates/{id}/notes` (`routes/api.py:119`) explicitly check auth at the route level:
```python
auth = getattr(request.app.state, "auth_provider", None)
if auth is not None and not isinstance(auth, NoAuthProvider):
    token = request.cookies.get(SESSION_COOKIE, "")
    username = validate_session(token)
    if not username:
        return JSONResponse(content={"error": "unauthenticated"}, status_code=401)
```

The middleware does protect `/api/*` routes, so this is defense-in-depth rather than an active vulnerability. However, if the middleware chain is reordered or a future refactor removes the middleware check, the audit endpoint would be exposed.

Similarly, `GET /api/hosts`, `GET /api/certificates`, `GET /api/alerts`, `GET /api/export/certificates.csv`, `GET /api/export/certificates.json`, and `GET /api/export/hosts.csv` all lack route-level auth checks.

## Suggested fix

Either:
- Add a reusable `_require_auth(request)` dependency that all API routes use (FastAPI `Depends()`)
- Or document the convention that auth is middleware-only for read endpoints and route-level for write endpoints
