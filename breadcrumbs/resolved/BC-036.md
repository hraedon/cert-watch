# BC-036: No integration test for openssl s_client scan path

**Type:** defect
**Severity:** medium
**Status:** resolved
**Filed:** 2026-05-29

## Problem

The `_scan_via_openssl()` code path (used on Python < 3.13) depends on an external `openssl` binary and runs as a subprocess. All existing tests monkeypatch `_open_tls_connection` and `_has_native_chain_api`, so the real openssl subprocess path is never exercised in CI. If `openssl s_client` output format changes, or if the PEM regex fails on edge-case certificates (cross-signed roots, empty chain responses, non-ASCII subjects), the tests won't catch it.

This is a load-bearing path — every Python 3.12 deployment uses it for chain extraction.

## Suggested fix

Add a test that:
1. Starts a local TLS server (using `trustme` or `cryptography` to generate a CA + leaf + intermediate).
2. Calls `scan_host()` against it with `_has_native_chain_api` patched to return `False`.
3. Asserts the returned `ScannedEntry` has both leaf and chain certs with correct fingerprints.

This can be a `@pytest.mark.integration` test skipped in CI by default (requires openssl on PATH) but run in E2E or local development.

## Resolution

Added `tests/test_scan_integration.py` with 3 integration tests that start a real TLS server (using `cryptography`-generated certs in a daemon thread) and exercise the actual `openssl s_client` subprocess:

1. **test_openssl_chain_extraction** — leaf + intermediate chain extracted via real openssl, fingerprints verified
2. **test_openssl_self_signed_leaf_only** — self-signed cert, single leaf, empty chain
3. **test_openssl_connection_refused** — ScanError returned when no server is listening

All tests use `@pytest.mark.integration` and `@pytest.mark.skipif(not _openssl_available())`. Only `_resolve_host` is monkeypatched (to bypass SSRF loopback protection); the openssl subprocess runs for real.
