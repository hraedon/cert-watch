## On the project

The maintenance-mode discipline shows — the AGENTS.md is unusually detailed, the conventions are clear, and the test suite is comprehensive enough to catch regressions from multi-file refactors. The kimi/glm cross-model review pattern the user asked for worked well: the glm reviewer caught a critical bug in the renewal webhook (cert_watch_url using fingerprint instead of UUID → 404), a medium bug (DB query not filtering by fingerprint → wrong cert in payload), and several error-handling gaps. ## On the work done

**Confident in:**
- WI-083 (SecurityContext consolidation) — clean, well-tested, the conftest env-var alignment is the key insight. The regression test proving OAuthProvider rejects foreign-key state is the right test.
- WI-087 (re-export removal) — purely mechanical, all names already re-exported by `__init__.py`.
- WI-099 (CSRF auto-mint) — clever approach: exercise the validation path even in bypass mode, no test call sites changed.
- Renewal webhook — solid after the cross-model review fixes. The structured payload is genuinely useful for integration.

**Less confident about:**
- The scheduler integration reads env vars directly instead of using `Settings`. This is the established pattern in `scheduler.py` (other env reads too), and the glm reviewer flagged it as medium. But passing `Settings` through to `_check_renewal_overdue` would require changing the `_run_cycle` function signature, which ripples into the unit-test harness. Not worth the risk for a best-effort webhook path.
- The renewal webhook has no retry logic (the alert webhook retries via `backoff_range`). For an action-oriented signal, a single attempt with no retry is fragile. Low priority but real.
- The CTW integration plan is based on reading the GitHub repo and docs — no hands-on validation against a real CTW Hub instance. The API shape may differ from what I inferred.

## Gaps to flag

- **Renewal webhook has no retry** (`renewal_webhook.py:send_renewal_webhook`) — a transient network blip drops the webhook. The alert webhook retries; this one doesn't. Low severity but real for an action-oriented signal.
- **Scheduler env-var bypass** (`scheduler.py:_send_renewal_webhook_if_configured`) — reads env vars directly instead of using `Settings`. The `renewal_webhook_url` and `renewal_webhook_headers` fields on `Settings` are dead code in the scheduler path. Only matters if someone configures these via the GUI Settings page (which doesn't exist yet for this feature).
- **No e2e test for the renewal webhook** — the scheduler integration (`_send_renewal_webhook_if_configured`) is not covered by any test. The unit tests cover the individual functions, but the wiring (detecting overdue → emitting event → sending webhook) is untested as an integration path.
- **CTW plan is unvalidated** — the API endpoint shapes in `plans/ctw-integration.md` are inferred from the GitHub repo, not tested against a live CTW Hub. The `GET /api/v1/certificate` endpoint and its response shape are assumptions.
- **Breadcrumbs sync timed out** — the `agent-notes breadcrumb sync` command timed out (30s). The DB is updated correctly (WI-083/072/087/099 closed), but the markdown files in `breadcrumbs/resolved/` may not have been regenerated. Low impact — the DB is the source of truth.
