## On the project

cert-watch is well-structured for a single-developer project. The database layer's repository pattern made the BC-073 grouped-dashboard rewrite tractable — the helpers `_build_dashboard_rows` and `_build_unified_from_dash` do the heavy lifting, and the SQL GROUP BY just needed to slot in above them. The auth module decomposition (Plan 021) paid off: the OAuth nonce changes touched only `oauth_provider.py` and `session.py`, with clean interfaces between them.

The template layer is the weakest part. Jinja2 templates with inline event handlers are a maintenance trap — every new interactive element requires a coordinated edit in the template (data-action attribute) and the base.html delegation handler. The ratchet test (`test_no_inline_handlers.py`) is a smart mitigation but the pattern itself is fragile. If the UI grows significantly, consider a lightweight JS framework or web components.

The `_effective_config` / `from_env_with_kv` merge pattern (env vars > kv_store > default) is repeated in multiple places with slight variations. It works but the encryption key threading through those functions is getting awkward.

## On the work done

BC-075 (CSP inline handlers) was the most mechanical — 23 handler conversions following the base.html reference pattern. The delegation handler in base.html is clean but centralizes all action routing in one place, which will get crowded if many more actions are added. The `data-action="switch-tab"` with pre-existing `data-tab` attribute required care to avoid duplicate attributes.

BC-073 (grouped dashboard) was the trickiest. The SQL GROUP BY approach works well for the common case, but the ungrouped entries (uploaded, pending) still need a full cert load to merge. The test failures revealed that the default `grouped=1` view includes uploaded entries — the initial implementation only returned scanned entries, breaking 8 tests. The fix (loading ungrouped entries separately and merging) is correct but means the function still does a full cert load for the ungrouped path. Acceptable at SMB scale.

BC-082 (kv_store encryption) was cleanest. Fernet + SHA-256 key derivation from the existing `.auth_secret` means no new secrets to manage. The `enc:v1:` prefix makes encrypted values identifiable and allows transparent decryption with backward compatibility. One concern: if the `.auth_secret` is regenerated (operator deletes the file), all encrypted kv_store values become unreadable. This is the correct security behavior but should be documented.

BC-071 (OAuth nonce) was the most security-sensitive. The `state:nonce:sig` format change to `_verify_state` required updating the return type from `str | None` to `tuple[str, str | None] | None`, which touched the auth route callback. The backward-compatible parsing (try new format, fall back to legacy) ensures existing sessions aren't broken during deployment.

## Gaps to flag

- **`list_dashboard_grouped_page` still loads all certs for ungrouped entries** (`queries.py:1262`). The SQL GROUP BY handles scanned entries efficiently, but uploaded/pending entries go through `_list_unified_entries_raw` which does `SELECT * FROM certificates`. At thousands of uploaded certs this would regress. Acceptable for now; flag if the upload feature grows.
- **Delegation handler in base.html is a single switch** (`base.html:160-190`). All `data-action` values route through one click listener. Adding many more actions will make this unwieldy. Consider splitting by namespace (e.g., `data-action-slide`, `data-action-tab`) if it grows past ~15 actions.
- **Encrypted kv_store values are unreadable if `.auth_secret` rotates** (`queries.py:32-36`). Fernet keys are derived from the signing key. If the operator deletes `.auth_secret` to regenerate it, all `enc:v1:` values in kv_store become permanently unreadable. The code logs a warning and returns the raw ciphertext, but there's no recovery path. Document this in the operator guide.
- **`_verify_state` return type change** (`session.py:63`) is a breaking API change for any code that expected `str | None`. The only callers are `oauth_provider.py` and `routes/auth.py`, both updated. But if external code imports `_verify_state`, it will break. The function is private (underscore prefix) so this is acceptable.
- **No test for the nonce verification path** in `test_oauth_provider.py`. The nonce is generated, embedded, and extracted, but there's no test that verifies an ID token with a mismatched nonce is rejected. The plumbing is correct but the security guarantee is untested at the unit level.
