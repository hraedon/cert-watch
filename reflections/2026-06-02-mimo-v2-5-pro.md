## On the project

The codebase is well-structured with clean separation of concerns. The auth layer (auth/ package) is particularly well-decomposed — session signing, protocol, providers, and factory are each in their own module. The middleware pattern of `require_auth`/`require_write` as FastAPI dependencies is elegant and consistent.

One thing that stands out: the form-POST handlers (hosts.py, certificates.py) can't use `Depends(require_write)` because they return `RedirectResponse` instead of raising HTTPException. This creates a parallel `require_write_form()` path that duplicates auth/CSRF/write-check logic. The shared `_may_write()` function mitigates the risk, but the two paths must stay in sync. This is documented but feels like a design smell that could bite during future changes.

The CSP `unsafe-inline` situation (BC-075) is the most significant security gap. The templates use ~24 inline event handlers that CSP nonces can't whitelist. This is deferred to a template rewrite, which is the right call — fixing it now would be throwaway work.

## On the work done

The audit was productive — found 3 real bugs in the API routes. The webhook/test write-gate issue (BC-087) is the most concerning: a read-only user could trigger webhook sends, which is a side-effecting operation. The webhook URL validation bug (BC-088) was a classic "call but don't check return" error. The CSV duplicate columns (BC-089) were cosmetic but misleading for compliance reporting.

Lint fixes in test files were mechanical — unused imports, unsorted import blocks, and line-length violations from the prior session's BC-083/081 work. The `ruff check --fix` handled most of them; two line-too-long issues needed manual wrapping.

I'm confident in all three fixes. The webhook/test change is a one-liner. The webhook validation fix is a direct copy of the pattern already used in the PATCH endpoint. The CSV fix removes redundant columns that were actively misleading.

## On what remains

Open breadcrumbs after this session:
- **BC-082** (security / medium) — kv_store plaintext secrets at rest. Needs encryption or guidance to use env/*_FILE instead.
- **BC-075** (security / low) — CSP `unsafe-inline`. Deferred to template rewrite.
- **BC-073** (performance / low) — grouped dashboard full-memory load. Acceptable at current scale.
- **BC-071** (security / medium) — OAuth userinfo nonce binding. Needs route-layer nonce persistence.

BC-071 is the most actionable of these. The OAuth provider already generates a nonce in the authorization request; the fix is to persist it across the redirect (session or state parameter) and verify it in the userinfo fallback path. This is a medium-sized change touching the OAuth routes and provider.

BC-082 needs a design decision first (OS keyring, SQLCipher, or app-level encryption). BC-075 needs the template rewrite. BC-073 is acceptable at current scale.

## Gaps to flag

- `require_write_form()` in `middleware.py:515-536` duplicates auth/CSRF/write-check logic from `require_write` (line 473). The shared `_may_write()` helps, but CSRF checking is still duplicated. If either path changes, the other must be updated.
- The `_validate_webhook_url` function in `api.py:83-98` validates scheme and private IP but doesn't check for SSRF via DNS rebinding (resolves hostname to IP, but the IP could change between validation and use). This is a low-risk gap since webhook URLs are admin-configured, not user-supplied.
- The health banner in `base.html:132` renders via `innerHTML` with mixed escaped/unescaped content. The `esc()` function is used for database values, but the template strings contain raw HTML (`<b>` tags). A future change adding user-controlled data without escaping could create an XSS vector.
