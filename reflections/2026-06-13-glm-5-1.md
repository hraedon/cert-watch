## On the project

This session marks the transition from development to maintenance. The project has reached a natural closure point — the surface is complete for the stated SMB positioning, the test suite is at 2039+ tests and 91% coverage, and the SC-081 window thesis gives a clear narrative for why this tool exists now.

The codebase is well-organized for maintenance: the decomposition of settings.py (the last large module) leaves no file over ~300 lines of route logic. The breadcrumb/agent-notes system is genuinely useful — I could query open items, update them, and the next agent will see the resolved state. The AGENTS.md maintenance-mode section is the right brake for rotating agents.

One concern: the project has accumulated a lot of conventions that only exist in AGENTS.md (nonce CSP, double-submit CSRF, data-action pattern, no inline handlers). A new maintenance agent that skims AGENTS.md will be fine, but one that doesn't will introduce subtle regressions. The ratchet test for inline handlers is the right structural guard.

## On the work done

The multi-agent strategy worked well for this volume of work. Dispatching 6 kimi/glm agents in parallel for the first batch, then 6 more for Plan 049, meant the total wall-clock was dominated by the slowest agent (~8 minutes) rather than the sum. The adversarial reviewers caught real bugs — the WI-030 button visibility issue (both reviewers agreed), the WI-035 response body size gap, and the WI-037 stderr stall were all genuine findings that needed fixing.

The coordination cost was manageable: one test failure from the settings decomposition (monkeypatch target needed updating after `ca_probe.py` extraction) and one syntax error from a sloppy dead-code removal. Both caught by the test suite.

What I'm least confident in: the E2E tests in `test_settings_post.py`. The authed-server fixture creates state that persists across the module, and while I fixed the password-order dependency, the pattern of module-scoped browser state is inherently fragile. The tests passed locally but haven't been validated in CI's Playwright environment.

The `freeze_time` tests for SC-081 are technically redundant today (the policy logic is pure `not_before` comparison, no `datetime.now()` calls), but both reviewers agreed they're a valuable hedge. I concur — the milestone dates are the one place where a future agent is most likely to introduce an off-by-one.

## Gaps to flag

- **`freezegun` not declared in `pyproject.toml`**: The SC-081 freeze-time tests use `freezegun` but the P3.3 agent only installed it via `uv pip install`. If CI doesn't have it, those 25 tests will fail with ImportError. Need to add `freezegun` to `[dev]` extras or the test dependencies.
- **E2E test suite not run locally**: The 18 new E2E tests haven't been executed against a running Playwright instance in this environment. CI will catch issues, but the AGENTS.md ritual says to run e2e locally after UI changes.
- **`_integration_servers.py` patch targets**: The stale `core._is_blocked_ip` target was fixed to `ca_probe._is_blocked_ip`, but if the settings decomposition continues to shuffle imports, these patch lists will silently go stale again. Consider a test that asserts each patch target is importable.
- **`alerts.py` exception annotation convention**: The project now has two styles — `# noqa: BLE001 — reason` and some older blocks that may not have the noqa. A `ruff` BLE rule would enforce consistency, but it's not currently configured.
- **The diff is 42 files, +1710 lines**: This is a large uncommitted changeset. The risk of merge conflicts or subtle interactions grows with time uncommitted. A commit soon would reduce that risk.
