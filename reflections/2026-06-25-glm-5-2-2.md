## On the project

This was my first session driving cert-watch as a pure orchestrator (no direct edits until the post-review fix pass). The thing that stands up close is how well the maintenance-mode safety net actually catches you: `test_css_class_coverage` fired the instant a JS-hook class landed in a template without a CSS rule, and the AGENTS.md "embarrassment checklist" was a literally accurate predictor of the failure class. A project where the ratchet tests and the prose conventions agree is rare, and it's the reason a six-item batch of mostly-template edits could land with confidence. The backlog itself is large but well-triaged — almost every open WI is a real, located defect rather than a vague aspiration, which made selection clean.

## On the work done

The orchestrator/subagent/reviewer pattern worked well for this batch. Splitting six small, self-contained WIs into two parallel implementer tracks and then cross-reviewing with the opposite model line caught three real defects that a single-pass implementation would have shipped: (1) the `/favicon.ico` route was added but not registered in `_PUBLIC_PATHS`, so it 301'd to `/login` under auth — exactly the unauthenticated probes it existed to serve; (2) WI-105 fixed the note-chip duplicate-ID bug but the inline note-edit *form* had the identical bug two functions below it; (3) the two new JS-hook classes tripped the CSS ratchet. All three were the kind of thing a reviewer reading the diff in context catches but an implementer heads-down in their own change misses.

What I'm less confident in: WI-106. The casing fix (`Oauth`→`OAuth`) is real and correct for the only reachable provider value, but the reviewer surfaced that `OAuthProvider.provider_name` always returns `"oauth"` for every OAuth-family provider, so the entra/azure/oidc branches of the `provider_labels` map are dead code in production, and the oidc regression test mocks a value no real provider emits. I left the map as defensive future-proofing and filed WI-111 for the proper seam rather than half-wiring it — wiring a display label through the auth layer is auth-adjacent and the user specifically asked for a WI rather than an inline fix. The honest read is that WI-106 as shipped fixes the visible symptom but the underlying "entra deployments show generic OAuth" issue remains until WI-111.
