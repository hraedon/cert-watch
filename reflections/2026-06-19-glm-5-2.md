## On the project

First session reading cert-watch deeply. The design-token layer is genuinely tasteful, not generic — the warm-charcoal palette has documented rationale ("instrument panel, not cold SaaS night"), the status-color budget is actually enforced (ok/warn/crit/expired reserved for status, bronze accent for chrome only), and the empty-state discipline is real. The colleague's review was all-negative and that's a skewed picture; the parts of the UI that aren't utility-sprawl are the parts a careful human would also ship.

The utility-class layer is exactly what the colleague described and worse. 473 selector lines, three separate "wave" sections that re-add the same utilities (`.cw-gap-7/10/12` and `.cw-mb-8` literally defined twice), and naming-convention drift within the file (decimal px, raw indexes, raw px, px pairs all coexist). AGENTS.md already records that `.cw-gap-9`/`.cw-gap-14` were referenced while undefined for months — that bug class will recur until the file is reconciled.

The maintenance-mode discipline (Plan 049) is real and shapes every recommendation. "Surface is closed; no new product surface without a priced plan" is the brake that prevents this from becoming a redesign project. The bar isn't "fix the tells elegantly"; it's "fix the tells within maintenance scope."

patina (Opus's charter) is on the right track but undersized for what the user is now asking. The token contract is correct and well-documented; vendor-not-install is the right call for this family; "same metal, different finish" is a good organizing metaphor. But token coherence ≠ component coherence — patina does nothing about the `.cw-seg` vs `.cw-segmented` drift or the fake-controls reflex, and the colleague's tells aren't token problems.

## On the work done

**What I'm confident in:**
- Each of the colleague's five tells verified against code with hard data (grep counts, line refs), not impressions. All confirmed except the "everywhere" framing of inline `--tone`, which is actually only 6 occurrences concentrated in urgency bars and the validity meter — real but overstated.
- The Source-button-is-dead claim is the strongest: `name="source"` appears in exactly one template location (`dashboard.html:115`), with no other UI anywhere to set the filter.
- The recommendation against adopting a real framework (Tailwind/Pico/shadcn) for this family. The aesthetic is too specific, the no-build principle is load-bearing, and the per-tool accent variation would fight any off-the-shelf system.

**What I'd want second eyes on:**
- The claim that patina + patterns-catalog (doc, not code) is the right next step rather than extracting shared `.pt-*` component CSS now. That's a judgment call. I deferred the shared-code question (matching patina's own plan 001 WI-3) on the theory that you should name patterns before you share them — but a reasonable reviewer could argue extraction forces the naming and is therefore the faster path. Two-sessions-of-drift vs. one-big-migration tradeoff.
- My dismissal of inline `--tone` as "debatable, not a clear tell." A designer with stronger opinions might call per-row Jinja-computed custom properties a tell regardless of count. I called it 6 occurrences and concentrated; they might call it the wrong default.
- The severity assigned to WI-068 (a11y audit, medium) — for a production tool in regulated SMB environments this could arguably be high. I held at medium because no specific user is blocked today and the existing pill system does have text labels, but the charts and urgency bars genuinely fail colorblind users.

**Process notes:**
- Dedup-searched before filing each WI — important, the agent-notes store's main failure mode is duplicates. No dupes found, but the search is fuzzy and I relied on score thresholds.
- I bundled small same-area fixes into single WIs (e.g., WI-070 covers three dashboard-header issues) rather than filing micro-WIs. Trade-off: easier to track, harder to partially close.
