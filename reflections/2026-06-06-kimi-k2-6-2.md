## On the project

cert-watch has reached a mature state where "small quick wins" are actually medium-sized because the codebase has a lot of interconnected wiring. The env-config pattern (config.py Settings.from_env() → dataclass fields → route-level request.app.state.settings) is consistent and clean, but threading new fields through all call sites (especially the async scan Host routes with their semaphore-guarded store_scanned_async closures) is fiddly.

The mypy cleanup revealed a real class of systematic problems:

1. `UploadFile | str` form fields across routes/settings.py — every `.strip()` call was a union-attr error. This is a FastAPI pattern; the fix is always `isinstance(raw, str)` guard + reassign.
2. `Any | None` union access on `request.app.state.settings` — routes assume settings is present but mypy sees `Any | None`. Adding `assert settings is not None` is the right fix (the app would 500 without settings anyway).
3. OAuth provider's conditional import dance (`joserfc` vs `authlib.jose`) produces `Name already defined`/`Cannot assign to a type` errors because mypy sees both branches. Moving to `typing.Any` variables typed outside the try blocks is the clean path.

The dashboard grouped view (`SELECT * FROM certificates`) is a genuine architectural hotspot — it was identified in the breadcrumb as O(n) memory, and looking at the code confirms it loads every cert then filters in Python. The fix involves SQL-level EXCEPT/NOT EXISTS, which is non-trivial because `_build_dashboard_rows` and `_build_unified_from_dash` expect cert + chain rows in a specific shape.
