## On the project

cert-watch continues to be a well-tended codebase. The security posture is genuinely good — not checkbox security, but thought-through: CSP nonces, session revocation, SSRF-safe URL opening, auto-provisioning for exposed instances. The fact that the project tracks security issues as numbered breadcrumbs (BC-xxx) with severity ratings is a strong signal of maintenance culture.

The biggest structural risk I see is the **optional-dependency auth paths**: OAuth and LDAP are the most security-critical surface, but their unit tests are skipped in the default run because `authlib`/`joserfc`/`ldap3` are extras. The coverage ratchet is 88%, but the *real* coverage of those modules when deps are absent is much lower. This is a deliberate build choice (fast CI, minimal core), but it means the highest-risk code is covered mostly by integration tests and skips.

## On the work done

**What went well:**
- The semaphore failure-isolation tests are real behavioral tests: they mock `store_scanned_async` to raise, then assert on `scan_history` rows and redirect locations. This is the right pattern for testing error paths in route handlers.
- The route-ordering bug (`/hosts/all/scan` captured by `/{host_id}/scan`) was a subtle issue. I found it because the `scan_all` test I wrote was failing with "host not found" for `host_id="all"`. Moving the static route above the dynamic route fixed it. FastAPI route ordering matters; the prior session didn't catch this because there was no test for `scan_all`.

**What was awkward:**
- Figuring out how to make e2e tests skip gracefully when Playwright is missing. I tried `collect_ignore` in `conftest.py`, `pytest_ignore_collect` hook, and `collect_ignore` in root conftest — none prevented the `ImportError` from the `from playwright.sync_api import ...` module-level imports. The only reliable mechanism was module-level `pytest.importorskip("playwright")` inserted before the import. This is a bit invasive but works.
- The `agent-notes` CLI interface is inconsistent: `find` uses `--identifier` as a flag, but `update` requires the identifier as a positional argument. Minor DX friction.

**What I'd want a second pair of eyes on:**
- The `hosts.py` route ordering change. I moved `scan_all_hosts` before `scan_host_now` so the static route `/hosts/all/scan` is checked before the dynamic `/{host_id}/scan`. This is correct for FastAPI, but any other route ordering assumptions in the file should be reviewed. I checked the rest of the file and no other static routes follow dynamic ones.

## Gaps to flag

- **`routes/hosts.py:462`**: The `asyncio.Semaphore(1)` is process-local only. It won't serialize writes across multiple uvicorn workers (gunicorn, etc.). For the intended single-writer SQLite deployment this is correct, but if the backend ever switches to PostgreSQL with multiple workers, the semaphore becomes a bottleneck and doesn't help. Documented in the code but worth flagging.
- **`tests/deploy/`**: The deploy tests are present and ruff-clean, but excluded by `-m 'not deploy'` in the default pytest config. They are effectively dead code unless someone explicitly runs `pytest -m deploy`. Bit-rot risk is real.
- **`pytest-cov` with `--collect-only`**: Running `pytest --collect-only` with coverage enabled shows 0% coverage because no source code is imported during collection. This triggers the `--cov-fail-under` failure and makes the output confusing. Not a bug, but a known quirk.
- **Optional auth dependencies**: The most security-critical code paths (OAuth provider, JWKS verification, ID-token validation) are only exercised when `joserfc` and `authlib` are installed. In the default CI run (which doesn't install `[auth-oauth]`), these paths are skipped. The project accepts this trade-off, but the gap is real.
