# Session Reflection — 2026-07-01

**Work summary:** Deployed 5 adversarial review subagents (Kimi-K2.7) across security/auth, scan/posture/chain, database/migrations, routes/alerts, and app/config/scheduler. Consolidated 50+ findings, fixed 19 actionable issues across 19 source files + 3 test files, added 22 regression tests. All CI green (ci, e2e, release, deploy-smoke). The critical fix is BasicConstraints.ca enforcement in `_is_signed_by` — a leaf cert could previously act as an intermediate to forge a trusted chain.

---

## On the project

The codebase is mature and well-structured for a maintenance-mode project. The AGENTS.md is unusually thorough — it documents conventions, known issues, architecture decisions, and even "decisions that must be surfaced to the human." The security posture is generally strong: signed sessions with per-user revocation, double-submit CSRF, SSRF-aware HTTP client, parameterized SQL throughout, CSP nonces. The findings from the adversarial review were mostly medium-severity gaps in scope enforcement and error handling, not fundamental design flaws. The main concern is that the tag-scoped access control (WI-051/052) was implemented for the dashboard and host endpoints but not consistently applied to alerts, events, analytics, and reports — a pattern that will require ongoing attention as new endpoints are added.

## On the work done

The BasicConstraints.ca fix is the highest-impact change — it closes a real chain-forgery path where any holder of a public-trust leaf cert could forge chains. The fix is correct but required updating test fixtures (chain_triplet) to generate CA certs with proper BasicConstraints. The `/metrics` auth gate is a **breaking change** — existing deployments with auth enabled and no `CERT_WATCH_METRICS_TOKEN` will now require a session for `/metrics`. This is the right security default but operators need to know.

The posture-eval-outside-write-lock fix was the most complex. The original WI-113 fix moved posture evaluation before `BEGIN` but the application-level `get_write_lock()` was still held by callers. The fix adds a `_posture_eval` parameter to `store_scanned` and has both callers (`store_scanned_async` and `_store_with_lock`) pre-compute posture before acquiring the lock. The pre-computation is unconditional (CAA DNS happens regardless of `check_revocation`).

The adversarial review of the fixes (second pass) caught several issues I'd missed: incomplete recipient validation in the digest path, silent PKCS#12 truncation instead of rejection, and the OCSP 405 edge case. All were addressed before commit.

## On what remains

- **Events/analytics scope filtering** — `/api/events`, `/api/renewal-analytics`, `/api/pivot/*`, `/api/trends/*`, `/api/calendar`, `/readiness` still return fleet-wide data to scoped users. Attempted to file a breadcrumb (BC pending) but regista DB needs migrations (36, 37 pending). This is the most significant remaining gap.
- **`/metrics` breaking change documentation** — Operators need to know that `/metrics` now requires auth or a token. Should be called out in release notes / upgrading docs.
- **Rate limiter fail-open** — Still fails open to in-memory on SQLite errors. Not fixed because changing to fail-closed could break deployments under SQLite pressure. Flagged but deferred.
- **API key SHA-256 unsalted** — Not fixed; tokens are high-entropy so risk is low, but a slow hash would be better practice.
- **Encryption key reuse** — Signing key still derives the Fernet encryption key via SHA-256. HKDF with distinct context labels would be better, but this is a schema/secret-management change that needs a migration plan.

## Gaps to flag

- **Events API scope bypass** (`routes/api/events.py:27-99`) — scoped users can read all event streams. Medium severity, not fixed.
- **Renewal analytics scope bypass** (`routes/api/renewal_analytics.py:17-32`) — scoped users see fleet-wide analytics. Medium, not fixed.
- **Insights scope bypass** (`routes/api/insights.py:54-125`) — pivot, trends, calendar endpoints ignore scope tags. Medium, not fixed.
- **`/metrics` breaking change** (`middleware.py:469-483`) — auth-enabled deployments without `CERT_WATCH_METRICS_TOKEN` will break unauthenticated Prometheus scrapers. Needs release notes.
- **Regista DB needs migrations** — `agent-notes` CLI fails with "Migrations pending: schema 'cert_watch' has applied [1..35], missing [36, 37]". Couldn't file breadcrumbs this session. Infrastructure issue.
- **Test coverage floor lowered** for `password.py` (90→89) — the new password length checks aren't covered by integration tests (need full auth setup). The regression tests verify the source contains the checks, but don't exercise the HTTP path.
- **Posture pre-compute fallback** — if pre-lock posture eval fails, `store_scanned` recomputes it inside the lock. This is a safe fallback but means the lock-during-network-I/O issue can still occur on transient failures.
