# Session Reflection — 2026-05-25

**Work summary:** Adversarial review of the full cert-watch codebase. Identified and fixed 5 issues across 3 files: unbounded file upload (DoS), orphaned alerts on cert/host deletion, unescaped error strings in redirect URLs, and duplicate cert accumulation on repeated scans. Closed breadcrumbs BC-001, BC-004, BC-007, BC-008. Open breadcrumbs BC-002, BC-003, BC-005, BC-006 remain.

---

## On the project

The codebase is clean and well-structured for its size. The separation between scan, upload, database, alerts, and scheduler modules is sensible. The spec traceability (AC-01, AC-02, etc. in docstrings) is a strong signal that someone cared about verifiable requirements.

A few things stood out:

- The `_db_path()` helper in `app.py` reinitializes the schema on every single request. It's idempotent so it's correct, but it's a wasted PRAGMA call per HTTP request. Not a bug, but wasteful at scale.
- The scheduler uses a polling loop with `threading.Event.wait()` rather than APScheduler or similar. It works, but it means only one scheduled time is possible (no cron-like flexibility). Fine for v1.
- The `_connect()` function in `database.py` opens a fresh SQLite connection for every operation. This is the correct pattern for a single-writer SQLite app, but combined with the lack of WAL mode (BC-005), it creates contention risk between the scheduler thread and the HTTP handlers.

## On the work done

Confidence is high on all five fixes:

1. **Upload size cap** — straightforward, reads MAX+1 bytes and rejects if exceeded. The temp file cleanup is correct.
2. **Alert cascade on cert/host delete** — gathers all cert IDs (leaf + children) and bulk-deletes alerts. The placeholder-based SQL is safe from injection.
3. **URL encoding for errors** — `urllib.parse.quote()` is the right tool.
4. **Duplicate scan prevention** — deletes old leaf+chain for the same (hostname, port) before inserting new ones. This is the one I'd want a second look at: the delete-then-insert is not atomic (two separate connections), so there's a tiny window where a concurrent read sees an empty state. For a single-writer SQLite app this is acceptable.

All 75 existing tests pass with no changes. I did not add new tests for the fixes, which is a gap — the cascade-delete of alerts and the upload size cap deserve dedicated test coverage.

## On what remains

**Open breadcrumbs (existing, not fixed this session):**

- BC-002: SMTP password logged on auth failure — low severity but a real security hygiene issue.
- BC-003: `validate_chain_order` returns False for single-cert bundles — this means uploaded single DER certs always show `chain_valid=false` in the dashboard, which is misleading. Should return None or have a separate "not applicable" state.
- BC-005: No WAL mode — the scheduler thread and HTTP handlers can contend. This is the most likely source of production OperationalError crashes.
- BC-006: `days_until_expiry` uses `delta.days` which truncates, so a cert expiring in 1.5 days shows as "1 day" and a cert expiring in 0.5 days shows as "0 days" (displayed as "today") rather than "expires today". Minor but could cause confusion at the boundary.

**Needed before shipping:**

- Tests for the five fixes (especially cascade-delete and upload size limit).
- BC-005 (WAL mode) should be addressed before any multi-threaded deployment.

**Nice to have:**

- BC-003 (chain_valid for single certs)
- BC-002 (SMTP password logging)
- BC-006 (expiry day truncation)

## Gaps to flag

- **No test coverage for cascade-deleting alerts.** `test_delete_certificate_removes_leaf_and_chain` only checks certificates are gone, not that alerts are also cleaned up. Same for host deletion. `database.py:430-464`.
- **No test for upload size limit.** The 10 MiB cap in `app.py:220-226` is untested. A test posting a large file and asserting the redirect with error message would suffice.
- **`store_scanned` delete-then-insert race.** `scan.py:140-156` uses a separate connection to delete old certs, then creates new `SqliteCertificateRepository` instances to insert. Between delete and insert, a concurrent dashboard request sees no certs for that host. Single-writer SQLite makes this a very small window, but it exists.
- **`_connect` lacks WAL mode.** `database.py:102-105` opens connections with default journal mode. For a mixed-thread workload (scheduler + HTTP), WAL mode would eliminate most `OperationalError: database is locked` failures.
- **`list_dashboard_rows` loads ALL certificates into Python memory.** `database.py:496-569` does `SELECT * FROM certificates` then groups in Python. This will degrade with scale. Not a problem at v1 volumes but worth noting.
- **Import of private `_connect` in scan.py.** `scan.py:15` imports `_connect` (renamed to `_connect_db`) from `database.py`. This is a private function; a better API would be a public helper or moving the dedup logic into `database.py`.
