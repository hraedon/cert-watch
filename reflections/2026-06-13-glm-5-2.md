## On the project

This is a well-built tool that's been over-built for its audience. The feature surface (compliance reports, SC-081 readiness, SIEM export, RBAC, policy packs, renewal analytics) is large for an SMB cert monitor, and the maintenance-mode decision is the right call — the priority should be depth on what matters (scan, posture, alert, audit) not breadth into speculative features. Removing CT was the correct first pruning step; it was the most complex subsystem with the least validated value and the heaviest external dependency (crt.sh).

The test suite is genuinely strong — 1981 tests that actually exercise behavior, not theater. The adversarial reviewers consistently found edge-case logic bugs (alert_type scoping, failed-alert dedup, urgency threshold inconsistency, allow_private override) but zero infrastructure defects (no SQL injection, no bare excepts, no unused deps). The codebase is clean at the level that matters.

The user built this for their employer and is running it in production. That changes the calculus on everything — this isn't speculative software, it's live tooling, and the feedback loop is closing (the email bug was found by the tool spamming them). The highest-value work going forward is validating against the real estate, not adding features.

## On the work done

The alert fix was the highest-impact change. The original design created separate alert rows for every crossed threshold, which meant a cert at 5 days got emails for both 14-day and 7-day stages — plus daily cooldown re-alerts. The fix (only the most urgent newly-tripped threshold fires, each pair fires once) was conceptually simple but had three subtle edge cases that the adversarial reviewers caught: (1) `expired` alerts were blocked by `expiry_warning` at the same threshold because the dedup didn't scope by alert_type, (2) failed delivery alerts permanently suppressed re-evaluation, (3) `renewal_stalled` alerts could collide with expiry thresholds on custom configs. All three were fixed by scoping the existing-thresholds set to `(alert_type, threshold_days)` pairs and excluding `status="failed"`.

The CT removal was mechanical but voluminous — 43 files touched. The scoping subagent's map was comprehensive and the only issue the adversarial reviewer found was a dangling redirect to the deleted `/discover` route. The CAA regex constants (`_DOMAIN_RE`, `_MAX_DOMAIN_LEN`) had to be relocated from `ct_lookup.py` to `caa_check.py` before the source file could be deleted.

The Prometheus metrics enhancement was the most satisfying because the metrics already existed — I just made them useful. The adversarial reviewer caught that my urgency thresholds (≤3/≤14) were inconsistent with the dashboard's (<7/<30) and that the posture query was fetching all scan_posture rows instead of using the existing `get_posture_grades_for_certs()` function.

The WI-042 CRL implementation was the most technically interesting work. The adversarial reviewer caught a security posture regression (hardcoded `allow_private=True` overriding operator SSRF policy) that would have been embarrassing to ship.

## On what remains

- **WI-040** (open): alerts visual baseline needs regeneration — the alert dedup change means fewer alert rows per cert, so any visual baseline showing multiple alerts for the same cert is stale. This will only surface on CI's visual gate, not locally.
- **e2e suite not run**: The Discover page removal touched nav links and e2e helpers. The functional e2e suite should be run before push to catch any broken selectors. I updated the helpers but didn't execute the Playwright suite.
- **kv_loader `alert_digest_only` bug**: The merge logic at `config/kv_loader.py:124-127` allows False→True via kv_store but not True→False, and ignores whether `ALERT_DIGEST_ONLY=0` is explicitly set in env. Pre-existing but surfaced by the digest-first recommendation. Should use `_kv_bool` for consistency.
- **CRL stale publication threshold (30 days)** is hardcoded in `posture.py`. For environments with quarterly CRL publication, this fires every scan. Not configurable yet.
- **No CRL fetch caching**: Multiple certs from the same private CA each independently download the same CRL. Acceptable at SMB scale but would matter at larger fleet sizes.

## Gaps to flag

- `app.py:332-335`: The digest-mode branch now calls `evaluate_renewal_window` + `process_pending` before `send_expiry_digest`. This means renewal-stall alerts are sent as individual emails even in digest mode — the operator gets both the digest summary AND individual renewal-stall emails. This is the correct behavior (renewal-stall is a distinct signal) but worth documenting.
- `posture.py:327`: CRL signature check silently swallows malformed issuer_der with only a debug log. An operator who passes a corrupt issuer cert gets no finding — the check just doesn't run.
- `scan.py:766-768`: Issuer matching uses string comparison (`chain_cert.subject == cert.issuer`) rather than DER bytes. Works for fresh scan data but would silently fail if chain certs are ever loaded from the database where string representations may differ.
- The `cooldown_hours` parameter on `evaluate_thresholds` is now a confirmed no-op, kept for backward compat. Tests still pass it (`cooldown_hours=24` / `cooldown_hours=0`) which could confuse a future reader.
- `prometheus-rules.yaml:47`: The `CertWatchScanFailures` alert name says "ScanFailures" but the expression triggers on expired certificates (`cert_watch_certificates_expired > 0`). Pre-existing misnomer.
