## Gaps to flag

- **SMTP production path bypasses SSRF validation** (`alerts.py:457-462`, `:844-849`): `smtplib.SMTP(SL)` connects without going through `resolve_and_validate_host` or `ssrf_safe_urlopen`. The `/settings/test-smtp` endpoint DOES validate. Admin-configured host, so defense-in-depth not exploitable, but a real inconsistency in the SSRF posture. LOW.
- **CompositeProvider timing leak on local-admin success** (`local_admin.py:189-203`): when local credentials match, the primary (LDAP) provider is skipped, creating a measurable timing difference. Known trade-off documented in code comment. The auto-generated admin password has 144-bit entropy so brute-force is infeasible. LOW.
- **OAuth state has no embedded timestamp** (`session.py:117-122`): replay window bounded only by the 10-minute `cw_oauth_state` cookie TTL. Mitigated by single-use authorization codes at the IdP. LOW.
- **The dirty tree is a hazard**: another session's uncommitted WIP (alert_groups feature) sits in the working tree. If anyone does `git add -A && git commit` without reading status, they'll sweep up work they don't understand. The migrations registry already references m0025 — if the migration file isn't committed alongside the registry change, the app will fail to start. Flag for whoever owns that WIP.
- **WI-041 / P3.2 dependency**: the restore-CLI decision was explicitly gated on P3.2 (restore runbook validation) running. If P3.2 never runs — which is a real risk since it's operator-paced — the CLI question stays unanswered forever and the doc-only mitigation stays unvalidated. The response to "validate later might mean never" is to commit to running P3.2, not to ship the CLI pre-emptively.
