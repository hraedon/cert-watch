# Session Reflection — 2026-06-11

**Work summary:** Implemented the entirety of Plan 048 (The 47-Day Regime) — 7 work items across 3 workstreams, totaling ~20 new/modified source files and ~150 new tests. All 1922 tests pass, lint clean. Used glm and kimi subagents for implementation, with adversarial reviewers catching 5 critical bugs that were fixed before commit.

---

## On the project

The cert-watch codebase continues to be well-organized and pleasant to work in. The migration system, database repository pattern, and route/auth conventions are consistent enough that subagents could follow them without much guidance. The AGENTS.md is genuinely useful — it's the first thing each subagent reads and it grounds them quickly.

The biggest tension I see is the gap between "tests pass" and "correct behavior." Multiple subagents produced green tests that encoded wrong behavior (the margin formula in readiness.py, the lifetime computation using `first_scanned_at` instead of `not_before`, the `stdev` vs `pstdev` choice). The adversarial reviewers were the only thing that caught these. This pattern — implementation passes its own tests but contradicts the spec — is the dominant failure mode for subagent-driven development in this project.

## On the work done

The work went smoothly given the scope. Sequencing was right: WI-1.1 (smallest, defect) → WI-2.1 (substrate) → WI-1.2 (depends on 2.1) → WI-2.3 (independent policy pack) → WI-2.2 (headline, depends on 2.1) → WI-3.1/3.2 (parallel, lower priority).

Critical bugs caught by adversarial review:
1. **WI-2.1:** Lifetime computed as `not_after - first_scanned_at` instead of `not_after - not_before` — needed a migration (0023) to add `not_before` column to `cert_history`. This was the most impactful catch.
2. **WI-2.3:** `chain_status=None` hardcoded in `scan.py:432` — SC-081 policy rules never fired in production despite passing all unit tests. Required plumbing `chain_status` through `_evaluate_and_store_posture()`.
3. **WI-2.2:** Margin formula contradicted the plan spec (`lifetime - max_days + lead_time` instead of just `lead_time`), milestone dates were wrong (2027 vs 2029 for 47d), workload forecast was per-year labeled as per-month.
4. **WI-1.2:** Missing "same fingerprint" guard (false-positive on recently renewed certs) and no event deduplication (would spam event_log every scan cycle).

I'm confident in the final state — every issue was traced to a file:line reference and verified with test runs. The one area I'd want a second pair of eyes on is the readiness report's `_compute_margins` — the semantics of "margin" are subtle and I had to re-read the plan prose multiple times to get the right interpretation.

## Gaps to flag

- `readiness.py:50-60` — `_get_chain_status()` calls `init_schema()` per host and opens a separate connection. Should be batched. Filed as WI-018.
- `renewal_analytics.py:_compute_trend` — for odd-length lists the middle element goes to the second half, biasing trend detection. Low impact but noted.
- `policy.py` — the subagent modified core classes (`PolicySet` added `name`/`version`, `PolicyViolation` added `grade_affecting`). These are backward-compatible but the policy API routes may silently drop `name`/`version` on GET/PUT round-trips. Not tested end-to-end.
- `scan.py:_evaluate_and_store_posture` — return type changed from 2-tuple to 3-tuple. Only two test callers existed outside scan.py and were updated, but this is a private function so the risk is contained.
- `digest.py` — the `send_renewal_digest` function depends on SMTP config being set up. If no SMTP is configured, digests silently don't send (consistent with existing alert behavior, but worth noting).
- `posture.py` — the scan_cadence finding was wired into `evaluate_posture()` but the caller in `scan.py` reads `scan_interval_hours` from the hosts table and passes it as `scan_interval_days`. The conversion logic (`max(scan_interval_hours / 24, 1)`) floors to 1 day minimum, which means hosts with sub-daily scan intervals won't trigger the finding even if they should.
