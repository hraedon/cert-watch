## On the project

The cert-watch codebase is genuinely well-structured for a solo/small-team project. The adapter seam was trivially introduced because `send_webhook` was already a clean, self-contained function with a clear contract. The renewal tracking path (`replace_scanned` → `replaces_cert_id`) was already in place and well-tested, so Slice 4 (resolve-on-renewal) was mostly wiring rather than design.

One thing that struck me: `config.py`'s `from_env_with_kv` is now a 530-line method that touches every field. It works, and the `_kv` helper pattern is consistent, but it's becoming a maintenance burden. BC-094 (decompose queries.py) is filed; a similar decomposition for config would be worth tracking.

## On the work done

The plan was clear and well-scoped — one of the better plans in the repo. The payload specs (Discord embeds, Teams Adaptive Cards, PagerDuty Events API v2) were exact enough to implement without external lookups.

What went well:
- The adapter seam was a clean extraction. Moving generic webhook logic into `GenericAdapter.build()` was mechanical and backward-compatible.
- `replace_scanned` returning `(leaf_id, replaced_cert_id)` instead of just `leaf_id` was a minimal API change that enabled Slice 4 without touching any callers that ignore the return.
- All 853 tests pass first try after the changes.

What I'd want a second pair of eyes on:
- The `webhook_config` threading through `store_scanned` → `store_scanned_async` → route handlers uses `object | None` as the type hint (to avoid circular imports from alerts.py in scan.py). It works but it's not ideal — a protocol or a `TYPE_CHECKING` import would be cleaner.
- `send_expiry_digest` still has a direct JSON webhook path (not going through the adapter). The plan explicitly scoped adapters to per-alert delivery, and digest is a batch concept. This is correct per spec but worth noting.

## On what remains

- The plan mentions a GUI "kind selector" and "Send test alert" button in Settings → Alerts. That's UI work not done this session — it's in the plan's Config surface section but not in the slices. Worth scoping as a follow-up.

## Gaps to flag

- `scan.py:660` — `webhook_config` typed as `object | None` instead of `WebhookConfig | None` to avoid circular import. Works at runtime, weak at type-check time. A `TYPE_CHECKING` guard or a small protocol in a shared module would fix this.
- `config.py:from_env_with_kv` — the merge function is now 530 lines and handles ~40 fields. Each new config field adds ~5 lines of merge logic. No bugs, but error-prone to extend.
- No test for the end-to-end renewal → PagerDuty resolve flow (i.e., calling `store_scanned` with a `webhook_config` after a cert is replaced). The individual pieces are tested (adapter resolve, `resolve_pagerduty_for_renewed_cert`, `replace_scanned` return tuple) but not the integrated path through `store_scanned`. A test with a mock DB and mock urlopen would catch wiring regressions.
- `test_alert_adapters.py` doesn't test the Teams Adaptive Card with a null `threshold_days` (the `—` dash display). Low risk but the Discord path has it covered and Teams doesn't.
