# Session Reflection — 2026-06-06

**Work summary:** Reviewed three plans (035 RBAC, 037 TOFU CA, 038 Synthetic LDAPS CI) implemented by three different models. Found and fixed a logic bug in `require_write` where RBAC-mapped admins fell to legacy `_may_write`, plus lint errors and weak tests in the RBAC module. ## On the project

cert-watch is in a mature state. The auth stack (LDAP, OAuth, local break-glass, RBAC) is the most complex surface in the codebase, and it shows — middleware.py is 640+ lines of interleaved concerns (CSRF, rate limiting, auth, RBAC, security headers). The modularity is good at the package level (`auth/`, `database/`, `routes/`) but the middleware itself could benefit from decomposition. The test suite is strong (1287 tests, ~90% coverage) and the conventions (markers, CI job separation, coverage ratchet) are well-established.

The TOFU CA flow (Plan 037) is a nice example of extending an existing endpoint without breaking the API contract — the `tofu` key in the JSON response is additive, and the capture probe reuses existing SSRF guards. Clean seam work.

## On the work done

Reviewing three plans by three different models in one session was instructive. The hardest part wasn't finding bugs — it was distinguishing genuine bugs from intentional patterns. The `require_write` condition `auth_ctx.roles != ['admin']` *looks* like it means "admins always pass" but actually means "admins always fall to legacy." It took me several reads to confirm it was wrong.

The RBAC bug was the session's real contribution. Everything else was confirmatory — Plan 037 was clean, Plan 038's remaining issues were cosmetic. I'm confident in the `require_write` fix; the test suite passing confirms backward compat, and the logic now correctly checks "is RBAC configured?" instead of "is the user an admin?"

What I'd want eyes on: the `require_write` fix reads `settings.role_map` from `app.state` on every request. This is a dict lookup, not a DB hit, so it's fine performance-wise — but it means changing the role map at runtime (via the settings UI) takes effect immediately on the next request, without a session invalidation. That's probably the right behavior, but it's worth noting.

## On what remains

- **Plan 035 Slice 3 (session carries roles):** `require_auth` currently passes `user_groups=[], user_roles=[]` to `build_auth_context`, so RBAC resolution always returns the default role. The session token needs to carry groups/roles from login for RBAC to work end-to-end. Without this, `CERT_WATCH_ROLE_MAP` is effectively inert.
- **Plan 035 Slice 4 (wire enforcement):** `require_write_form` still calls `_may_write` directly, bypassing RBAC. Template `may_write`/role display is not wired.
- **Plan 037 minor:** `_is_cert_verify_error` doesn't match "expired certificate" phrasing. No PEM size cap on captured chain. `capturedLdapCaPem` not cleared after pin in JS.
- **Plan 038:** `test_mutation_verify.py` duplicates the Samba fixture instead of sharing via conftest. Minor — works, just spins up 2 containers if both files run.

## Gaps to flag

- `middleware.py:568` — `require_write_form` does not use RBAC (`AuthContext`); it calls `_may_write` directly. Form-POST mutations (add host, upload, delete) bypass role-gating even when a role map is active. (`middleware.py:598-619`)
- `middleware.py:548` — `require_auth` always passes `user_groups=[], user_roles=[]` to `build_auth_context`. Until the session token carries IdP claims, RBAC resolution is cosmetic — everyone gets the default role. (`middleware.py:548`)
- `routes/settings.py:562` — `_is_cert_verify_error` omits "expired certificate" / "cert has expired" (OpenSSL phrasing). A server with an expired cert wouldn't trigger TOFU capture. (`routes/settings.py:562-577`)
- `routes/settings.py:443` — `_capture_ldaps_chain` SSRF guard only checks IP literals, not hostnames. The caller (`test_ldap_connection`) checks hostnames, but direct calls to this function would bypass the guard for DNS-resolved targets. (`routes/settings.py:474-479`)
- `templates/settings.html:347` — `capturedLdapCaPem` JS variable is never cleared after a successful pin. Idempotent but retains the PEM in memory after the UI hides the panel. (`templates/settings.html:347`)
