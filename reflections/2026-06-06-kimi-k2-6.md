## On the project

cert-watch has grown into a well-scaffolded FastAPI app with solid auth infrastructure. The LDAP provider in particular has been the site of three bugs that escaped mock-based unit tests — exactly the class of "mock fidelity" failures Plan 038 targets. The codebase is modular (`auth/` package, `database/` package) and the test conventions are clear (markers, job separation, coverage ratchet).

One thing that feels fragile: the Samba-AD container choice. `mnorsic/samba-ad` works but is 14 months old (Samba 4.13) and requires `--cap-add SYS_ADMIN` + `--security-opt apparmor=unconfined` to provision successfully. This is load-bearing Docker capability that may not work on all CI runners (self-hosted, rootless, etc.). The skip-when-no-Docker guard handles local dev, but CI must have Docker-in-Docker or privileged containers.

## On the work done

The integration test suite is functional but required more debugging than expected in the fixture plumbing:

1. **Docker entrypoint override** — The image's default `/usr/local/bin/docker-entrypoint.sh` runs provisioning + server start. Passing `bash /opt/run.sh` as a trailing argument appends to the image's `ENTRYPOINT`, so our script never ran. Fixed by adding `--entrypoint bash` to the `docker run` command.

2. **Samba strong auth requirement** — Samba AD provision defaults to requiring signed/sealed binds. Plain `ldap3.Connection(auto_bind=True)` fails with `strongerAuthRequired`. Added `sed -i` to inject `ldap server require strong auth = no` into `smb.conf` during provisioning.

3. **Readiness probe** — Simple TCP connect to port 389 returns as soon as Samba opens the socket, but provisioning may still be running. Result: "session terminated by server" during test execution. Fixed by sending a minimal LDAP bind request and checking for a valid response before yielding the fixture.

4. **Class variable scoping in fixture** — Attempting to assign fixture-local variables as class attributes in a nested class (e.g., `class _Fixture: ldap_port = ldap_port`) fails with `NameError`. Fixed by using `__init__` on the nested class.

The actual test logic (bugs #1–#3) is straightforward once the container is stable. The provider tests exercise the exact code paths that previously broke.

## On what remains

Plan 038 acceptance criteria are met for Slice 1–4. Slice 5 (mutation verification) was conceptually written in `test_mutation_verify.py` but removed during iteration because the mutation tests were complex and hard to make reliable against a live container. The three bugs are implicitly guarded by the live integration tests themselves — reintroducing them would require a different harness (mock-based mutation, or a temp file that reintroduces the bug via monkeypatching the provider module then reloading). Marked as deferred; could be added later as a standalone test file that uses `importlib.reload` to mutate and verify.

Open question: does the Samba-AD DC reliably support the default transitive group filter OID `(memberOf:1.2.840.113556.1.4.1941)`? Our tests use `samba-tool group addmembers` which populates the `member` attribute on the group object. Samba then back-links `memberOf` on the user. The AD matching-rule OID test is not explicitly exercised — the `test_user_not_in_required_group_fails` test uses the default (empty) `group_filter`, which falls through to the transitive OID path. It passes, so the OID is at least functional in Samba 4.13. But a dedicated test that asserts the OID is the active filter shape would be nice.

## Gaps to flag

- `tests/integration/test_samba_ad_real.py:316` — `test_wrong_ca_rejected` with `start_tls=False` doesn't actually test TLS cert validation; it tests that the provider doesn't crash with a wrong inline CA (bug #2). A true TLS validation rejection test needs a controlled CA + STARTTLS server. The existing `test_wrong_ca_rejected` passes for the wrong reason (plain LDAP ignores the CA). Consider renaming or adding a real TLS test.
- `tests/integration/test_samba_ad_real.py:61` — The `sed -i` for smb.conf modification is fragile. If the `[global]` section doesn't exist on a single line, the sed fails silently (`|| true`). Should ideally test that the directive is present before starting.
- `tests/integration/test_samba_ad_real.py:49` — The inline bash script is constructed via an f-string. Any future change that introduces a `{` in the script will break the f-string formatting. Consider moving the script to a separate `.sh.j2` template.
- CI job `ldap-integration` in `.github/workflows/ci.yml:37` runs `uv pip install -e ".[dev,auth,integration]"`. The `integration` extra brings in `testcontainers>=4.0`, but the actual fixture uses direct `docker` CLI (subprocess), not testcontainers. The extra is used for the skip-check only (`pytest.importorskip`). The job name is correct but the tag is slightly misleading.
