## Gaps to flag

- `test_route_coverage_api.py` error response format is inconsistent — some endpoints return `{"error": "..."}`, others return `{"detail": "..."}`. The content assertions I added use `.json()["error"]` which works for the tested paths, but a wrong error key would still pass if it's in a different field. The API should standardize on one error key.
- The `_is_blocked_ip` tests in `test_ssrf_csrf_ratelimit.py` test the same function more thoroughly than the deleted `test_extra_coverage.py` copies did, but the IPv6-mapped private/loopback cases (`::ffff:127.0.0.1`, `::ffff:10.0.0.1`) are only in `test_ssrf_csrf_ratelimit.py` — if that file is ever refactored, those edge cases could be lost.
- The parametrized config tests in `test_extra_coverage.py` now test `base_url` with a trailing slash (exercising `rstrip`), but the original `test_config_base_url` didn't — so the parametrized test is strictly better. Still, `Settings.from_env()` is a thin wrapper around `os.environ.get` and the tests remain somewhat tautological even when parametrized.
