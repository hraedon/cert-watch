## On the project

cert-watch is a lean certificate monitoring tool that's evolved beyond its original spec into something with real production concerns: auth providers, SSRF protection, CT monitoring, rate limiting, and chain validation. The architecture is straightforward — FastAPI + SQLite + daemon scheduler — and the codebase is well-organized despite being largely agent-built.

What works well: the repository pattern for DB access, the clear separation of scanning from persistence, the optional auth design. The spec documents in `docs/spec/` are a good contract.

What feels fragile: the chain extraction path on Python 3.12 (now has an openssl fallback that opens a second connection), and the string-based DN comparison (just fixed to use DER bytes). The project has accumulated several "works on 3.13, workaround on 3.12" patterns that will need cleanup when the container base is upgraded.

## On the work done

The BC-023 fix (DER-based issuer/subject comparison) is the one I'm most confident in — it's a clean change that adds optional bytes fields to the Certificate model and uses them when available, falling back to string comparison for DB-loaded certs.

The BC-022 openssl fallback is functional but imperfect. It opens a second TLS connection which is wasteful and could hit rate limits. It's the right trade-off for Python 3.12, but deserves the BC-027 observation breadcrumb.

The BC-024 CA validation is straightforward and correct — BasicConstraints.ca is the authoritative check. The BC-025 private IP hint is a small UX improvement that makes the error actionable.

The E2E test improvement (BC-017) is minimal but addresses the specific gap: asserting the host row and scan-history failure entry.

## Gaps to flag

- **`scan.py:142`** — The openssl subprocess fallback runs `subprocess.run()` with a 10s timeout. Combined with the Python SSL timeout (10s default), a single failed scan can take up to 20s. No backoff or cancellation mechanism.
- **`cert_chain.py`** — The `_subject_bytes`/`_issuer_bytes` helpers fall back to UTF-8 string encoding for DB-loaded certs. This means chain validation of DB-loaded chains is still string-based. The DER bytes are only populated during fresh TLS scans or uploads. Not a bug per se, but means the fix is incremental.
- **`app.py:45`** — The `_PRIVATE_NETWORKS` import from `scan.py` inside `_is_blocked_host()` is a late import to avoid circular dependencies. Works, but makes the dependency relationship implicit.
- **`database.py`** — The `TrustAnchorEntry` dataclass was missing (F821 lint error). This is a pre-existing bug that would have caused a runtime error if `list_entries()` was called. Fixed this session.
- **Test coverage** — The openssl fallback path (`_get_chain_via_openssl`) is untested. It requires either mocking `subprocess.run()` or an integration test against a real TLS server.
