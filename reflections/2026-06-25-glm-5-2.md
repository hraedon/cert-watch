## On the project

This is a genuinely well-tended codebase, and the maintenance-mode framing isn't a fig leaf — it shaped my decisions all session. The embarrassment-class UI bugs that the AGENTS.md checklist warns about (silent CSS-class collapses, inline `on*=` handlers, raw ISO timestamps, `|safe` on user data) are all **clean**. The token system has full dark/light coverage and reserves status colors for status, accent for active/focus. The test estate is large (2256 unit + 60 functional e2e) and gates are real. For an SMB cert-lifecycle tool this is more discipline than most shipped products.

The sharpest *design* debt (not bug debt) is coherence across surfaces: the three report pages (readiness/compliance/crypto) each reinvent their own `rdy-*`/`cmp-*`/`cry-*` component vocabulary in local `<style>` blocks instead of using `cw-*`, and the dashboard carried a redundant stat-card + urgency-segment-row that showed the same numbers twice. Both are consolidation opportunities, not feature work — and the operator explicitly confirmed UI consolidation is in-bounds even under maintenance mode (the designation is about halting *new feature surfaces*, not freezing the existing UI). That clarification mattered: it unblocked the stat-card-as-filter redesign I'd otherwise have WI'd and left.

## On the work done

The implementer→adversarial loop earned its keep both rounds. Round 1 (defect fixes): the reviewer caught that the team-dashboard Healthy card had the identical green-at-zero bug the implementer fixed on the main dashboard (a parallel-template miss), a third lowercase "expired" location, and that the page-scope stat loop was now dead code. Round 2 (coherence): the reviewer caught that making the "Tracked certificates" card show the *filtered* total while the urgency cards showed *fleet* totals created a fresh inconsistency under an active filter — exactly the kind of regression a "remove redundancy" change can introduce. I fixed all three. This is the argument for always pairing implementation with an independent review pass on a different lineage: the implementer normalized past the edge cases, the reviewer caught them because they hadn't built the thing.

I'm confident in: the `dashboard_urgency_stats()` SQL (thresholds exactly mirror `compute_urgency` and `pivot_urgency_stats`, reviewed line-by-line); the stat-card-as-filter wiring (param-carry consistent with the existing `sort_link` pattern, live-verified under `?urgency=expired`); the WI-088 auth change (sig-length consistency verified — `create_session` issues 64-char, so only genuinely legacy tokens are rejected; regression tests genuinely assert the reversal).

I'm **less** confident in two things. (1) The visual baseline fallout: `94bc049` structurally changes the dashboard, so `dashboard.png`/`dashboard-populated.png` will fail the CI visual gate and must be regenerated from the ubuntu-latest artifact — I did not do this (correctly, per AGENTS.md) but it means the tree is "green locally, red-on-push" until that baseline update lands. (2) The CSRF 32→64 change in `fa0726f` is marginal security value (128-bit hex was already collision-resistant); it's harmless and brings consistency with session sigs, but I'd characterize it as hygiene rather than a real vulnerability fix. The session-token legacy rejection is the substantive security change.
