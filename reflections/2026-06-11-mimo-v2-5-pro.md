## On the project

cert-watch is a well-architected single-binary FastAPI/SQLite app. The spec-driven approach (docs/spec/wi_*.md) gives the project a clarity that most projects lack — every feature has acceptance criteria before code is written. The decomposition milestones (Plans 018-048) show a disciplined approach to technical debt: auth was decomposed from a monolith, database got the repository pattern, config was modularized. The ~90% test coverage is genuine (not theater) — the 2026-06-04 test-quality pass converted assertion-of-redirect into actual behavioral checks.

The SQLite single-writer model is the right call for this deployment profile (single-instance, k8s Recreate). The SSRF guard, CSP nonces, session revocation, and compliance signing are all solid implementations. The project's biggest structural weakness is the error-handling pattern: 100+ bare `except Exception` blocks that swallow everything. This is the natural consequence of the "failure isolation" design (scheduler stages must not kill each other), but it's been applied too broadly — `scan.py` had 11 identical `try/except Exception` blocks with no way to tell which stage failed.

## On the work done

**What went well:** The multi-agent parallel dispatch worked effectively. The key design decision was giving each agent exclusive file ownership — zero conflicts between the 5 fix agents. The review phase was thorough: the adversarial reviewers found genuine security bugs (OIDC exp inversion, DNS rebinding TOCTOU, CSRF binding to wrong cookie) that would have been missed by a single-pass review.

**What was awkward:** The HMAC truncation change (32→64 hex chars) broke backward compatibility with existing 3-part session tokens. The agent that made the change didn't add a legacy fallback, so I had to fix `test_validate_session_3_part_valid_shape` manually. This is a pattern risk: when you change cryptographic primitives, the blast radius includes every existing token in the wild. The test caught it, but only because the test constructs tokens manually — if it used `create_session()` it would have passed silently.

**What I'm less confident in:** The SSRF IP pinning fix (http_client.py) is the most architecturally significant change — it replaces urllib's default connection flow with a custom handler that pins to a pre-resolved IP. This is the right approach (scan.py does the same thing for TLS scanning), but the interaction between the pinned HTTPS handler, the redirect handler, and urllib's internal connection pooling is complex. I'd want integration tests against a real HTTP server to verify it doesn't break on edge cases (redirects to HTTPS, IPv6 literals, non-standard ports).

**Net impact:** 16 files changed, +670/-351 lines. 3 critical/high security bugs fixed, 5 architecture improvements, 4 code quality fixes, 3 bug fixes, 4 security hardening changes.

## Gaps to flag

- **SSRF IP pinning needs integration tests.** `http_client.py` now has a custom `_PinnedHTTPSHandler` that overrides urllib's connection logic. The unit tests mock urllib, so they don't exercise the actual TLS/SNI behavior. An integration test against a real HTTPS server (or a local test server with SNI) would catch edge cases. `http_client.py:86-104`
- **Legacy session token fallback is silent.** `auth/session.py:243-247` — when a 32-char legacy signature validates, there's no log message. An operator can't tell how many old-format tokens are still in circulation. Consider logging at DEBUG on legacy fallback hit.
- **`_stage_posture` in scan.py swallows exceptions silently.** The refactored stage helpers use `contextlib.suppress(Exception)` for non-critical stages. This preserves the original behavior but means posture evaluation failures are still invisible. The tagged logging in `_stage()` wrapper helps, but only if the caller checks the return value. `scan.py:~355-365`
- **`filters.py:compute_urgency_with_chain` is new and untested.** The dedup of `_urgency` in `views.py:team_dashboard` now delegates to this function. There's no dedicated unit test for the chain-aware variant — the team dashboard e2e test covers it implicitly but doesn't exercise edge cases (e.g., `chain_status="incomplete"` with `min_chain_days < leaf_days`). `filters.py:87-102`
- **compliance.py key derivation uses raw `signing_key.encode()` without length check.** If `signing_key` is empty or very short, the derived key is weak. The `signing_key` comes from `SecurityContext` which is generated via `secrets.token_hex(32)` (64 chars), so this is fine in practice, but the function's contract doesn't enforce a minimum length. `compliance.py:139`
