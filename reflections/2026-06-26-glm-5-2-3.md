## On the work done

**Confident in:** WI-117 (deny-by-default — unambiguously correct, reviewers confirmed no legitimate path is blocked), WI-090 (file-first ordering — the fail-closed path already existed, I just wired the provisioning to use it), WI-085 (delegation + property test — the property test is non-vacuous, verified by injecting a divergence).

**Less confident in:** WI-089's exception narrowing in scan.py. The kimi implementer's argument that all ssl.SSLError subtypes are OSError subclasses (and thus caught by the preceding `except (TimeoutError, OSError)`) was verified by the glm reviewer empirically. But the narrowing to `ValueError` only means that if a future Python version adds a non-OSError TLS exception type, it would propagate as an unhandled crash. The old `except Exception` was too broad; the new catch is correct for today's exception hierarchy but is a tighter contract.

**The WI-091 lock change** (holding `_init_lock` across the entire `init_schema` body) is the change I'd most want a second pair of eyes on. I verified no migrations call `kv_set`/`init_schema` recursively (which would deadlock a non-reentrant Lock), but that's a fragile invariant — if a future migration writes to kv_store, the lock would need to become an RLock or the pattern would need restructuring. The test doesn't guard against this future regression.

## Gaps to flag

- **WI-091 lock invariant is undocumented**: `init_schema` holds a non-reentrant `threading.Lock` across `ensure_base` + `run_pending_migrations`. This is safe ONLY because no migration calls `kv_set`/`kv_set_multi` (which call `init_schema` recursively). There's no test or comment guarding this invariant. A future migration that writes to kv_store would deadlock. (`schema.py:339`)
- **WI-085 per-cert performance**: `resolve_group_recipients` now runs the full batch resolution (3 SQL queries) on every call. The only production caller is the single-request preview endpoint, so this is fine — but if someone wires it into a loop, it's O(N) full-table scans. No guard or comment. (`alerts.py:940`)
- **WI-090 stale password file**: if `kv_set_multi` fails after the file is written, a stale `initial-admin-password` remains on disk. Overwritten on next boot, but the window exists. The kimi reviewer suggested unlinking on failure — I skipped it as a nit. (`app.py:159`)
- **E2E suite not run**: the unit suite (2288 tests) is green, but the e2e Playwright suite was not run locally. CI's e2e gate is the authority for UI-interacting routes, but the WI-085 endpoint change touches the alert-routing preview which has e2e coverage. The visual baseline job (screenshot comparison) was not triggered since no templates changed.
- **The `_CSRF_BYPASS` conftest flag** (WI-099, still open): the unit suite globally disables CSRF. None of my new tests exercise CSRF validation. A CSRF regression would pass the unit suite and only be caught by CI's e2e job.
