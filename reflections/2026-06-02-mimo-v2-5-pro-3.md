## On the project

The codebase is in solid shape. The prior session's webhook SSRF fix was the last real security gap; what remains are design-level decisions (kv_store encryption, CSP nonce rewrite, OAuth nonce binding) that need deliberate architectural choices before implementation. The logging discipline in `store_scanned` was a genuine blind spot — four bare `except Exception` blocks logging at DEBUG meant posture, drift, and history failures were invisible to operators. The rest of the codebase uses WARNING consistently for data-loss scenarios; `scan.py` was an outlier because the inline `import logging` pattern (instead of a module-level logger) made it easy to treat each block independently rather than as part of a consistent logging strategy.

## On the work done

Small, surgical session. Three files changed, 24 lines added, 12 removed. The `scan.py` fix was mechanical but important — adding the module-level logger and removing 4 redundant inline imports. Both follow the existing ≥4-char threshold convention to avoid false positives.

The AGENTS.md sync revealed BC-031 was listed as unresolved but doesn't exist in the agent-notes store — likely resolved or never formally filed. Removed it and added BC-020/BC-021 which are tracked as "new" status.

## Gaps to flag

- `send_webhook` in `alerts.py:345-347` stores raw exception text in `alert.error_message` (persisted to DB). If the webhook URL contains embedded auth tokens, those end up in the database. Lower priority than logging since DB access is already privileged, but worth noting.
- The `store_scanned` bare except pattern (now logging at WARNING) still silently continues execution. If posture evaluation fails, the cert gets stored with an empty `posture_grade`. This is intentional (scan should succeed even if posture doesn't), but the empty string could confuse downstream queries that assume non-empty grades.
