## On the project

cert-watch is in a solid maintenance phase. The spec-driven architecture (plans 001–019), SQLite-backed everything, and clean separation between `routes/`, `database/`, and `alerts.py` make it obvious where a change should live. The test suite at 617 tests is genuinely fast enough (~80s) to run on every commit.

One observation from touching the templates: Jinja2 macros are well-organized (icons macro extracted), but the inline JS in `base.html` is growing. The new health-banner script is ~60 lines inside a script tag — not a problem today, but if more widgets follow this pattern it'll become the same kind of maintenance issue the CSS macros were. A `static/js/` split for page-agnostic utilities would be a natural next refactor.

## On the work done

Plan 018 A1 was straightforward — the kv_store fallback logic was already isolated; moving it into `build_auth_provider()` was mostly deleting the duplicate block in `app.py`. The 25 setup-bootstrap tests all stayed green without changes, which is the best validation that the refactor was behavior-preserving.

Plan 018 A2 (batch resolver) I felt good about. The `resolve_all_group_recipients()` function is self-contained, uses only the query layer's `_connect()`, and has the same matching semantics as the per-cert resolver. The only caveat: it builds a `dict[str, list[str]]` in Python so memory usage scales with cert count — fine for the intended SMB scale, but if cert count grows past ~10K the dict would be worth streaming. At current volumes this is a non-issue.

Plan 012 Phase 2.1 was more involved than I expected. The `/api/health` endpoint needed four separate data sources (scheduler thread, scan_history, alerts, auth state). Templating the banner into `base.html` was fine, but wiring the JavaScript fetch loop, dismiss state via `sessionStorage`, and the tone class toggling required care. I'm moderately confident the JS is correct but it has no automated test coverage — it was verified by reading the rendered HTML in browser-source mode but not via a headless assertion. We should not rely on this staying correct across template refactors without a small E2E test or at least a DOM-integration test.

## On what remains

Plan 018 A3 (Depends(require_auth/require_write) sweep) is the last remaining Phase A item. It's a mechanical refactor — replace ~29 `_require_api_auth` / `_require_api_write` call sites in `routes/api.py`, and the inline auth checks in `certificates.py` and `hosts.py`, with the `Depends()` versions I already added to `middleware.py`. It's low-risk but tedious.

Phase B items (SecurityContext + create_app factory, purpose-built dashboard queries) are larger and should be a dedicated session.
