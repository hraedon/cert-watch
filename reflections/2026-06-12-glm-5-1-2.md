# Session Reflection — 2026-06-12

**Work summary:** Two-round adversarial audit of the full codebase using parallel subagents (kimi + adversarial-reviewer + adversarial-reviewer-2). Round 1 covered security, auth, data layer, and API correctness. Round 2 focused on scan.py (the lowest-coverage module at ~73%) and template XSS surface. Remediated 13 issues across both rounds, all 1955 tests passing.

---

## On the project

The codebase is genuinely well-engineered for its size and purpose. Multiple independent reviewers noted this unprompted. The security posture is solid: parameterized queries everywhere, HMAC-signed sessions with version-based revocation, CSP nonces, CSRF double-submit, SSRF guards on webhooks and OCSP probes. The architecture is straightforward (FastAPI + SQLite + Jinja2) and doesn't try to be clever.

The main weakness is the scan.py/scan_conn.py subsystem — it's the most complex module (subprocess management, native TLS fallback, chain extraction across Python versions) and has the lowest test coverage. The adversarial audit found real bugs there: an assert in production code, a hardcoded verify=False in the fallback path, IPv6 broken in the openssl connect string, unbounded subprocess output buffering. These are the kind of bugs that survive because the code path only fires on Python <3.13 or when openssl is missing — environmental edges the test suite can't easily cover.

## On the work done

Confident in the fixes. Each was verified against the full test suite (1955 pass). The most impactful:

- **Transaction split in `replace_scanned`** — the `renewal_status` update was in a separate transaction from the cert write, meaning process crashes could leave stale `renewal_status='renewed'` that silently suppressed expiry alerts. Moved into the main transaction. This is the most production-impactful fix.
- **`/audit` missing auth gate** — the HTML page was open to unauthenticated users while the JSON API was gated. Simple fix, high-severity finding.
- **`store_uploaded` per-cert transactions** — each chain cert was its own transaction, leaving partial uploads on failure. Refactored to single transaction.
- **TLS verify downgrade** — the openssl fallback hardcoded `verify=False` regardless of caller intent.

The two findings I'm less sure about:
- The `is_admin` fail-closed fix — I changed it to `no_auth → True, auth_on but no context → False`. This preserves the auth-off (NoAuthProvider) path while closing the window where auth is on but the middleware pipeline failed to set a context. I believe this is correct but it touches the auth/template interaction and deserves a second look.
- The clock-skew clamp in `evaluate_thresholds` — `max(0.0, ...)` prevents negative diffs from permanently suppressing alerts. The logic is sound but I didn't verify what the alert_created_at timestamp source looks like in practice (DB clock vs Python clock).

## On what remains

The unfixed findings from the audit, in rough priority:

1. **WI-036: IPv6 in openssl -connect** — I filed this but the fix is already in the diff for this session (bracketed IPv6 in `scan_conn.py`). WI-036 should be closable on push.
2. **WI-037: Unbounded subprocess stdout** — needs a size cap before regex matching.
3. **WI-035: OAuth token exchange outside SSRF guard** — requires architectural change (patch authlib session or validate endpoint URL).
4. **Template audit M-1/M-2/M-3** — error params not plumbed on settings/events page, query_params read directly in discover.html, literal `+` instead of `quote()` in hosts.py error redirects. These are correctness/consistency issues, not security.
5. **Dead code in `scan.py:692-696`** — legacy repo-object path that bypasses all safety logic.

Beyond fixes, the project is in Plan 049 (maintenance entry) territory. The audit confirms what the plan says: the high-signal surface is mostly covered, remaining work is operational maturity rather than bug discovery.

## Gaps to flag

- **`scan.py` coverage at 73%** — the subprocess/socket paths are the main gap. These need environmental mocking (subprocess, socket) or integration tests with a real TLS server, not more unit tests.
- **`_stage` double-logging** — the resolve_pending_alerts stage was the only one with an outer try/except that double-logged with the inner `_stage` wrapper. Fixed, but the pattern is fragile — any future stage that adds its own try/except will reintroduce it.
- **Template `request.query_params` direct access** — `discover.html` reads `request.query_params.error` directly instead of going through the route handler. This is the only template that does this. It's safe today (autoescape) but inconsistent with every other error-display pattern.
- **No e2e coverage for scan.py fixes** — the IPv6 bracketing and verify-fix are not covered by e2e tests. They'd need a dual-stack TLS server fixture to test properly.
- **`settings/events` error/success feedback silently dropped** — the POST handler redirects with `?error=...` or `?saved=1` but the GET handler doesn't read these params. The template has the blocks ready; it's a plumbing gap.
- **"Expiring within 14 days" label was wrong** — the stat card showed `ns.critical + ns.warning` (which is ≤29 days) but the label said 14. Fixed to "Expiring within 30 days". This is exactly the "read the words on the screen" bug class AGENTS.md warns about.
