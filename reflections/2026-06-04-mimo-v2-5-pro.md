## On the project

This is a genuinely well-engineered security tool. The adversarial review surfaced only 6 issues in ~7,300 LOC, and the highest-severity one (CSP nonce on login/setup) was a functional bug, not an exploitable vulnerability. The defense-in-depth is reasoned, not checkbox — SSRF guards with IP pinning, DNS-rebinding hardening, double-submit CSRF with header-only acceptance, scrypt with timing-safe dummy verification, per-request CSP nonces with delegated event handlers. Each addresses a specific, real attack class.

The spec-driven approach (`docs/spec/wi_*.md`) with acceptance criteria is the project's backbone. The breadcrumb system for tracking open questions works well — the problem this session was that the AGENTS.md summary was stale, not that the system was broken.

The architecture is proportionate. Repository pattern, provider pattern for auth, adapter pattern for alert channels, dependency injection via `create_app()` — these are the right abstractions for the problem size. The `SecurityContext` as an immutable dataclass on `app.state` is a clean solution to the request-less signing problem.

## On the work done

The adversarial review was the most valuable part. Reading the full codebase with an attacker's mindset found issues that incremental development wouldn't: the CT log module bypassing the SSRF guard (the only module that did), the CSP nonce missing on the two standalone templates that don't inherit `base.html`, the `fernet_decrypt` silently returning raw ciphertext on failure.

The breadcrumb audit was revealing — 7 of 12 "open" breadcrumbs were already resolved in code. The AGENTS.md backlog listed 24 identifiers, of which 16 had never been filed as breadcrumbs at all. The disconnect between the documentation and the actual state was the biggest gap.

The `_load_unified_filtered` refactor (BC-073) went deeper than expected — I added a `_build_host_filter` helper with a column-name whitelist, an `include_uploaded` flag, and deleted `_list_unified_entries_raw` entirely. The `list_unified_entries` function now uses SQL-level EXISTS subqueries instead of loading the full inventory into Python.

I'm confident in all the fixes. The `fernet_decrypt` change (returning `None` instead of raw ciphertext) required updating 4 callers — I traced each one and verified the fallback behavior is correct. The `_escape_like` + `ESCAPE '\'` pattern is the right SQLite approach. The BC-071 nonce verification in the userinfo path checks the claim when present, rejects on mismatch, and warns when absent — three distinct outcomes, each appropriate.

## On what remains

The 16 non-breadcrumb work items in AGENTS.md (BC-020, BC-021, BC-093–BC-100, BC-103, BC-104, BC-114, BC-120–BC-123) are real but untracked in the breadcrumb store. They should be filed or dropped. The most impactful are:

- **BC-094** (decompose queries.py by concern) — 2,400+ lines, the biggest maintenance risk
- **BC-104** (REST API machine-to-machine auth) — needed for any automation story
- **BC-120/BC-122** (compliance report N+1 and full-table load) — performance at scale

The `style-src 'unsafe-inline'` CSP gap is documented and intentional, but worth noting: any injected `<style>` block would be allowed. Jinja2 autoescaping prevents template injection, so the realistic vector is supply-chain (compromised static CSS files).

## Gaps to flag

- **`queries.py` is 2,480 lines.** The monolithic file is the single biggest maintenance risk. `_build_host_filter` is a band-aid; the real fix is decomposition by concern (BC-094).
- **16 AGENTS.md backlog items never filed as breadcrumbs.** They exist only in a text paragraph — invisible to the breadcrumb store, search, and the `/start` skill. Either file them or remove them.
- **`ct_lookup.py:_ct_log_url()` uses a module-level cache.** The env var is read on first call and cached forever. Changing `CERT_WATCH_CT_LOG_URL` at runtime requires a restart. This is consistent with other import-time config but could surprise operators.
- **The compliance report calls `list_dashboard_rows` without pagination.** At fleet scale (thousands of certs), this loads everything into memory. Acceptable for on-demand admin exports, but not for scheduled jobs.
