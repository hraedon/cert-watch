## On the work done

Kimi did the bulk of the implementation (14 files, ~800 lines) but ran out of context before committing. The code was functionally correct but had several issues that the adversarial review caught:

1. **LIKE wildcard injection** (critical) — scope tags containing `%` or `_` would be interpreted as SQL wildcards, letting a scoped user see the entire fleet. Fixed by escaping with `_escape_like()` and adding `ESCAPE '\\'`.

2. **Missing scope enforcement** (critical) — `scope_write_denied` was only wired into 1 of 13+ mutation routes. A scoped operator could delete, edit notes, update owners, and trigger rescans for any host/cert. Fixed by wiring it into all 11 mutation routes.

3. **Code duplication** — `_scope_tags_from_auth` was duplicated 3 times, `_tags_with_scope` twice. Extracted to shared `_scoped.py` module.

4. **Inline SQL** — `api/hosts.py` hand-rolled its own LIKE-based tag filter instead of using the shared `_add_effective_tag_filter` helper. Replaced with the shared helper.

5. **Duplicate frozensets** — dashboard.py had `_SORT_COLUMNS_ALIAS`, `_SORT_COLUMNS_GROUPED`, and `_SQL_DIRS` defined twice (Kimi's diff inserted code between the originals and accidentally duplicated them). Removed the duplicates.

6. **Import issues** — Several files had unused imports, missing imports, or incorrect import ordering after the refactoring. Fixed.

The subagent (general) did good work wiring `scope_write_denied` into all 11 mutation routes — it found the right insertion points in each file and verified tests pass.

## On what remains

The adversarial review identified several medium/low issues that can be deferred:

- **Multi-role scope union** (design concern) — when a user matches multiple roles, scope tags are unioned. This means a user with `team-a:viewer` and `team-b:operator` gets write access to both teams. The behavior is documented and intentional, but worth noting.

- **Pivot stats scope leakage** — the aggregate stats in the pivot view (expired/critical/warning/healthy counts) don't respect scope tags. The pivot groups themselves are filtered, but the stats box leaks information.

- **`_effective_tags` for hosts** — when checking a host (not a cert), only host tags are returned, not effective tags (host ∪ cert). This means if a host has no tags but its cert has the team tag, mutation is wrongly denied.

- **`from_roles` sets tier to role name** — the legacy path sets `tier` to the first role name (e.g., "viewers") rather than a valid tier string. Works because the legacy path doesn't use tier, but could confuse future readers.

- **Test gaps** — no tests for LIKE wildcard injection, scope_write_denied, multi-role scope union, or uploaded cert auto-tagging round-trip.

## Gaps to flag

- `routes/api/hosts.py:48` — uses `SqliteHostRepository(db)._row_to_host(r)` (private method) to convert raw rows. Fragile if the method's interface changes.
- `database/dashboard.py:582` — uploaded cert scope filter passes `col_host="''"` which means the host-side LIKE never matches. Uploaded certs can only be seen by scoped users if the cert itself has a matching tag. Auto-tagging on upload (WI-052) handles this for UI uploads, but API-key-based uploads may miss it.
- `auth/rbac.py:240-254` — `_resolve_tier_and_scope` unions scope tags from all matched roles independently of tier. A viewer-role scope tag gets unioned with an operator-role scope tag, giving the operator write access to the viewer's scope.
