## On the work done

**What I'm confident in:**
- **WI-074 (chain_status):** The self-signed-root check was genuinely wrong — it returned "public" for any self-signed cert without verifying it was in a trust store. The fix reorders the check to verify against the system CA store first. This is the correct behavior per X.509 trust semantics.
- **WI-076 (HSTS timeout):** Clean plumbing fix. The setting was parsed and stored but never passed through to the function that uses it. Now it flows through the full call chain.
- **WI-077 (SMTP fallback):** The original logic was `if not any_smtp_success` — which excluded partial failures. Simplified to "webhook unless all SMTP succeeded" which is the correct fallback semantics.
- **WI-073 (certifi fallback):** Added certifi.where() as a fallback when Linux system CA paths don't exist. This is the right fix for Windows where the hardcoded paths are irrelevant.

**What I'd want second eyes on:**
- The certifi fallback adds a try/except ImportError block inside `_load_system_ca_cache`. certifi is a transitive dependency (via requests/urllib3) but isn't declared as a direct dependency. If the dependency tree changes, this could silently degrade. Consider adding `certifi` to pyproject.toml as an explicit dependency.
- The WI-077 simplification (webhook unless all SMTP succeeded) means that if SMTP is configured but ALL sends fail, the webhook fires with the full cert list. This is correct behavior, but the webhook recipient might get duplicate alerts if both channels are configured and SMTP partially succeeds later on retry.
