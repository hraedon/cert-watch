## On the project

The codebase held up well under a broad 9-issue fix sweep. The separation between scan_conn/scan/scan_resolver and the compliance/readiness modules kept blast radii small — each fix was contained to one or two files. The test suite (2410 tests) caught nothing directly (the bugs were in template logic and data-mapping, not unit-tested code paths), but it provided confidence that the backend changes didn't regress. The adversarial review was essential: it caught a dead retry loop (store_scanned swallowed the OperationalError before the retry could see it), a Jinja2 `loop.nextitem` null-deref, and a 5-week vs 4-week off-by-one. Without it, three of the nine fixes would have shipped broken.

## On the work done

**What went well:** The scan_conn.py regex fix was a one-liner with large impact — every OpenSSL 3.x deployment was storing empty protocol_version. The compliance fixes (uploaded pseudo-host, expired bucket, fallback posture eval) were clean data-modeling. The readiness unknown-classification fix is the right design: SC-081 doesn't apply to unclassified certs, and showing "not applicable" instead of red risk pills is correct.

**What I'm less confident in:** The `expiring_soon_count` now counts 4 weekly buckets (current week + 3 ahead). The stat card says "in next 4 weeks" which could mean "the next 4 weeks starting now" (which would be 5 buckets if current week has expiries). I chose 4 buckets because the calendar is weekly — a daily count would be more precise but would require changing the calendar data structure. The bucket tone coloring (`tone-crit` for current week, `tone-warn` for next) is a simplification — certs expiring later in the current week are treated the same as certs expiring today. This is acceptable for weekly buckets.

**The urgency change (Issue 10)** removes the chain-status downgrade from `compute_urgency_with_chain`. This means certs with incomplete chains that are far from expiry now show "Healthy" in the dashboard rows instead of "Warning". The chain chip and posture grade still surface the chain issue. I believe this is the right call — "Warning" was overloaded — but it's a behavior change that operators may notice. No regression test was added for this specific change; the existing tests were updated to match the new behavior.

## Gaps to flag

- **`store_scanned` re-raises `sqlite3.OperationalError`** — this is a behavior change for all callers of `store_scanned`, not just `store_scanned_async`. The scheduler's `_store_with_lock` in `app.py` calls `store_scanned` directly and doesn't retry. If it gets a database-locked error, it will now propagate instead of returning `""`. The scheduler's `run_scan_now` catches this as a store failure, so it's handled, but the error message will be the raw SQLite exception rather than "store failed".
- **No regression test for the urgency change** — `compute_urgency_with_chain` no longer downgrades healthy→warning on incomplete chains. The existing tests were updated, but no new test asserts "cert with incomplete chain and 100 days remaining shows Healthy, not Warning" as a behavioral contract.
- **Compliance fallback posture eval calls `evaluate_posture(cert=parsed, chain_status=None)`** — this evaluates posture without chain context, which may produce a different grade than the cert detail page (which passes the real chain_status). For uploaded certs without a chain, this is the best we can do, but the grades may differ.
- **Visual baselines updated** — the dashboard-populated.png and insights.png baselines were regenerated from CI. These are now the new ground truth; any future UI change will need the same `scripts/update-visual-baselines.sh --latest` treatment.
