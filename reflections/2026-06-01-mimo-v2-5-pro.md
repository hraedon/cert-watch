The migration runner pattern is pragmatic — numbered files with a registry, no ORM overhead. ## Gaps to flag

- **`Settings.from_env_with_kv()` `_kv_bool` helper** (`config.py:366`) — Can't distinguish "env var set to False" from "env var not set (default False)". If an operator sets `LDAP_START_TLS=0` in env and `ldap_start_tls=1` in kv_store, the kv_store value wins because the env var looks like the default. This is a known limitation; documenting it would help.

- **Settings page sensitive fields** (`routes/settings.py:130`) — When displaying masked passwords (`••••••••`), saving the form with the masked value would overwrite the real secret. The save handler should skip sensitive fields when the value is the mask string. Not yet implemented.

- **`/settings` route has no CSRF on test endpoints** — The `test-ldap` and `test-smtp` POST endpoints check CSRF, but the JavaScript `fetch()` calls in the template don't include the CSRF token in headers. They send it as form data, which works because `check_csrf()` reads form data too. But if CSRF is re-enabled (currently disabled in tests), the pattern should be verified.

- **No `/settings` link in the main navigation** — The settings page exists but there's no link to it from the dashboard header. Users would need to know the URL. Adding a gear icon or "Settings" link in the header would improve discoverability.
