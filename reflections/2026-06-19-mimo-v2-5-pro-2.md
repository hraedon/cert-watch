## On the work done

**What I'm confident in:**
- **`/readyz` not in public paths** — clear bug. Kubernetes readiness probes would fail when auth is enabled. One-line fix in `_PUBLIC_PATHS`.
- **`create_session` roles-stripping no-op** — the fallback code was supposed to strip roles from the payload but kept them. The `f"::{encoded_roles}"` should be `f"::"` to actually clear both groups and roles.
- **OAuth userinfo size limit** — JWKS and token responses were capped but userinfo wasn't. Added `_USERINFO_MAX_BYTES` constant and applied it.
- **`mark_alert_read` tag-scope bypass** — a scoped user could mark any alert as read, including those outside their team. Fixed by looking up the alert's cert_id and checking `scope_write_denied` before the UPDATE.
- **Fleet grade ignoring GRADE_WORST_ORDER** — the code computed `worst` ordinal correctly but then used an ad-hoc if/else for the grade string. Now derives from the ordinal directly.
- **CSS variable names** — `--color-text-3` and `--fz-12-5` don't exist; fixed to `--text-3` and `--fs-sm`.
- **Metrics urgency thresholds** — replaced inline if/elif chain with `compute_urgency()` to eliminate duplication.

**What I'd want second eyes on:**
- The `mark_alert_read` scope check does a SELECT before the UPDATE, adding one extra query per mark-read call. This is the same pattern used by `scope_write_denied` in certificate/host routes, so it's consistent, but it's worth noting the overhead.
- The `response_model=None` on the `mark_alert_read` decorator — needed because the function returns both `dict` and `JSONResponse`. Other routes in the codebase use this pattern, so it's consistent.

**False positive from the sub-agent review:** The "alerts mark-read button broken (JS syntax error)" finding was wrong. The template `{{ csrf_token | tojson }}` renders as `"token"` (consuming the `{{ }}` braces), producing valid JS: `fetch(url, {method: 'POST', headers: {'x-csrf-token': "token"}})`. The reviewer counted template braces as JS braces.

**Reverted:** The scan.py dead re-export cleanup was reverted after tests failed — `_is_blocked_ip`, `resolve_and_validate_host`, `_parse_allowed_subnets`, `resolve_hostname`, and `_PROTOCOL_RE` are all imported from `cert_watch.scan` by tests and other modules. They're intentional re-exports, not dead code. The `# noqa: F401` comments are correct.

## Gaps to flag

- **`capChips` dead code in dashboard.html:698-722** — the JS function selects `.cw-cert-chips` then looks for `.cw-san-list` inside, but `.cw-san-list` is a sibling. The SAN chips are never capped by JS. Server-side Jinja already caps at 2 SANs, so this is invisible dead code.
- **`escHtml` defined in both dashboard.html:614 and settings.html:975** — two slightly different implementations of the same function. Should live once in base.html.
- **`confirm-form` listener duplicated** in dashboard.html:723, certificate_detail.html:664, host_detail.html:217 — same handler copy-pasted three times.
- **`auth_ctx` type shadowed in routes/views.py:209 vs 252** — the variable changes type mid-function from `AuthContext | None` to `dict`. Works but fragile.
- **`init_schema` TOCTOU in database/schema.py:321-334** — the lock is released between the check and the work. Operations are idempotent (IF NOT EXISTS), so this is safe, but the locking pattern is misleading.
