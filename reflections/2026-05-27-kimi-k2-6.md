## On the project

cert-watch is a surprisingly complete single-binary TLS tracker. The codebase is well-structured: clear separation between `scan`, `upload`, `alerts`, `database`, `scheduler`, and `auth`. FastAPI + Jinja2 makes the web layer straightforward. The SQLite/WAL single-writer model is simple and correct for the intended single-replica deployment.

What feels fragile:
- The OAuth flow in `auth.py` was doing manual base64 decoding of ID tokens. It now validates `iss` and `aud`, but it's still not doing full JWT signature verification (authlib can do this; the code takes a shortcut).
- Auth middleware exempts ALL `/api/*` paths. This means read-only API endpoints are public, which may be intentional for monitoring, but it means any future mutating API endpoint must remember to add its own auth check (like I had to do for PATCH notes).
- The in-memory rate limiter and CSRF token store won't survive a process restart. Fine for a single pod, but worth noting.

## On the work done

The fixes were mechanical and well-scoped by the breadcrumbs. The ones I'm most confident in:
- **BC-019** (hosts UNIQUE migration): the `PRAGMA index_list` + dedupe + `CREATE UNIQUE INDEX IF NOT EXISTS` pattern is safe and idempotent.
- **BC-013** (Secure cookie flag): added `CERT_WATCH_COOKIE_SECURE` env var, defaulted to `True`, applied to all `set_cookie()` calls.
- **BC-011** (secrets in deploy): added `secretKeyRef` placeholders to k8s, commented examples to compose + systemd.
- **BC-018** (scan_history in add_host/import_hosts): straightforward `record_scan_history()` calls in the success/failure branches.

The ones I'd want a second pair of eyes on:
- **BC-009** (OAuth state verification): I used `_sign_state` / `_verify_state` with the same `_signing_key` as session cookies. This works, but it conflates session signing with OAuth state signing. A separate key would be cleaner.
- **BC-015/014** (ID token iss/aud): I compare `iss` exactly, but some OIDC providers include a trailing slash or version segment. The exact string match could be too strict in production.
- **BC-012** (PATCH notes auth): I gated auth/CSRF only when `AUTH_PROVIDER` is set. This matches the existing convention but means the endpoint is still unauthenticated in default mode.

## Gaps to flag

- `auth.py:complete_oauth_flow()` still doesn't verify the JWT signature. It only checks `iss` and `aud`. Full verification should use `authlib.jose` or similar. Location: `src/cert_watch/auth.py:355-370`.
- `_is_public_path()` in `app.py` exempts the entire `/api/` prefix. Any new mutating API endpoint must add its own auth guard. Location: `src/cert_watch/app.py:309-316`.
- `BC-010` was already fixed in templates (no `?_csrf_token` in form actions), but `_check_csrf()` still falls back to reading the query param. The fallback should probably be removed to prevent accidental leakage. Location: `src/cert_watch/app.py:175`.
- No test covers the `CERT_WATCH_COOKIE_SECURE=False` path. Could be tested by asserting `secure=0` in a `Set-Cookie` header.
- The k8s deployment manifest references `cert-watch-secrets` Secret, but the repo doesn't include a Secret manifest. Users must create it manually. Worth adding a `deploy/k8s/secret-example.yaml`.
