**Work summary:** Resolved BC-048 (fleet pivot views loading full inventory into memory) by rewriting `list_fleet_pivot()` with SQL-level GROUP BY aggregation and lazy-loading group entries via a new `/api/pivot/{pivot}/{key}` endpoint. ## On the project

cert-watch is a well-architected certificate monitoring tool with a clear spec-driven development process. The breadcrumb system is effective at tracking work, and the migration runner (Plan 009) is minimal but functional. The codebase has matured significantly from its monolithic beginnings — the `database/` package decomposition, route splitting, and middleware extraction are all solid. The test suite (402 tests) provides good coverage of the core paths.

The project's convention of keeping SQLite as the sole backend is pragmatic for a tool of this scope, but BC-031 (multi-database support) would be a significant architectural investment. The current repository pattern provides a clean abstraction boundary, but the concrete implementations are deeply SQLite-specific (raw SQL with `?` placeholders, manual datetime serialization, WAL mode, etc.).

## On the work done

**BC-048 (Pivot lazy loading):** The SQL-level aggregation for pivot summaries is straightforward and effective. The `julianday` trick for computing days remaining in SQL avoids Python post-processing. The lazy-loading pattern (AJAX fetch on expand) is clean and matches the existing expand/collapse UX. The template JavaScript for rendering entries is a bit verbose but functional.

**BC-049 (SQLite rate limiting):** The migration from in-memory to SQLite-backed rate limiting required careful handling of the connection cache. The `_connect` cache with WAL mode creates subtle interaction with backup/restore operations — the test fix for `test_backup_restore_round_trip` (deleting WAL/SHM artifacts) exposed a latent issue that would have caused corruption in production backup scenarios. The graceful fallback to in-memory mode on DB errors is important for resilience.

**Confidence level:** High for BC-048 (all 7 fleet lens tests pass, behavior preserved). Medium-high for BC-049 (rate limit tests pass, but cross-worker sharing hasn't been tested in a real multi-process deployment).

## On what remains

- **BC-031 (PostgreSQL/MSSQL support):** Deferred. Requires SQLAlchemy Core integration, repository abstraction, dialect-specific SQL, and expanded test matrix. Estimated 2-3 sessions.
- **WAL file cleanup in backup/restore:** The test fix for WAL/SHM artifact cleanup should probably be applied to the production backup code as well (`runner._backup`), since the same issue could affect production backup/restore operations.
- **Connection cache invalidation:** The `_connect` cache doesn't detect when a database file is replaced (e.g., during backup restore). This could cause issues in production if the database file is rotated externally.

## Gaps to flag

- **`middleware.py:120`** — The SQLite rate limiting path imports `_connect` and `init_schema` inside the function, which adds latency to the first rate-limited request after startup. Consider eager initialization.
- **`queries.py:760`** — `get_pivot_group_entries()` still calls `list_unified_entries()` which loads the full inventory. This is acceptable for lazy loading (only called when a group is expanded), but at fleet scale, it could be optimized with SQL-level filtering.
- **`runner.py:73`** — The `_backup` function uses `VACUUM INTO` which is SQLite 3.27+. The minimum supported Python version (3.12) ships with SQLite 3.39+, so this is fine, but worth noting for portability.
- **`dashboard.html` pivot JavaScript** — The inline JavaScript for rendering pivot entries duplicates urgency computation logic that exists in the template filters. If the urgency thresholds change, both places need updating.
- **No integration test for cross-worker rate limiting** — The SQLite-backed rate limiting works correctly in single-process tests, but hasn't been verified with multiple concurrent workers writing to the same database.
