## On the work done

**What went well:** The WI-126 subagent delegation was clean — the mechanical work of wrapping ~30 routes in `get_write_lock()` was well-scoped, the subagent followed instructions precisely (left `record_audit` and `bump_session_version` outside the lock), and the result passed the full suite on first run. The regression tests for WI-014 caught a subtle bug in my own test (used `PolicySet()` instead of `default_policy_set()`) that would have produced a false positive — the control test failed, forcing me to fix the test before it could pass. This is exactly why control tests matter.

**What I'm less confident in:** The `import_hosts` route wraps each row's `host_repo.add()` in its own `get_write_lock()` block rather than wrapping the entire loop. This is non-atomic (a failure mid-loop leaves partial state), but the alternative — holding the lock for the entire CSV import — could block writes for a long time. I chose to leave it as-is rather than refactor to a single-lock approach, because the existing per-row pattern is consistent with how the route already worked (each row is independently validated and inserted). The adversarial reviewers flagged this as a WARNING, not CRITICAL.

**The adversarial review finding I'm most glad about:** The `flush_alert_queue` lock-during-network-IO bug was introduced by the WI-126 subagent and caught by the adversarial reviewers. The subagent wrapped `process_pending()` in `get_write_lock()` because the instruction said "wrap DB mutations" — but `process_pending()` does both DB writes AND SMTP/webhook sends with retries. The scheduler calls the same function without the lock. This would have caused a global write-lock DoS under any alert flush with a slow SMTP target. The fix (remove the wrapping) is simple, but the pattern gap is worth noting: the codebase documents this anti-pattern in `scan.py` code comments but not in AGENTS.md conventions.

## Gaps to flag

- **`flush_alert_queue` anti-pattern was introduced and caught** — but the convention "lock wraps DB mutations only, never network I/O" is documented only in `scan.py` code comments, not in AGENTS.md. A one-line addition to the conventions section would prevent recurrence.
- **`password.py` coverage floor at 89%** — the uncovered paths are the `enc_key=None` branches (no encryption key available). These should have tests, not a lowered floor. `src/cert_watch/routes/settings/password.py:86-91`.
- **`_validate_email` had zero tests** — it's a security-critical SMTP header injection guard. How many other security-critical helpers have no tests? Worth a targeted audit of `alerts.py` and `middleware.py` private functions.
- **`import_hosts` is non-atomic** — per-row locking means partial imports. `src/cert_watch/routes/hosts.py:303-313`. Acceptable for a bulk import but should be documented as a conscious design choice.
- **No e2e test for the flush_alert_queue route** — the lock-DoS bug would have been caught by an e2e test that actually triggers a flush with a mock SMTP server. The unit tests mock at the function level, which is why the lock wrapping wasn't visible.
