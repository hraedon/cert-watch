Fixed 3 actionable findings: webhook URL SSRF via DNS resolution bypass (the most significant), missing webhook digest error logging, and inconsistent owner_email validation. ## On the project

The codebase remains well-structured. The audit surfaced mostly design concerns rather than real bugs — the codebase has good defensive patterns (parameterized queries, SSRF guards on scans, CSRF double-submit, HMAC-signed sessions). The one real security gap was the webhook URL validation: it checked if the hostname was a private IP *literal* but never resolved DNS, so `http://internal-service.local/` would pass validation even if it resolved to `10.0.0.1`. This is the kind of bug that only matters if an attacker can influence the webhook URL configuration, which requires admin access — but defense-in-depth says fix it.

## On the work done

The subagent audit was thorough and well-structured. It correctly identified the webhook SSRF as the highest-severity finding and correctly dismissed several false positives (e.g., the `sort_by` SQL injection concern — the ternary guard makes it safe). The three fixes I applied are all small, surgical changes that follow existing patterns in the codebase.

The webhook SSRF fix reuses `_ALWAYS_BLOCKED_NETWORKS` and `_PRIVATE_NETWORKS` from `scan.py` rather than duplicating the network lists. Webhook URLs are always blocked from pointing to private IPs (no `allow_private` override), which is stricter than scan targets — this is intentional since webhook URLs should always be public endpoints.

## Gaps to flag

- `_validate_webhook_url` in `routes/api.py:83-114` now resolves DNS, but doesn't cache results or pin the IP for the actual webhook request. A TOCTOU race is possible (resolve passes, DNS changes, request goes to private IP). Low risk since webhook URLs are admin-configured and rarely change.
- `store_scanned` in `scan.py:740-801` has 4 nested bare `except Exception` blocks that log at DEBUG. Posture, drift, and history data can silently go missing with no operator-visible signal. Consider logging at WARNING for the outer block.
- `send_expiry_digest` webhook path in `alerts.py:535` now logs at WARNING, but the SMTP path (line 513) sanitizes the error via `_sanitize_smtp_error` while the webhook path logs the raw exception. Minor inconsistency.
