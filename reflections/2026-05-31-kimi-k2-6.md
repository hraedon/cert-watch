## On the project

The codebase is mature and well-tested (402 tests). The decomposition into `database/` package (`schema.py`, `queries.py`, `repo.py`, `connection.py`) makes data-layer changes tractable. The template layer is also well-organized with shared macros (`macros/icons.html`) and CSS tokens.

The biggest architectural tension remains the dashboard queries. `list_unified_entries()` is essentially a full outer join performed in Python because SQLite doesn't have `FULL OUTER JOIN`, and because the unified view merges three sources (hosts, scanned certs, uploaded certs) with computed fields (`urgency`, `chain_status`). Any true SQL-level pagination would need to either (a) materialize the unified view as a SQLite view/CTE, or (b) accept approximate counts for the stat cards. Neither is trivial.

## On the work done

BC-047 was more complex than it appeared. The dashboard has three modes:
1. **Pivot views** (`?view=issuer`) — loads everything, groups in Python. Not optimised.
2. **Grouped mode** (`grouped=1`, default) — needs full set for cross-host search within fingerprint groups (e.g. searching "beta" must find a group containing both "alpha" and "beta"). Not optimised.
3. **Fast path** (`grouped=0`) — only materializes 25 rows. This is the actual win.

I initially tried to move filtering/sorting into `list_unified_entries_page()` and have `grouped=1` use it too, but that broke `test_dashboard_grouped_search_matches_host` because cross-host search within a fingerprint group requires the full filtered set before grouping. I ended up with a three-branch `dashboard()` that falls back to `list_unified_entries()` for grouped/pivot modes and uses `list_unified_entries_page()` only for the fast path. This is pragmatic but leaves the default path (grouped=1) still loading everything.

BC-050 and BC-051 were straightforward and I was confident in both.

## On what remains

**Open breadcrumbs (3):**
- **BC-031** (medium) — PostgreSQL/MSSQL support. Deferred until scale ceiling.
- **BC-048** (medium) — Fleet pivot views still load full inventory. The `list_fleet_pivot()` function calls `list_unified_entries()` unconditionally. This needs dedicated SQL for each pivot (issuer/owner/renewal_method) to aggregate at the DB level.
- **BC-049** (medium) — In-memory rate limiting not shared across workers. `middleware.py` uses a process-local dict. For multi-worker deployments, this is a real gap.

**Recommended next session:** BC-048 is the highest user-visible remaining value — SQL-level aggregation for pivot views would eliminate the last full-inventory load on the dashboard.

## Gaps to flag

- **`dashboard()` stat cards are approximate in fast path** (`routes/views.py:160-170`). When `grouped=0`, `all_entries` is the page slice (25 rows), so the Expired/Critical/Warning/Healthy counts shown in the stat cards are only for the current page. This is a UX regression for the fast path. The template computes stats from `_stat_entries`, which is `all_entries` when defined. For the fast path, `all_entries` should probably be loaded from a lightweight SQL `COUNT(*)` per urgency bucket, or the stat cards should be removed from the fast path.
- **`list_unified_entries_page()` still loads full dataset** (`database/queries.py:445-500`). It loads all certs/hosts/scans into Python before filtering/sorting/slicing. The "page" in the name is accurate about the return shape but misleading about the memory cost. The real next step for BC-047 is SQL-level filtering (e.g. `WHERE hostname LIKE ?`, `WHERE not_after <= date('now', '+14 days')`) and only fetching the page slice.
- **`metrics` endpoint still loads full dashboard rows** (`routes/views.py:301-326`). `list_dashboard_rows()` is called unconditionally. For large fleets this is as bad as the old `healthz`.
- **`_list_unified_entries_raw()` has no test coverage for the new path** — the regression test `test_list_unified_entries_page_pagination` tests the paginated wrapper, not the raw merge logic. If `_list_unified_entries_raw()` is ever refactored for SQL-level joins, the existing `test_list_unified_entries_scanned_pending_uploaded` should still cover it.
