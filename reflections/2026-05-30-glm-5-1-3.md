## On the project

The codebase is well-organized after the BC-030/BC-035 decompositions. The `database/` package split (schema/repo/queries/connection) makes it easy to add new columns, migrations, and repository methods without touching unrelated code. The template layer is the heaviest part — audit.html follows the same conventions as the other view templates (extending base.html, using existing CSS classes).

The migration system design is deliberately minimal — a `schema_version` table, numbered Python modules, and a registry that runs on import. This contrasts with Alembic's heavier approach, which the plan correctly rejected for the single-SQLite-file shape. The `ensure_base()` function still does PRAGMA-based column migrations for backward compatibility, which is the right call for existing deployments but means the column additions are in two places (the DDL in `_BASE_TABLES` for fresh DBs, and the ALTER TABLE guards in `ensure_base()` for legacy DBs). Over time, those PRAGMA guards should be folded into numbered migrations.

## On the work done

The audit log implementation went smoothly. Instrumenting the routes was mechanical but required reading each mutating route handler to find the right insertion point. The `resolve_actor()` / `resolve_source_ip()` helpers are simple and correct. The integration tests initially failed because the `TestClient` lifespan calls `Settings.from_env()` which overrides `app.state.settings` — a known gotcha that the conftest handles via `monkeypatch.setenv("CERT_WATCH_DATA_DIR", ...)`.

The migration runner implementation was clean. The main decision was whether to have `ensure_base()` handle the PRAGMA-based column additions or leave them entirely to migrations. I kept them in `ensure_base()` for backward compatibility with existing databases that predate the migration system, which is the safe choice. The `_stamp_baseline()` function handles the upgrade path for existing databases correctly — it detects an empty `schema_version` table and stamps 0001 so existing DBs don't re-apply schema that already exists.

The `cert-watch backup` CLI subcommand was straightforward — `VACUUM INTO` is WAL-safe and works while the app runs. The argparse subcommand pattern replaces the old bare `uvicorn.run()` default, which now runs when no subcommand is given.

I'm least confident about the `_initialized_paths` module-level set in `schema.py`. It uses `Path.resolve()` for dedup, which means symlinks or relative paths could cause the same DB to be initialized twice if accessed via different path strings. This hasn't caused problems in practice but is a latent test-isolation risk.

## Gaps to flag

- `schema.py:_initialized_paths` is a module-level set using `Path.resolve()` for dedup. If the same database is opened via different path strings (e.g. relative vs absolute vs symlink), it could be initialized more than once. Not a data corruption risk since everything is idempotent, but wastes time and creates duplicate connections.
- `audit_log` table creation is now in two places: `_BASE_TABLES` DDL (for fresh databases) and migration `m0002_audit_log.py` (for the migration runner). Both use `CREATE TABLE IF NOT EXISTS` so it's safe, but it's a minor DRY violation. Future schema additions should go through the migration system only, not be added to `_BASE_TABLES`.
- The `ensure_base()` PRAGMA-based column migrations (lines 140-170 in schema.py) should eventually be folded into numbered migrations so `ensure_base()` becomes purely `CREATE TABLE IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS`. This is a gradual cleanup — don't do it on the next change, but plan to retire each PRAGMA guard as a migration.
- `audit.py:record_audit()` catches all exceptions and logs WARNING. The log message includes `exc_info=True` which produces a full traceback. In production, a failed audit write (e.g., disk full) could produce very verbose logs. Consider logging the traceback at DEBUG level instead.
- No test for the `cert-watch hash-password` CLI helper mentioned in Plan 010 — that subcommand doesn't exist yet but is part of the break-glass admin design.
