## On the work done

**What went well:** Parallel dispatch of 5 implementers with exclusive file ownership meant zero merge conflicts. The adversarial review phase caught genuine bugs:

1. **stop_scheduler() double-release** (WI-022): The implementing agent added `_cycle_lock.release()` to `stop_scheduler()`, but the scheduler thread's `finally` block also releases — causing `RuntimeError: release unlocked lock` inside a `finally` block. The test didn't catch it because it never ran the scheduler thread mid-cycle. Fixed by removing the release from `stop_scheduler()` and increasing join timeout.

2. **NULL sentinel collision** (WI-018): `_batch_chain_statuses()` used `None` as the sentinel for "no data yet", but `chain_status` is a nullable column — `None` from the DB is indistinguishable from the sentinel. This caused stale `chain_status` values to be returned when the latest scan had NULL. Fixed with `_UNSET = object()`.

3. **Missed send_expiry_digest OOM** (WI-021): The implementing agent fixed `evaluate_all_certs()` and `evaluate_renewal_window()` but missed `send_expiry_digest()` which had the same `SELECT c.*` pattern loading all `raw_der` blobs. Fixed with explicit column list.

**What I'm less confident in:** The `_UNSET` sentinel pattern in readiness.py is correct but unusual for this codebase. If someone later adds a call path that doesn't go through `_batch_chain_statuses`, they'll reintroduce the N+1. The dead `_get_chain_status()` was removed, which helps.

## Gaps to flag

- **No test for middleware ordering (WI-020).** The adversarial reviewer flagged this. `test_rate_limit_headers_on_api` only checks 200 responses. A test asserting `X-RateLimit-Remaining` appears on 401 responses would lock in the new ordering. `tests/test_extra_coverage.py:303-308`.
- **No test for data-action="print" (WI-019).** The print handler in base.html is untested. A static HTML assertion (`'data-action="print"'` in rendered page) would prevent silent regression. `tests/test_readiness.py`, `tests/test_compliance.py`.
- **Batch query fetches ALL scan_posture rows.** `_batch_chain_statuses()` in `readiness.py:68-74` fetches every scan_posture row for every host, then takes the first per hostname. For large fleets this is O(N*M). A window function or correlated subquery would be O(N log M). Not urgent for typical deployments but worth noting. `readiness.py:68-74`.
- **`raw_der=b""` is a silent sentinel.** If Certificate gains a method that parses raw_der, the alert path will silently break. Consider a property that raises when `raw_der` is empty and accessed for parsing. `alerts.py:234`.
- **E2E gap for /readiness page.** Flagged by both the GLM reflection and the adversarial reviewer. `/readiness` has no E2E test. Needed before v0.11.0 ships per the GLM reflection. `tests/e2e/`.
