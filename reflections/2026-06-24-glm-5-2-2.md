## On the project

This is a well-tended codebase in declared maintenance mode (Plan 049), and it shows: a 2253-test unit suite, ruff+mypy clean, strong conventions documented in AGENTS.md, and a disciplined breadcrumb/WI store. The maintenance-mode framing genuinely shaped my decisions — I deliberately did **not** touch schema (dropping `ct_issuer_first_seen` is WI-082, not my call here), auth/session posture, public API shapes, or the bigger refactors (collapsing the `config.py` shim, the `database/queries.py` re-export layer — WI-087), because those need a priced plan or a human sign-off per AGENTS.md's "decisions that must be surfaced." The review request was well-scoped for maintenance mode: defect fixes and dead-code removal are explicitly in-bounds.

The sharpest structural smells the three agents converged on are all already tracked: the `store_scanned` multi-stage pipeline with no transaction boundary and per-stage `contextlib.suppress(Exception)` silently dropping posture/history/drift rows (WI-084), the `SecurityContext`-vs-module-global two-sources-of-truth (WI-083), and the alert-group resolution triplicated across batch/per-cert/route-local (WI-085). The codebase knows about its own debts. What was less tracked was the *amplifier* I fixed: those silent stage-drops were being recorded as `status="success"` in `scan_history`, which then told the scheduler "stop retrying this host." The bug and the debt were compounding.

## On the work done

I used the implementer → adversarial-review loop the user asked for (kimi implemented, glm reviewed), and it earned its keep: the reviewer caught that my failure-recording `record_scan_history` sat *inside* the store-fn `except` and could itself raise the same `sqlite3.Error` and crash the scan loop — directly contradicting the noqa's "must not crash scan loop." I'd missed that. Fixed by guarding that one call (`sqlite3.Error` → warning). The reviewer also flagged a stale test docstring and a too-weak test assertion (`fetchone` instead of `fetchall` + an unasserted complementary contract). I applied all four. This is the argument for the two-agent loop: the implementer was fast and correct-on-the-happy-path, but the second pass found the edge the first normalized away.

I'm confident in: the `run_scan_now` fix (control flow walked, two regression tests, would-fail-on-pre-fix verified by the reviewer); the exception narrowings (each narrowed set covers the legit failure modes — `sqlite3.Error` is the parent of OperationalError/DatabaseError, migration 0019 creates users/roles unconditionally so the role-routing catches won't fire every cycle). I'm **less** confident in one judgment call: raising debug→warning on the role-routing catches. The reasoning is sound (the catches only fire on ImportError/sqlite3.Error, neither of which happens in steady state), but if I'm wrong about some deployment where those fire routinely, it's now warning-level log spam every alert cycle. The full suite doesn't exercise that because no test runs the alert cycle against a missing-users-table state. If it bites, it'll bite in production logs, not CI.

I deliberately left several "simple" findings on the table rather than fixing them, and I think that was right: deleting `resolve_all_group_recipients`/`resolve_group_thresholds` looked like dead code until grep showed 3 tests + spec + AGENTS.md reference them — they're documented public API, not dead. And I did **not** narrow the `routes/certificates.py` posture-eval page-render catch or the `scan.py ssl_sock.version()` suppress: those are user/scan-path catches where the current "always render / always degrade" behavior is defensible, and narrowing them changes runtime behavior on a detail page and in posture grading — too high a blast radius for a maintenance-mode cleanup pass without an operator-visible failure to point at.

## On what remains

The 7 new WIs (WI-090..096) are the remainder, sequenced by what I'd do first:
- **WI-090 (admin-provisioning atomicity)** and **WI-092 (scheduler-vs-request write-lock)** are the two real defects with concrete failure scenarios (admin lockout; silent stage drops under contention). WI-090 is a one-line `kv_set_multi` swap but is auth-provisioning-adjacent — I flagged it rather than auto-fixing per the surface-to-human rule. WI-092 is the higher-leverage one: making `store_scanned` acquire `get_write_lock` internally closes the hazard for all callers and reduces the silent-drop window that WI-084's suppress stages create.
- **WI-091 (schema-cache stale on restore)** is real but narrow (only bites on a restore-from-backup with older schema); fix is to key the cache on `(st_ino, st_size, mtime)` like the connection cache already does.
- WI-093..096 are low-severity duplication refactors (SMTP open, scan/store pipeline, kv magic strings, webhook kind tuple) — fine to leave for a batched cleanup.

The deeper WI-084 (give `store_scanned` a real transaction boundary) is still the highest-value structural fix in the project, and my WI-092 is the concurrency complement to it — together they'd eliminate the "scan succeeds, half the artifacts silently vanish" class.
