## On the project

The cert-watch codebase is mature and well-organized. The AGENTS.md discipline is strong — it tells me not to hand-maintain backlog lists, and I listened. The `agent-notes` CLI is the source of truth; the markdown files under `breadcrumbs/` are just a mirror. This is a healthy state for a project at v0.6.5.

The scan/auth/middleware stack has been through multiple consolidation plans (018–025) and the seams are clean. The one place that still feels procedural is `config.py` — 700-line `Settings.from_env()` — but that was explicitly deferred (BC-144a, wontfix for v0.6.5), so it is a known, tracked gap rather than a surprise.

## On the work done

**BC-144 split:** Straightforward. The original BC-144 bundled three very different items (config decomposition, spec completeness, and test reload cleanup). Splitting them was the right call — the backlog is now honest about what is deferred vs. what is actionable. The agent-notes sync/export loop is clean.

**BC-095 (SSRF deduplication):** The original `_is_blocked_host_check()` in `routes/hosts.py` was a near-copy of the logic in `scan.py`. I centralized it into `scan.py` as `resolve_and_validate_host()` and refactored `_resolve_host()` to delegate to it. This eliminated ~70 lines of duplicated code. The tests in `test_ssrf_csrf_ratelimit.py` and `test_route_coverage_api.py` were updated to monkeypatch the new shared location. I am confident this is correct — the tests pass and the error messages are identical.

**BC-143 (template linting):** `djlint` was added to the `dev` extras. The CI step uses a curated ignore list (`T003,J004,J018,T028,H021,H023,H030,H031`) because the project intentionally uses bare `{% endblock %}`, hardcoded internal URLs, inline styles, and entity references — these are not bugs, they are conventions. The remaining rules catch real syntax errors (unclosed tags, malformed template syntax). The `pyproject.toml` config section was attempted but `djlint` did not pick it up for some rules; the CLI `--ignore` flag works reliably, so CI uses that. The `uv.lock` was regenerated.

## Gaps to flag

- **`djlint` pyproject.toml config not honored:** I added `[tool.djlint]` to `pyproject.toml` but the linter still emitted the ignored rules. Removed the section to avoid confusion; CI uses explicit `--ignore` flags. If the project wants per-file ignores or other config, a `.djlintrc` or `.djlint.toml` file would be needed. Not a gap worth tracking separately.
- **`OPEN_BREADCRUMBS.txt` regeneration:** The export-index command is fast and reliable, but it is easy to forget. The `agent-notes breadcrumb sync` after editing active/resolved files is also easy to miss. The next agent should verify the file matches the DB if breadcrumbs were edited.
- **No new tests were added** for the deduplication itself. The existing SSRF tests (`test_ssrf_csrf_ratelimit.py`) exercise the same paths through the new function, so coverage is preserved, but there is no dedicated test for `resolve_and_validate_host()` in `scan.py`. The test that was monkeypatching `_is_blocked_host_check` now patches `resolve_and_validate_host`, so the behavioral coverage is intact.
