## On the project

cert-watch is in maintenance mode and the codebase shows it — the surface is stable, the test suite is deep (2059 unit tests), and the conventions are well-documented in AGENTS.md. The project is genuinely well-engineered: SSRF pinning, CSP nonces, session revocation, tamper-evident compliance reports. The migration system is clean and idempotent. The main complexity is in the alert routing pipeline (groups, thresholds, recipients, digests) which is where this session's work lived.

The maintenance-mode contract (no new features without a priced plan) creates an interesting tension — WI-056 is a feature, not a bug fix, so it technically violates the contract. But it was a tracked work item, so the human clearly wanted it done.

## On the work done

The implementation went smoothly thanks to the subagent (GLM). The core changes are clean: migration adds two nullable columns, the AlertGroup dataclass gets two new fields, and the alert evaluation pipeline checks group thresholds before falling back to per-host/global defaults. The digest cadence wiring through `app.py` was the trickiest part — `_max_group_cadence()` as a closure is a pragmatic choice but the adversarial reviewer correctly flagged it as untestable.

The adversarial review was the most valuable part of the session. BUG-1 (digest_cadence_days collapses to global max, not truly per-team) is a real design limitation — the naming suggests per-team behavior but the implementation is a global window. This is documented now but worth flagging. BUG-4 (duplicate DB queries) was a real performance issue that I fixed by consolidating into `_resolve_group_config`. BUG-6 (theater tests) was embarrassing — the subagent wrote tests that asserted `isinstance(result, list)` which proves nothing.

The general project review surfaced a pre-existing bug: `views.py:227` uses `datetime('now')` string comparison which fails for T-separated ISO timestamps. That's a real correctness bug in the dashboard's expired count, not related to WI-056, and I didn't fix it (out of scope).

## On what remains

- **WI-056 design limitation**: `digest_cadence_days` is a global window (max across groups), not truly per-team cadence. True per-team dispatch would require iterating groups and sending separate digests — a bigger architectural change. The current design is a reasonable v1.
- **WI-054**: Remove compliance report module — non-trivial, still wired into routes/views/templates/CLI.
- **WI-055**: Remove SIEM export module — similarly wired into audit.py, routes, __main__.
- **Dashboard expired count bug** (`views.py:227`, `fleet.py:52`): `not_after < datetime('now')` fails for T-separated ISO strings. Needs a regression test.
- **`init_schema` cache race** (`schema.py:320-326`): cache is added before work completes — two threads can interleave.
- **Per-request `local_admin_autogenerated` read** (`middleware.py:383`): immutable flag read on every authenticated page.

## Gaps to flag

- **`digest_cadence_days` naming creates false expectations**: The field suggests "how often digests are sent" but actually controls the data window. Send frequency is hardcoded weekly. Operators setting `digest_cadence_days=1` expecting daily digests will be disappointed. (`digest.py:69`, `app.py:338`)
- **Group threshold silently overrides per-host threshold**: No logging, no documented priority. A host admin's carefully tuned `threshold_days=30` is silently replaced by a group with `threshold_days=3`. (`alerts.py:265-270`)
- **`resolve_group_thresholds` was consolidated but the `_UNSET` sentinel leaks**: `SqliteAlertGroupRepository._UNSET` is a class attribute accessed from the API route module. (`repo.py:800`, `routes/api/alerts.py:169`)
- **Pre-existing dashboard bug**: `views.py:227` and `fleet.py:52` use `not_after < datetime('now')` which is a string comparison that fails for T-separated ISO timestamps. Tests don't catch this because they insert space-separated format.
- **AGENTS.md CSP nonce claim is stale**: Says "nonce CSP active" but `middleware.py:678` says the header still uses `'unsafe-inline'` for `script-src`.
