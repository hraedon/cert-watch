## On the work done

BC-098 was a no-op investigation — the wave-2 restructure already fixed it. Good outcome, but worth noting that the original bug report was vague ("reportedly loses its panel cards") with no repro steps. If the issue had persisted, finding it would have required running the app and visually inspecting, since the Jinja template looks correct on paper.

BC-101 was a real fix. The root cause was a data shape mismatch: the SQL queries return daily `{date, protocol_version, count}` rows but the template expected monthly pivoted `{month, tls_1_3, tls_1_2, tls_1_0}` data. The route handler was passing raw query output without transformation. The fix adds two pivot functions and a dynamic max for bar scaling. The API endpoints still return raw daily data — backward compatible, which is the right call since the API is a public contract.

## Gaps to flag

- The trend bar height calculation (`total / tls_max * 100`) assumes `tls_max >= 1` via `max(max_total, 1)`, but if all months have zero counts the chart section still renders (the `{% if total > 0 %}` guard protects individual bars, not the container). Cosmetic only — `routes/views.py:461`.
- The pivot functions use `OrderedDict` for insertion-order grouping. Fine for Python 3.7+ (guaranteed dict ordering), but the import is technically unnecessary — a plain `dict` would work. Not worth changing.
