## On the project

The codebase is well-structured with clear separation of concerns. The AGENTS.md is genuinely excellent — the most useful project orientation doc I've seen. The convention of "lock wraps DB mutations only, never network I/O" is load-bearing and correctly applied in most places. The test suite is comprehensive (2435 tests) but has some fragility around test pollution (ldap3 module caching, parallel test ordering). The dual security model (tests use _CSRF_BYPASS, production doesn't) is a pragmatic choice that works.

## On the work done

**What went well:** The P0 fixes were mostly straightforward — the SSRF bypass, compliance port bug, and IDOR fixes were one-line or few-line changes. The write lock change (Lock→RLock) was clean and necessary for the nested lock acquisition pattern. The adversarial review process caught real bugs that I would have shipped.

**What I'm confident in:** The SSRF fix (IPv4-compatible/NAT64), compliance port, write locks, CSV import semaphore, audit re-raise, WAL mode, OAuth nonce/exp. These are well-tested and the logic is clear.

**What I'd want a second pair of eyes on:**
- The encryption HKDF change — the auto-migration at startup is correct in theory but has not been tested against a real database with v1 values. The migration runs `re_encrypt_kv_store` which tries the legacy key; if the signing key hasn't changed, it should work. But this is the kind of change that should be validated on a backup before deploying.
- The API key hashing change — the dual-lookup SQL (`WHERE key_hash = ? OR key_hash = ?`) is correct but the pepper defaults to a known string when CERT_WATCH_AUTH_SECRET is unset. In production this should always be set, but there's no guard.
- The posture-eval-skip — when _posture_eval is None, no posture data is stored. This means the cert detail page evaluates posture live. This is correct but means the compliance report won't have posture data for these certs (it has its own fallback, but the grade distribution may differ).

**What was awkward:** The ca_probe changes were the hardest — the DNS rebinding fix required restructuring the SSRF validation flow, and the test pollution from ldap3 module caching caused 8 tests to fail in parallel mode but pass in serial. The `_BLOCKED` sentinel pattern works but feels like it should be a proper enum or exception.

## Gaps to flag

- **Encryption migration untested against real data** (`app.py:264-278`): The auto-migration from v1→v2 at startup uses `re_encrypt_kv_store` with the legacy key. This has not been validated against a production database. If the migration fails, encrypted settings (LDAP bind passwords, SMTP creds) become empty strings silently.
- **Regista migration 38 pending**: Could not file breadcrumbs because the regista schema needs migration 38. The `agent-notes` CLI exits non-zero with `RegistaError: Migrations pending`. This needs to be resolved outside this session.
- **ldap3 test pollution** (`tests/conftest.py:23-27`): Added `import ldap3.core.exceptions` at conftest level to work around partial module caching in parallel test runs. This is a band-aid; the root cause is some test partially importing ldap3 without its submodules.
- **HANDOFF.md untracked**: There's an untracked `HANDOFF.md` file in the repo root. I didn't create it — should not be committed.
- **Posture data gap when eval fails** (`scan.py:948-968`): When pre-lock posture evaluation fails, the cert is stored without posture data. The compliance report's fallback evaluation passes `chain_status=None` which may produce different grades than the scan-time evaluation.
- **Event `rate_limited` status not surfaced** (`events.py:348-350`): Rate-limited events are persisted with `delivery_status="rate_limited"` but no UI or retry path surfaces them. They're invisible to operators.
