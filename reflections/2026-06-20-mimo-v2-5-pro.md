## On the project

This codebase is genuinely mature. The 5-agent deep scan found security posture is strong (parameterized SQL, SSRF pinning, CSRF double-submit, session revocation, CSP nonces, timing-attack mitigation), the test suite is real (not theater — mutation verification, per-module coverage floors, CSS class coverage ratchets), and the deployment story is complete (k8s, systemd, IIS, Docker with non-root + Trivy). The maintenance-mode posture (Plan 049) is appropriate — the remaining open WIs are cleanup and design decisions, not existential issues.

The architectural debt is concentrated in a few oversized modules: `middleware.py` (1010 lines, 4 concerns), `alerts.py` (1274 lines, mixed concerns), `dashboard.py` (1310 lines, almost no direct tests). These are the kind of files that grew organically and were never decomposed because they worked. Not urgent, but they're the first things I'd refactor if this project were to grow.

## On the work done

The scan_history purge fix is the most impactful — the table had no retention and `list_scan_batches` materializes all rows in Python. At 1000 hosts × daily scans, that's 365K rows/year with no cleanup. The fix reuses `history_retention_days` (365d) and follows the exact pattern of the existing purge functions. Simple, but it prevents a slow-motion OOM.

The template JS dedup (WI-080 partial) was satisfying — `escHtml` was defined 3 times across templates with slightly different null handling, and `confirm-form` was copy-pasted 3 times. Moving both to `base.html` as globals is clean. The `esc` → `escHtml` rename in the health banner IIFE required care since the function was scoped inside a closure.

The siem.py `assert` fix is the kind of thing that would only surface in production when someone runs with `python -O` — the asserts would be stripped, leaving `NoneType` unpacking errors. The middleware silent `except` fix is similar — DB errors were being swallowed without any trace.

**What I'm confident in:** All fixes are mechanical and follow existing patterns. The tests pass, ruff and mypy are clean.

**What I'd want second eyes on:** The `escHtml` global function in `base.html` — it's now in a separate `<script>` tag before the health banner IIFE. The CSP nonce is the same for both tags (it's per-request, not per-tag), so both scripts get the same nonce. This should be fine, but if any CSP strictness tool flags multiple script tags, that's the cause.

## Gaps to flag

- **`database/dashboard.py` (1310 lines) has almost no direct tests.** The SQL-heavy dashboard module is the largest database submodule and relies primarily on indirect coverage through routes. A query-level unit test suite with seeded data would catch SQL regressions faster. (`database/dashboard.py`)
- **`digest.py` duplicates SMTP sending logic from `alerts.py`.** Both build `EmailMessage` objects and call `_open_smtp_connection` independently. ~80 lines of duplication. (`digest.py:192-332`)
- **`readyz` exposes version/commit to unauthenticated users.** `/healthz` correctly omits this, but `/readyz` returns `{version, commit}` in its JSON body and is in `_PUBLIC_PATHS`. Consider redacting when auth is enabled. (`routes/views.py:122-127`)
- **`capChips` function in dashboard.html:698-722 is dead code.** It selects `.cw-cert-chips` then looks for `.cw-san-list` inside, but `.cw-san-list` is a sibling (not a descendant). The SAN chips are never capped by JS. Server-side Jinja already caps at 2 SANs, so this is invisible. (`templates/dashboard.html:698-722`)
