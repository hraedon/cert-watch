## On the project

The codebase is in very good shape. The test infrastructure (368→386 tests this session) is comprehensive and well-organized. The breadcrumb/plan system is excellent — every open issue has a clear description, severity, and suggested fix, which makes triaging and implementing straightforward.

The auth module is the most complex part of the codebase. The `_CompositeProvider` pattern (local admin + primary provider) is clean but the mock infrastructure for testing is heavy — every OAuth test needs to inject mock authlib modules into `sys.modules`. A dependency-injection approach would be cleaner but is a larger refactor.

The CSS architecture (`tokens.css` with `cw-` prefixed utility classes) has matured well. The distinction between static utility classes and dynamic CSS custom property bindings (`style="--tone:{{ value }}"`) is the right balance — you can't avoid the inline style for Jinja2→CSS value injection, but all static styles are now in classes.

## On the work done

BC-042 was already resolved by a prior session — 37 posture tests, scan wiring in `store_scanned()`. I verified and moved it to resolved.

BC-043 (OAuth JWKS) was the main deliverable. The implementation uses `joserfc.jwt.decode()` for signature verification + `authlib.oidc.core.CodeIDToken` for OIDC claim validation (iss/aud/exp/nonce/at_hash). The fallback chain is: JWKS verification → if fails, userinfo endpoint (which requires a valid access token, so it's still secure). 11 JWKS tests + 7 `_validate_claims_manual` tests added.

The `joserfc` API was tricky — `RSAKey(private_key, parameters={...})` doesn't work in v1.6.8; you need `RSAKey.import_key(private_key, parameters={...})`. This was discovered through test failures, not documentation.

BC-037 and BC-039 were small template cleanups. The scan_history duplicate `class` attribute was a real bug (the second `class` attribute was silently ignored by browsers, so `cw-empty-state` was never applied).

## Gaps to flag

- **`_verify_id_token` catches all exceptions** (`auth.py:494`) — this means a bug in the verification logic (e.g. wrong OIDC claim schema) would silently fail and fall back to userinfo. The logging at WARNING level helps, but consider adding a counter metric for JWKS verification failures to detect regressions.
- **JWKS cache never expires** (`auth.py:426-437`) — `self._jwks` is set once and reused for the provider's lifetime. If the IdP rotates signing keys, the provider won't pick up the new JWKS until restart. For production Entra usage, consider adding a TTL-based cache invalidation (e.g. refresh every 24h) or re-fetch on `InvalidKeyIdError`.
- **`_discover()` caching** (`auth.py:373`) — `self._discovered` is cached for the provider's lifetime. If OIDC discovery fails on startup, the provider will use the partial/explicit config forever. This is acceptable for now but could surprise operators debugging discovery failures.
- **The `CryptographDeprecationWarning` in cert_chain.py:118** — serial number parsing for negative/zero values. This is in a PEM parsing fallback path and will break when `cryptography` removes the deprecated behavior.
- **No E2E test for OAuth JWKS flow** — the unit tests mock authlib and use self-signed JWTs. An E2E test with a real OIDC provider (even a mock one like `oidc-provider`) would catch integration issues.
