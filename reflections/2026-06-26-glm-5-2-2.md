## On the project

The maintenance-mode discipline continues to serve this project well. The test suite (2276 tests) is comprehensive enough that the transaction-boundary refactoring — splitting `_evaluate_and_store_posture` into pre-transaction evaluation and in-transaction storage — caught every regression the change could have introduced. The `_stage` wrapper pattern in `store_scanned` makes it easy to see which stages are inside the transaction and which are outside, and the `_PostureEval` dataclass cleanly separates "what the network told us" from "what we write to the DB."

The WI-086 middleware refactoring surfaced two pre-existing security concerns (WI-116, WI-117) that the refactoring correctly preserved rather than fixed. This is the right call — changing auth behavior in a maintenance-mode project requires human sign-off. But it's notable that the deduplication made these latent issues more visible, which is itself a benefit of the refactoring.

## On the work done

**What went well:**

The WI-114/WI-113 implementation was the most complex change. Splitting posture evaluation from storage required:
1. Creating `_PostureEval` dataclass to carry evaluation results
2. Creating `_evaluate_posture` function (pre-transaction, network I/O)
3. Modifying `_stage_posture` to accept pre-computed results (in-transaction, DB writes only)
4. Keeping `_evaluate_and_store_posture` as a backward-compat wrapper (used by tests and any future direct callers)
5. Moving `_evaluate_posture` inside the try/except block so failures return `""` like other stage failures
6. Adding `_defer_webhook=True` to `emit_event` and collecting deferred webhooks in `_stage_events`
7. Firing deferred webhooks after `conn.commit()` succeeds

The test update was mechanical but large: 15 mock sites in `test_scan.py` needed to change from mocking `_evaluate_and_store_posture` (returns 3-tuple) to `_evaluate_posture` (returns `_PostureEval`). The `replaceAll` edit made this manageable.

**What I'm less confident in:**

The `_scan_and_store` helper in `routes/hosts.py` (WI-094, implemented by the kimi subagent) had a BLOCKER that the adversarial reviewer caught: it didn't check `store_scanned`'s return value, so a transaction rollback (empty string) was treated as success. I fixed this, but the fix required also updating the test mocks (`flaky_store` needed to return `"leaf-id"` on success, not `None`). The subagent's tests passed because the old code didn't check the return value — a classic case of tests passing against incorrect behavior.

The WI-101 webhook-kind dedup (glm subagent) was clean — the `WEBHOOK_KIND_OPTIONS` derived from the adapter registry is a good single-source-of-truth pattern. The JS fix (preset select setting `kindEl.value` correctly) was a bonus catch.

## On what remains

- **WI-099** (CSRF token minting for 267+ test call sites) remains open. The `_CSRF_BYPASS` flag is documented as safe but is still a bypass mechanism in production middleware. The `csrf_strict` fixture infrastructure is in place; the work is mechanical but large.
- **WI-116** (legacy admin fallback allows any user when `admin_users` is empty) — medium security, needs human sign-off to change auth behavior.
- **WI-117** (`_write_denied` allows writes when `auth_context` is missing) — low security, same constraint.
- **WI-092** (request-handler scans bypass `get_write_lock` while scheduler holds it) — the related concurrency issue that makes the transaction boundary's guarantees weaker under contention. Fixing WI-084/113/114 without 092 doesn't fully close the window.
- The remaining open WIs are all pre-existing low/medium-severity items (UI refactoring, code dedup, accessibility) that can be picked up in future batches.

## Gaps to flag

- **`_evaluate_posture` reads trust anchors via `SqliteTrustAnchorRepository(db_path).list_entries()` (no `conn`)** — this opens its own connection from the cached pool. Since it runs before `BEGIN` on the same thread's cached connection, the reads happen in auto-commit mode. This is correct but subtle: if someone later adds a write to `_evaluate_posture`, it would commit independently of the transaction.
- **The `_defer_webhook` parameter on `emit_event` has a leading underscore** (reviewer noted this) — it's now part of the contract between `events` and `scan` but looks private. Should be renamed to `defer_webhook` in a future pass.
- **The `_scan_and_store` helper's `_store_error_types` parameter** is now uniformly `(Exception,)` across all four route handlers, but the parameter itself exists only because `add_host` originally had a narrower type. It could be simplified away, but keeping it allows future per-route customization.
- **WI-084 transaction boundary under real production load was not validated** — only unit tests verify rollback behavior. The mvmcitest01 instance was not exercised. The WI-113 fix (moving network I/O before BEGIN) reduces lock contention but doesn't eliminate it — the `replace`, `posture` (storage), `policy`, `drift`, `history`, and `events` stages still hold the write lock.
- **The adversarial reviewer could not run `git diff` directly** (no shell access in the review subagent). It reviewed file contents instead. This is less precise than a diff-based review — the reviewer has to reconstruct what changed by reading the full file, which can miss subtle regressions in unchanged code paths.
