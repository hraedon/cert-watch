# Reflection — 2026-05-29 — mimo-v2.5-pro

## Session summary

Implemented 19 items from a comprehensive review of the cert-watch codebase: 6 fixes, 6 improvements, 6 features. Also decomposed the ~1500-line app.py monolith into 7 modules and filed 5 new breadcrumbs (BC-030 through BC-034).

## What was done

**Fixes:**
- Removed stale `settings` global, centralized config via `app.state.settings`
- Rate limit middleware now actually returns 429 (was just reporting headers)
- OAuth state cookie cleaned up on all callback error paths
- `AuthProvider.complete_oauth_flow` ABC now declares `state` param
- CT monitor uses `(serial, issuer)` composite key for dedup (was comparing serial against fingerprint — a silent bug)
- `_parse_dn_field` handles RFC 4514 escaped commas

**Improvements:**
- Thread-local DB connection cache (avoids repeated connect + WAL setup)
- `init_schema()` tracks initialized paths, repeat calls return early
- `list_dashboard_rows()` accepts sort/pagination params for SQL-level pagination
- Rate limiter evicts stale windows after 10 min inactivity
- CSV host import scans concurrently via ThreadPoolExecutor
- Added `dnspython>=2.6` as `[dns]` optional dependency

**Features:**
- `POST /api/webhook/test` — test webhook delivery
- `ALERT_DIGEST_ONLY=1` — single daily digest email instead of per-cert alerts
- `GET /api/export/hosts.csv` — bulk host export
- `tags` column on hosts for categorization
- `scan_interval_hours` per host for different scan frequencies
- `process_pending()` retries failed alerts up to 3 times

**Decomposition (BC-030):**
- `app.py` reduced from ~1500 lines to ~110 lines
- Split into: `middleware.py`, `filters.py`, `routes/views.py`, `routes/auth.py`, `routes/hosts.py`, `routes/certificates.py`, `routes/api.py`
- Tests updated to monkeypatch the correct modules

## What remains

- BC-030 is in-progress: the initial split is done but the route modules could benefit from further cleanup (some still have inline imports, the SSRF check logic in hosts.py duplicates scan.py)
- BC-031 (PostgreSQL/MSSQL) — significant effort, blocked by FEAT-006 (migrations)
- BC-032 (Structured logging) — straightforward, needs `CERT_WATCH_LOG_FORMAT` env var
- BC-033 (Dashboard grouping by fingerprint) — UI work, no schema change needed
- BC-034 (Owner/contact + renewal status) — schema change + UI + alert routing

## Gaps to flag

- The CT monitor dedup fix (BC-005) changed behavior — the old test was testing a buggy comparison. The new test reflects correct behavior but the CT monitor can no longer dedup against DB certificates using serial numbers (because the DB stores fingerprints, not serials). This is a known limitation documented in the code.
- The test `test_csrf_rejects_tampered_token` was updated to actually tamper with the token (was previously passing by accident due to module reload changing the CSRF secret).
- BC-028 and BC-029 remain open from prior sessions. BC-027/028 are duplicates — one should be closed.

## Subjective notes

The codebase is well-engineered. The repository pattern and optional auth dependencies are good design choices. The main risk is the SQLite single-writer constraint — it works for the current deployment model but blocks horizontal scaling. The spec-driven approach (docs/spec/wi_*.md) is unusual and effective.
