## On the project

The codebase continues to impress for a maintenance-mode project. Zero TODO markers, 90% coverage with per-module ratchets, defense-in-depth security. The scope bypass (WI-128) was the most significant finding — six API endpoints returned fleet-wide data to scope-tagged users, defeating RBAC that was carefully applied to dashboard and host routes. The fact that this survived multiple security review passes suggests the scope filtering pattern needs to be more discoverable — perhaps a decorator or middleware that applies it automatically, rather than each route handler having to remember to call `scope_tags_from_auth()`.

The adversarial review was high-value as always. Kimi found that `Settings.from_env()` would crash on invalid JSON in `CERT_WATCH_RENEWAL_WEBHOOK_HEADERS` — a startup-time crash that I introduced. Nemotron found the same thread pool lifecycle issues. Both found the race condition in `_flush_renewal_webhook_pool()`. The cross-model approach (two different model families) caught things a single reviewer would have missed.

## On the work done

**What went well:**
- The scope bypass fix (WI-128) was dispatched to a subagent that handled 11 files + 27 tests autonomously. The existing `_add_effective_tag_filter` helper made the SQL filtering pattern consistent.
- The scheduler fixes (WI-131/133) were tightly coupled and solved together — the thread pool, Settings integration, and env-var fallback all touch the same code path.
- Visual baselines updated cleanly via the CI artifact script. The settings page layout change was the only visual regression, as expected.

**What I'm less confident in:**
- The `_renewal_webhook_pool` lifecycle is now correct but still somewhat fragile — the global mutable state with a lock is a pattern that works but could be cleaner with a proper lifecycle object. The `hasattr(settings, "build_renewal_webhook_config")` duck-typing check in the scheduler is a code smell — it should just check `isinstance(settings, Settings)`.
- The events scope filtering (WI-128) hides hostname-less events from scoped users. This is a defensible design choice but may surprise operators who expect to see system-level events. Filed as WI-135 for a design decision.
- The calendar scope filtering uses an EXISTS subquery on the hosts table, which means uploaded certs without a matching host row are hidden from scoped users. Filed as WI-136.
- The `kv_loader.py` merge path for GUI-configured renewal webhook settings was not wired — the Settings fields are loaded from env but not yet merged from kv_store. Kimi flagged this; it means GUI-configured renewal webhook settings won't survive restart. This is a follow-up that should be tracked.

**What I did NOT verify:**
- Visual appearance of the settings left-rail layout on actual devices (only CI visual baselines match)
- The `kv_loader.py` merge path for renewal webhook settings
- Real-AD/Windows instance behavior on mvmcitest01
